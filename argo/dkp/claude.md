Отлично, у меня достаточно данных для полноценного анализа. Построю его по той же структуре, что и для ArgoCD — для каждого модуля: что он делает, какие риски снимает, какие вводит.

## 1. 🌐 `cni-cilium` + `cilium-hubble`

### Что делает
`cni-cilium` предоставляет сетевую связность в кластере Kubernetes с использованием Cilium CNI на базе eBPF. Cilium полностью заменяет функциональность модуля `kube-proxy`, который автоматически отключается при его включении.

Hubble — это полностью распределённая платформа наблюдения за сетью и безопасностью для cloud native нагрузок, построенная поверх Cilium и eBPF для глубокой видимости коммуникаций между сервисами.

---

### ✅ Риски, которые СНИМАЮТСЯ

| Угроза | Как снимается |
|---|---|
| Неконтролируемый трафик между подами | NetworkPolicy на уровне eBPF — самый эффективный enforcement |
| Отсутствие видимости сетевых потоков | Hubble даёт L4/L7 визуализацию всего трафика |
| Незашифрованный межузловой трафик | Режим WireGuard включает шифрование трафика между подами. |
| Неконтролируемый egress | Реализована возможность отказоустойчивого egress gateway, настраиваемого через ресурсы `EgressGateway` и `EgressGatewayPolicy`, которые позволяют назначить группу узлов в качестве шлюзов для внешних запросов. |

---

### ⚠️ Новые риски

**🔴 Ошибочная кластерная сетевая политика = отрез control plane**

`CiliumClusterwideNetworkPolicy` при отсутствии параметра `policyAuditMode` может привести к некорректной работе control plane или потере SSH-доступа ко всем узлам кластера.

**🟠 Нешифрованный трафик control-plane**

Трафик между control-plane нодами намеренно оставлен незашифрованным на уровне WireGuard во избежание проблем при обновлении публичных ключей — хотя трафик между компонентами control plane шифруется на прикладном уровне.

**🟠 Hubble как инструмент утечки данных**

`cilium-hubble` предоставляет визуализацию сетевого стека кластера при включённом Cilium CNI. Это означает, что пользователь с доступом к Hubble UI видит **все сетевые потоки** — включая чувствительные API-вызовы, заголовки запросов, endpoint-ы. Без должного ограничения доступа к UI это равносильно сетевому сниффингу.

**🟡 policyAuditMode как "дыра" для обхода политик**

В настройках модуля `cni-cilium` добавляется параметр `policyAuditMode: true` для безопасного тестирования политик. Если этот режим забыть отключить в prod — NetworkPolicy будут только логироваться, но не применяться: полная иллюзия защиты.

---

### 📊 Итог cni-cilium

| | ✅ Снимает | ⚠️ Вводит |
|---|---|---|
| Микросегментация | eBPF NetworkPolicy | Риск отрезать control plane |
| Шифрование трафика | WireGuard | Частично не покрывает CP-трафик |
| Видимость сети | Hubble | Hubble = потенциальная утечка |
| Egress-контроль | EgressGateway | — |

## 2. 🔀 `ingress-nginx`

### Что делает
`ingress-nginx` устанавливает и управляет Ingress NGINX Controller с помощью Custom Resources, поддерживает несколько контроллеров и режим высокой доступности для маршрутизации трафика и балансировки нагрузки.

---

### ✅ Риски, которые СНИМАЮТСЯ

- Единая точка входа — весь внешний трафик через один контролируемый компонент с TLS-терминацией
- Централизованное логирование запросов (access log) — аудит входящего трафика
- Возможность реализации IP Whitelist через аннотации (`nginx.ingress.kubernetes.io/whitelist-source-range`)
- Встроенная интеграция с `user-authn` через `external-auth` механизм

---

### ⚠️ Новые риски

**🔴 Профилирование в prod = root-процесс**

Режим профилирования включён в Ingress NGINX Controller. Это может увеличить потребление памяти, замедлить обработку запросов, и процесс может работать как root. Рекомендуется отключать профилирование, если оно не нужно активно для отладки.

**🟠 Конфигурационные инъекции через аннотации Ingress**

Аннотации типа `nginx.ingress.kubernetes.io/configuration-snippet` позволяют разработчикам **инжектировать произвольный NGINX-конфиг** в блок `location`. Если у разработчиков есть право создавать Ingress-ресурсы — это вектор атаки на другие приложения за тем же контроллером (например, проксирование заголовков авторизации чужого сервиса).

**🟠 Basic Auth как слабое звено**

Basic-авторизация недостаточно безопасна; рекомендуется реализовывать дополнительные меры защиты, например, аннотацию `whitelist-source-range`.

**🟡 CVE в nginx и Ingress controller**

Известные уязвимости были устранены в следующих модулях: `ingress-nginx`, `istio`, `prometheus` и `local-path-provisioner`. Это означает регулярное появление CVE — критично держать модуль на актуальной версии.

## 3. 🕸️ `istio`

### Что делает
Istio решает задачи взаимной аутентификации сервисов, шифрования трафика и авторизации доступа между сервисами прозрачным для приложений образом.

Каждый pod сервиса получает sidecar-контейнер `sidecar-proxy`. Этот контейнер содержит Envoy, который проксирует трафик и отвечает за всю функциональность Istio, включая маршрутизацию, аутентификацию, авторизацию.

---

### ✅ Риски, которые СНИМАЮТСЯ

| Угроза | Механизм Istio |
|---|---|
| Незашифрованный east-west трафик | mTLS между всеми сервисами |
| Неаутентифицированные вызовы между сервисами | PeerAuthentication + AuthorizationPolicy |
| Отсутствие видимости L7 трафика | Сбор и визуализация трассировок через Jaeger, экспорт метрик в Prometheus и визуализация через Grafana. |
| Неконтролируемые внешние вызовы | ServiceEntry + Egress Gateway |

---

### ⚠️ Новые риски

**🔴 Istio multicluster открывает API server наружу**

При включении multicluster добавляется прокси для публикации доступа к API server через стандартный Ingress. Deckhouse автоматически обменивается доверенными публичными ключами при настройке мультикластера. Сам прокси имеет только read-only доступ к ограниченному набору ресурсов.

Это означает, что API server частично открывается во внешнюю сеть — критический вектор атаки при неправильной конфигурации.

**🔴 Компрометация Istiod = компрометация всех mTLS сертификатов**

Istiod является Certificate Authority для всего mesh. Его компрометация позволяет расшифровать весь трафик или выдавать поддельные сертификаты сервисам.

**🟠 Federation требует обмена root CA между кластерами**

Federation требует взаимного доверия между кластерами. С технической точки зрения это достигается взаимным обменом корневыми сертификатами.

Компрометация одного кластера в федерации — потенциальная компрометация всех остальных.

**🟡 Sidecar injection — расширение поверхности атаки**

Каждый под получает дополнительный контейнер с Envoy. Уязвимость в Envoy (CVE) распространяется на все поды в mesh. Известные уязвимости регулярно закрываются в модуле `istio`.

**🟡 Версионирование data plane vs control plane**

Поды с версией Istio data plane, отличной от желаемой, будут изменены после перезапуска. Это означает период, когда в кластере одновременно работают разные версии sidecar. Разные версии могут иметь несовместимые политики безопасности.

## 4. 📊 `prometheus` + `grafana`

### Что делает
Модуль устанавливает и настраивает систему мониторинга Prometheus, настраивает сбор метрик для типичных приложений и предоставляет базовый набор алертов Prometheus и дашбордов Grafana.

Модуль мониторинга обеспечивает встроенную отказоустойчивость всех ключевых компонентов DKP. Все сервисы мониторинга развёртываются в нескольких экземплярах по умолчанию, что гарантирует продолжение работы при отказе отдельного инстанса.

---

### ✅ Риски, которые СНИМАЮТСЯ

- **Слепота при инцидентах**: метрики дают context для расследования
- **Отсутствие алертинга**: Deckhouse предоставляет расширенный режим мониторинга с алертами по метрикам, включая истечение сертификатов и другие события кластера Kubernetes.
- **Ручной мониторинг**: автоматизированный сбор метрик со всех компонентов
- Видимость сертификатов: добавлена поддержка мониторинга сертификатов в секретах типа `Opaque`.

---

### ⚠️ Новые риски

**🔴 Prometheus содержит чувствительную информацию о топологии**

Метрики раскрывают: список всех сервисов, namespace-ов, имена подов, IP-адреса, версии компонентов, конфигурацию ресурсов. Для атакующего это **идеальная разведка** инфраструктуры.

**🟠 Grafana — вектор инъекций через плагины**

Список кастомных плагинов Grafana содержит имена плагинов из официального репозитория. Также можно устанавливать плагины из других источников, передавая ссылку на zip-архив плагина.

Установка плагина из внешнего источника — прямой вектор supply chain атаки.

**🟠 PrometheusRemoteWrite = утечка метрик наружу**

PrometheusRemoteWrite позволяет отправлять метрики во внешние системы (например, VictoriaMetrics) с базовой авторизацией. Неправильно настроенный RemoteWrite может непреднамеренно отправлять метрики с чувствительными label-ами (имена пользователей, IP-адреса) во внешние системы.

**🟠 Доступ к Prometheus API = доступ к метрикам всего кластера**

Подключение к Prometheus защищено через `kube-rbac-proxy`. Для подключения нужно создать ServiceAccount с необходимыми правами. При избыточных правах SA — полный доступ к данным мониторинга всех тенантов.

**🟡 Alertmanager → внешние системы**

Alertmanager поддерживает отправку алертов напрямую в Telegram. Токены ботов в секретах кластера — ещё одна точка хранения credentials, требующая защиты.

## 5. 📜 `log-shipper` + `loki`

### Что делает
`log-shipper` упрощает настройку сбора логов в Kubernetes, позволяя организовать сбор логов приложений и самих нод. `loki` разворачивает хранилище операционных логов на базе Grafana Loki.

---

### ✅ Риски, которые СНИМАЮТСЯ

- **Потеря доказательной базы при инцидентах**: централизованное хранение логов
- **Прямой доступ к нодам для чтения логов**: логи агрегируются — нет нужды `kubectl exec` или SSH на ноду
- **Аудит приложений**: логи из всех namespace-ов собираются централизованно
- **Ваш кейс в действии**: вместо `kubectl logs` — Grafana + Loki с контролем доступа

---

### ⚠️ Новые риски

**🔴 Логи содержат чувствительные данные**

Приложения часто логируют: токены, пароли, PII (персональные данные), SQL-запросы с данными. Централизованный Loki становится **хранилищем всех потенциальных утечек**.

**🟠 log-shipper имеет доступ к логам ВСЕХ namespace-ов по умолчанию**

`log-shipper` агент запускается как DaemonSet с доступом к `/var/log` на каждой ноде. Он читает логи **всех контейнеров**, включая системные. Компрометация агента = компрометация всех логов.

**🟠 Отсутствие разграничения в Loki по умолчанию**

Loki не имеет встроенной мультитенантности «из коробки» в базовой конфигурации. Без явной настройки — пользователь с доступом к Grafana/Loki видит логи **всех** namespace-ов.

**🟠 Forwarding логов наружу**

log-shipper поддерживает форвардинг в Elasticsearch, Kafka, Loki, Vector и другие системы. Неправильная конфигурация endpoint-а → утечка логов в незащищённую систему.

**🟡 CVE в компонентах**

Известные уязвимости были закрыты в следующих модулях: `loki`, `extended-monitoring`, `operator-prometheus`, `prometheus`, `prometheus-metrics-adapter`, `user-authn`, `cloud-provider-zvirt`.

## 6. 🔑 `cert-manager`

### Что делает
`cert-manager` управляет TLS-сертификатами в кластере: поддерживает автоматическое получение сертификатов из различных CA, обновление сертификатов и мониторинг их действительности.

---

### ✅ Риски, которые СНИМАЮТСЯ

- Ручное управление сертификатами (истечение = инцидент)
- Использование самоподписанных сертификатов без ротации
- Хранение сертификатов вне Kubernetes (снижение контроля)
- Автоматический мониторинг истечения

---

### ⚠️ Новые риски

**🔴 cert-manager имеет доступ к созданию/обновлению Secret во всём кластере**

ServiceAccount cert-manager требует широкого RBAC для управления Secret-ами. Компрометация cert-manager = возможность подмены TLS-сертификатов любого сервиса в кластере.

**🟠 Credentials к внешним CA (ACME, Vault) в Secret-ах**

API-ключи для Let's Encrypt, токены Vault, credentials к внутренним PKI — всё хранится в Secret-ах. Требует защиты через Stronghold/External Secrets.

**🟡 Let's Encrypt rate limits**

При активном использовании — риск блокировки domain validation (слишком много запросов). В air-gapped средах Let's Encrypt недоступен — нужна явная настройка.

## 7. 🏗️ `node-manager`

### Что делает
`node-manager` управляет узлами кластера Kubernetes как связанной группой: настраивает и обновляет узлы, управляет масштабированием в облаке и локальными пользователями на узлах.

---

### ✅ Риски, которые СНИМАЮТСЯ

- Неконтролируемая конфигурация узлов (drift)
- Ручное управление SSH-доступом (rotated через `SSHCredentials`)
- Неавтоматизированное обновление OS и компонентов

---

### ⚠️ Новые риски

**🔴 `SSHCredentials` — хранение SSH-ключей в кластере**

Для управления bare-metal нодами `node-manager` хранит SSH credentials в Kubernetes Secret-ах. Компрометация `etcd` или широкого `get secrets` = получение SSH-доступа ко всем нодам.

**🔴 Автоматическое выполнение bashible-скриптов на нодах**

`node-manager` выполняет скрипты конфигурации на всех нодах. Компрометация механизма доставки скриптов = RCE на всех нодах кластера.

**🟠 Cloud provider credentials**

Для облачных кластеров `node-manager` использует credentials к cloud API (AWS IAM, Azure SP, GCP SA). Утечка → создание/удаление нод атакующим. Для AWS провайдера добавлен параметр `iamNodeRole` для привязки IAM-роли ко всем инстансам узлов, что может быть нужно для дополнительных разрешений, например, доступа к ECR.

## 8. 🏢 `multitenancy-manager`

### Что делает
Добавлена функциональность работы с проектами. Проект описывается ресурсами `ProjectTemplate` (шаблон) и `Project` (экземпляр). Модуль предлагает три преднастроенных шаблона: «базовый», «безопасный» и «безопасный, с выделенными узлами».

---

### ✅ Риски, которые СНИМАЮТСЯ

- Хаотичное создание namespace-ов без стандартных политик безопасности
- Отсутствие isolation между командами
- Ручная настройка RBAC, LimitRange, NetworkPolicy для каждого проекта
- Теперь можно создавать проект из существующего namespace и переносить объекты в него, что позволяет переходить на Projects без пересоздания ресурсов.

---

### ⚠️ Новые риски

**🟠 Шаблон "базовый" = слабая изоляция**

Три шаблона имеют разный уровень защиты. Команда может выбрать "базовый" шаблон без NetworkPolicy и PodSecurityPolicy — создав иллюзию управляемого тенанта без реальной изоляции.

**🟡 Наследование настроек от шаблона**

При изменении `ProjectTemplate` все связанные `Project`-ы получают обновление. Ошибка в шаблоне → одновременное нарушение безопасности для всех команд, использующих этот шаблон.

## 9. 🖥️ `dashboard` (Deckhouse Web UI / Console)

### Что делает
Deckhouse Kubernetes Platform Web Interface — веб-интерфейс для управления кластером, ориентированный на простоту управления и прозрачность состояния системы.

---

### ✅ Риски, которые СНИМАЮТСЯ

- Снижение потребности в прямом `kubectl` доступе для операционных задач
- Единый UI с контролем аутентификации через `user-authn`/Dex

---

### ⚠️ Новые риски

**🔴 Web UI = новая поверхность атаки**

UI доступен по Ingress. Любая XSS/CSRF уязвимость в веб-интерфейсе потенциально даёт атакующему возможность выполнять действия от имени залогиненного администратора.

**🟠 Раскрытие внутренней топологии**

Через UI видны: все namespace-ы, поды, сервисы, ноды, версии компонентов. При слабом RBAC — избыточная информация для нарушителя.

## 10. 🌩️ `commander` + `commander-agent`

### Что делает
`commander` позволяет создавать кластеры DKP на основе шаблонов, управлять их конфигурацией и жизненным циклом.

`commander-agent` отвечает за синхронизацию ресурсов Kubernetes в прикладном кластере и отправку телеметрии: метрик CPU, памяти, количества узлов, версии DKP и доступности компонентов.

---

### ✅ Риски, которые СНИМАЮТСЯ

- Ручное управление множеством кластеров через разрозненные kubeconfig
- Отсутствие единого аудита операций над кластерами

---

### ⚠️ Новые риски

**🔴 Commander = единая точка управления ВСЕМИ кластерами**

Компрометация Commander → компрометация всего fleet кластеров. Это максимально привилегированная система в экосистеме.

**🟠 Агент в каждом кластере**

В каждом кластере Commander автоматически устанавливает `commander-agent`, который синхронизирует ресурсы Kubernetes и отправляет телеметрию на API-сервер Commander.

Агент имеет исходящий канал к центральному серверу — компрометация этого канала или сервера = утечка метаданных всех кластеров.

**🟠 PostgreSQL как хранилище состояния**

Commander имеет внешнюю зависимость — базу данных PostgreSQL, которая является центральным хранилищем данных.

БД содержит credentials ко всем управляемым кластерам — критический актив, требующий шифрования at-rest и строгого контроля доступа.

## 11. 💾 Модули хранилища (`sds-*`, `local-path-provisioner`, `csi-*`)

### ⚠️ Ключевые риски

**🔴 Данные на PVC без шифрования at-rest**

Большинство CSI-драйверов не шифруют данные на диске по умолчанию. При физическом доступе к ноде или снятии snapshot — данные доступны в открытом виде.

**🟠 StorageClass без ограничений**

Без правильно настроенного `ResourceQuota` и контроля `StorageClass` через `admission policy` — любой тенант может занять всё доступное хранилище (DoS).

**🟡 Снапшоты как вектор утечки**

VolumeSnapshot API позволяет создавать снимки PVC. Без RBAC-ограничений — разработчик может создать snapshot с данными prod БД и прочитать его в dev-окружении.

## 12. 🤖 `virtualization`

### Что делает
Модуль предназначен для запуска и управления виртуальными машинами и их ресурсами на платформе Deckhouse. Предоставляет простой декларативный интерфейс для создания и управления ВМ.

---

### ⚠️ Ключевые риски

**🔴 VM escape — принципиально иной threat model**

ВМ внутри Kubernetes — это вложенная виртуализация. Уязвимость в гипервизоре (KubeVirt/Virsh) может привести к побегу из ВМ на уровень хоста, а оттуда — в кластер.

**🔴 Перезапуск kubelet/containerd при включении**

Включение модуля виртуализации требует перезапуска kubelet/containerd на всех нодах, где предполагается запуск ВМ, для настройки связности containerd и DVCR.

Это плановый disruption, но в неправильно спланированном окне — потенциальный DoS.

**🟠 Legacy ОС внутри ВМ**

Модуль позволяет запускать legacy-приложения, которые не могут работать в контейнере, и приложения, требующие non-Linux ОС.

Legacy ОС (Windows, старые Linux) — неpatched уязвимости, которые попадают в периметр кластера.

## 💡 Системные рекомендации для всех модулей

1. **Принцип наименьших привилегий** для ServiceAccount каждого модуля — аудировать регулярно
2. **Secrets → Stronghold (Vault)**: все credentials модулей через External Secrets, не в etcd
3. **Мониторинг целостности модулей**: начиная с версии 1.74 в DKP есть встроенный механизм контроля целостности модулей, защищающий от замены или модификации, автоматически включается при поддержке `erofs` ядром.
4. **Разные кластеры — разные зоны риска**: Commander, Prometheus, Loki — в отдельном управляющем кластере
5. **NetworkPolicy для системных namespace-ов** — `d8-monitoring`, `d8-ingress-nginx`, `d8-istio` должны быть изолированы между собой
6. **Аудит плагинов Grafana** и кастомных datasource — только из доверенных источников
7. **Update policy** — в ручном режиме обновления только критические исправления применяются автоматически, обновление до новой версии требует ручного подтверждения — используйте для prod-кластеров

## 🔍 Что такое `operator-argo` в Deckhouse

Модуль `operator-argo` разворачивает ArgoCD Operator в кластере Deckhouse Kubernetes Platform. Argo CD входит в состав платформы Deckhouse «из коробки». До появления этого модуля пользователям, которые работали с Argo CD, приходилось самостоятельно устанавливать и обновлять утилиту — теперь всё это делает Deckhouse, экономя время и трудозатраты инженеров.

Помимо CLI, можно использовать веб-интерфейс: в UI можно наблюдать за развёрнутыми приложениями, их ресурсами, состоянием, различными событиями, а также получать логи.

В основе подхода лежит GitOps: определения приложений, конфигурации и окружения должны быть декларативными и версионированными, а деплой и управление жизненным циклом — автоматизированными, аудируемыми и понятными.

## ✅ Риски и трудности, которые РАЗРЕШАЮТСЯ

### 1. Устранение прямого доступа к `kubectl` — ваш кейс
Это ключевое преимущество, которое вы уже реализовали. ArgoCD становится **единой контролируемой точкой входа** для разработчиков и тестировщиков. Вместо того чтобы давать им `kubectl` с широкими правами, вы даёте им:
- Видимость состояния деплоя и ресурсов
- Доступ к логам через UI
- Информацию о событиях (events) синхронизации

При этом прямой `exec` в поды, `port-forward`, `get secrets` — всё это остаётся под контролем RBAC Kubernetes, а не ArgoCD.

---

### 2. Разграничение доступа через AppProject и RBAC ArgoCD

Мультитенантная установка — наиболее распространённый способ установки ArgoCD. Такой тип инсталляции обычно используется для обслуживания нескольких команд разработчиков в организации и поддерживается командой платформы. Конечные пользователи могут получить доступ через Web UI или argocd CLI.

Через `AppProject` ArgoCD позволяет изолировать:
- Namespace-ы, к которым может деплоить конкретная команда
- Source-репозитории (whitelist)
- Разрешённые виды ресурсов Kubernetes

---

### 3. Интеграция с Dex (SSO) в Deckhouse
Deckhouse позволяет создать OAuth2-клиента для аутентификации в ArgoCD через ресурс `DexClient`, после чего DKP регистрирует клиента с идентификатором `dex-client-argocd@argocd`.

Это означает, что аутентификация в ArgoCD централизована через корпоративный IdP (LDAP, OIDC и т.д.) — **нет отдельных локальных учёток**.

---

### 4. Аудируемость и версионирование изменений
GitOps-подход фиксирует **каждое изменение в Git**: кто, когда и что изменил. Это критично с точки зрения ИБ — появляется полный аудит-трейл деплоев, которого нет при прямом `kubectl apply`.

---

### 5. Предотвращение дрейфа конфигурации (Configuration Drift)
Application controller — это Kubernetes-контроллер, который непрерывно мониторит запущенные приложения и сравнивает текущее живое состояние с желаемым целевым состоянием из репозитория. Он обнаруживает состояние OutOfSync и опционально принимает корректирующие действия.

С точки зрения ИБ — это защита от **несанкционированных изменений в кластере**: если кто-то вручную поменял что-то в кластере в обход GitOps, ArgoCD это обнаружит и может откатить.

## ⚠️ Новые риски, которые ПОЯВЛЯЮТСЯ

### 1. 🔴 ArgoCD — новая высокопривилегированная точка атаки

Стандартная установка Argo CD даёт доступ уровня `cluster-admin`. Используется, если вы планируете деплоить приложения в тот же кластер, где работает ArgoCD.

**Что это значит:** компрометация ArgoCD (его API, учёток, репозиториев) = компрометация всего кластера. ArgoCD имеет ServiceAccount с правами на изменение ресурсов во всём кластере.

---

### 2. 🔴 Риск через Git-репозиторий (Supply Chain Attack)

ArgoCD доверяет тому, что лежит в Git. Если злоумышленник:
- Получит доступ к репозиторию
- Внедрит вредоносный манифест
- Или скомпрометирует CI/CD-пайплайн, создающий PR

...то ArgoCD **автоматически применит** вредоносные изменения в кластер. Это классический Supply Chain риск.

**Митигация:** использовать `syncPolicy.automated` только в dev/test; в prod — требовать ручного подтверждения синхронизации.

---

### 3. 🟠 Утечка секретов через UI и API

API-сервер ArgoCD — это gRPC/REST сервер, который экспонирует API, потребляемый Web UI, CLI и CI/CD-системами. Repository server поддерживает локальный кэш Git-репозитория с манифестами приложений.

В кэше и через API могут быть видны:
- Helm values с чувствительными параметрами
- ConfigMap/Secret, если они отображаются в ресурсах приложения
- Пути к приватным репозиториям и credentials к ним

---

### 4. 🟠 Неправильная конфигурация RBAC ArgoCD ≠ RBAC Kubernetes

ArgoCD имеет **собственную систему ролей**, независимую от Kubernetes RBAC. Возможна ситуация:
- Пользователь ограничен в ArgoCD RBAC, но через баг или misconfiguration получает доступ к ресурсам кластера
- Или наоборот: в Kubernetes RBAC прав нет, но через ArgoCD можно инициировать деплой ресурсов в чужой namespace

---

### 5. 🟠 Риск при отключении модуля
Отключение модуля удалит ArgoCD Operator (все ресурсы из namespace `d8-operator-argo`). Однако развёрнутые инсталляции ArgoCD и приложения останутся нетронутыми.

Это значит, что после отключения модуля ArgoCD-инстанс продолжает работать **без контроля Deckhouse** — обновления, патчи безопасности перестанут применяться автоматически.

---

### 6. 🟡 Расширение поверхности атаки через Ingress

ArgoCD разворачивается с доступом через Ingress с TLS, что означает появление публично доступной (или доступной в корпоративной сети) точки входа. Необходимо:
- Жёстко ограничить IP-диапазоны через NetworkPolicy
- Настроить WAF
- Включить MFA через Dex

## 💡 Рекомендации по усилению модели ИБ

1. **Использовать `namespace-install`** вместо `cluster-admin` где это возможно
2. **`AppProject` с whitelist** namespace-ов и resource-ов для каждой команды
3. **Отключить `selfHeal` в prod** или требовать ручного approve
4. **Secrets в Git — никогда**: использовать External Secrets Operator + Deckhouse Stronghold (Vault)
5. **NetworkPolicy** ограничивающая доступ к ArgoCD API только с доверенных сетей
6. **Audit Log ArgoCD** → агрегировать в SIEM наравне с k8s audit log
7. **Отдельный ArgoCD-инстанс** для prod и non-prod окружений

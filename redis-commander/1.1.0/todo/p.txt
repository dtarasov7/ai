Прошу доработать redis клиента.

1. добавить auto-reconnect

Что нужно реализовать: 

- Автоматический reconnect при потере соединения
- Экспоненциальный backoff (чтобы не DDoS'ить сервер при проблемах)
- Retry логика для команд с настраиваемым количеством попыток
- Health check соединений в пуле
- Callbacks для мониторинга (on_connect, on_disconnect, on_reconnect)

2. Connection Pool

**Проблема:** Каждое соединение создаётся заново и никогда не переиспользуется. 
При высокой нагрузке это убьёт производительность и исчерпает файловые дескрипторы.

3. Автоматическое обновление топологии кластера

**Проблема:** Топология грузится только при старте. При failover или добавлении 
узлов клиент не узнает об изменениях до получения MOVED (неэффективно).

4. Настоящий Pipeline (batch через один сокет)

**Проблема:** Текущий pipeline просто последовательно выполняет команды — это не даёт никакого ускорения. `RedisPipeline` не уменьшает количество RTT: каждый вызов по-прежнему отправляется и читается отдельно.
  Нужен pipeline, который:
     - кодирует все команды в один большой буфер,
     - делает один `sendall`,
     - затем последовательно читает столько же RESP-ответов.


5. Multi-key команды с группировкой по слотам (MGET, MSET, DEL)
Команды типа `MGET`, `DEL` с ключами из разных слотов сейчас упадут с CROSSSLOT ошибкой.

6. Буферизация чтения RESP (ускорение I/O)

Сейчас `_read_line` читает по 1 байту через `sock.recv(1)` – это очень медленно.

Сделать тонкую обёртку над сокетом, которая:
- читает из ОС крупными блоками (например, 8К),
- но наружу через `.recv(n)` отдаёт ровно n байт из внутреннего буфера.

Тогда:
- `_read_line` по‑прежнему вызывает `recv(1)`, но это почти бесплатные операции в памяти;
- число системных вызовов `socket.recv` резко падает.

7. Поддержка `SENTINEL get-master-addr-by-name` и автоматический выбор мастера.

8. Для подключения по SSL сделать параметр в RedisClient, который отменяет проверку сертификата (CA, hostname, Время действия)

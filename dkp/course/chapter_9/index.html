Где эта глава в сквозном сценарии

После того как shop-demo получил политики допуска, сеть, доступ, секреты и scanning, появляется следующий вопрос: как доказать, что минимальные требования действительно выполняются, и как объяснить исключения. Эта глава учит строить audit trail для уже внедрённых мер защиты.

меры защиты в кластере
      |
      v
CIS checks / reports / dashboards
      |
      v
evidence pack -> audit discussion
      |
      v
engineering actions, not just reports

Базовые понятия

  • Compliance — соответствие внешнему стандарту, внутренней политике или регуляторному требованию.
  • CIS Kubernetes Benchmark — набор рекомендаций по безопасной конфигурации Kubernetes.
  • Compliance finding — найденное несоответствие требованию: например, слишком широкая RBAC-роль или Pod без нужного securityContext.
  • Evidence — доказательство для проверки: отчёт, YAML, лог, dashboard, ссылка на change request или результат команды.
  • Compensating control — компенсирующая мера, которая снижает риск, если требование нельзя выполнить буквально.
Хорошая формулировка для новичка: compliance отвечает на вопрос “можем ли мы показать соответствие требованию”, а security engineering отвечает на вопрос “действительно ли риск снижен”.

CIS Kubernetes Benchmark в DKP

DKP выполняет CIS-проверки через operator-trivy. Проверяются как настройки control plane и worker nodes, так и Kubernetes-объекты: RBAC, Pod Security Standards, network policies и другие элементы конфигурации.

apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: operator-trivy
spec:
  version: 1
  enabled: true
---
apiVersion: v1
kind: Namespace
metadata:
  name: payments
  labels:
    security-scanning.deckhouse.io/enabled: ""

Для compliance важно понимать расписание и область покрытия: CIS checks выполняются регулярно, но отчёт не освобождает команду от анализа исключений, false positives и ручных проверок.

  • VulnerabilityReport — уязвимости образов.
  • SbomReport — состав образов.
  • ConfigAuditReport — проблемы конфигурации Kubernetes-объектов.
  • ExposedSecretReport — секреты, найденные в контейнерах.

В Grafana

  • Security / CIS Kubernetes Benchmark.
  • Security / Trivy Image Vulnerability Overview.
  • Kubernetes audit logs.
  • Runtime audit engine logs.

Почему системные namespace имеют исключения

В CIS-отчётах DKP часть checks отключена или исключена для системных namespace. Это не “дырка”, а следствие архитектуры: сетевые агенты, storage-компоненты, node exporters и другие системные workloads иногда обязаны работать с host network, hostPath, root или дополнительными capabilities.

Что важно объяснить Практический вывод
Системные компоненты могут иметь привилегии по технической необходимости. Эти исключения нельзя копировать в пользовательские namespace.
Некоторые CIS checks требуют ручной интерпретации. Compliance report — начало анализа, а не автоматический verdict.
DKP может исключать checks из метрик, но сохранять результаты для аудита. Нужно знать, где находится raw evidence и как его объяснять.

DKP Certified Security Edition

DKP Certified Security Edition (CSE) — сертифицированная редакция платформы. По публичным материалам Deckhouse CSE имеет сертификат ФСТЭК России № 4860 от 04.10.2024 на ПО Deckhouse Platform CSE, соответствует требованиям приказа ФСТЭК № 118 к средствам контейнеризации по 4-му классу защиты и требованиям приказа № 76 по 4-му уровню доверия.

Что CSE даёт

  • Сертифицированную поставку платформы для регулируемых сред.
  • Основание для включения DKP в контур compliance и регуляторной документации.
  • Упрощение разговора с аудитом о статусе платформенного ПО.

Чего CSE не заменяет

  • Проектирование RBAC, NetworkPolicy, ProjectTemplate и secret delivery для конкретной организации.
  • Разбор findings, исправление уязвимых образов и сопровождение исключений.
  • Локальные процедуры доступа, change management, incident response и evidence collection.
Сертифицированная платформа не делает любую установку автоматически безопасной. Она даёт сильную базу и юридически значимый статус продукта, но эксплуатационная конфигурация всё равно остаётся ответственностью команды.

Что показать аудитору

  1. Версию DKP, редакцию, release channel и статус модулей безопасности.
  2. CIS summary из Grafana и raw ClusterComplianceReport.
  3. Список пользовательских namespace и их labels для scanning/policy enforcement.
  4. Примеры OperationPolicy, NetworkPolicy, RoleBinding, Project/ProjectTemplate.
  5. Список исключений: кто владелец, почему исключение нужно, когда оно пересматривается.
  6. Action log: какие findings привели к изменению образа, роли, политики или процесса.

Лабораторная работа

Задача

  1. Собрать evidence pack по shop-demo.
  2. Показать, какие CIS findings являются реальными проблемами, а какие требуют объяснения или исключения.
  3. Подготовить список corrective actions и владельцев.
  4. Сформулировать, что меняется при использовании DKP CSE и что всё равно остаётся на стороне команды.

Ожидаемый результат

  • Короткий compliance report для shop-demo.
  • Таблица finding -> риск -> владелец -> действие -> evidence.
  • Список исключений и компенсирующих мер защиты (compensating controls).

Что DKP даёт из коробки, а что делает команда

DKP из коробки Что должна сделать команда
Даёт CIS checks, reports и security dashboards. Назначить владельцев findings и процесс их разбора.
Даёт CSE как сертифицированную редакцию для регулируемых сред. Спроектировать локальную эксплуатационную конфигурацию и evidence process.
Документирует часть исключений для системных namespace. Не переносить системные исключения на приложения и объяснять локальные отклонения.

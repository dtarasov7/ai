# Лабораторная работа 11. Модули Deckhouse как часть модели угроз

## Цель

Научиться оценивать модули Deckhouse не по списку возможностей, а по изменению модели угроз: какой риск модуль закрывает, какой новый риск создаёт, какие ограничения нужны перед включением.

## Входные артефакты

- [глава 11](../chapter_11/index.html);
- markmap [`markmaps/chapter_11_module_risk_matrix.mm`](../markmaps/chapter_11_module_risk_matrix.mm);
- учебный стенд [`shop-demo`](../shop-demo/README.md);
- список модулей, которые команда хочет включить или уже использует.

## Последовательность действий

1. Выберите три модуля Deckhouse.

Минимальный набор для упражнения:
- `istio` в multicluster/federation-сценарии;
- `ingress-nginx`;
- `registry` или `registry-packages-proxy`.

Можно заменить один из них на `control-plane-manager`, `deckhouse`, `monitoring-kubernetes`, `upmeter`, `cert-manager`, `operator-trivy` или другой модуль, который реально обсуждается в вашей организации.

2. Для каждого модуля заполните risk matrix.

| Модуль | Какой риск закрываем | Какие новые привилегии появляются | Какие данные видит модуль | Blast radius при компрометации | Guardrails | Решение |
| --- | --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  | Включить / Пилот / Не включать |

3. Для одного модуля подготовьте хорошее решение.

Пример для `istio`: разрешён только один межкластерный вызов от `orders-api` к `payments-api`, а не “всё со всем”.

```yaml
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: allow-orders-to-payments
  namespace: payments
spec:
  selector:
    matchLabels:
      app: payments-api
  action: ALLOW
  rules:
    - from:
        - source:
            principals:
              - cluster.local/ns/orders/sa/orders-api
      to:
        - operation:
            methods: ["POST"]
            paths: ["/authorize"]
```

В пояснении укажите:
- почему правило узкое;
- кто владеет trust root;
- как обнаружить неожиданный межкластерный вызов;
- кто может менять это правило.

4. Для одного модуля подготовьте контрпример.

Пример плохого решения:

```yaml
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: allow-any-mesh-client
  namespace: payments
spec:
  selector:
    matchLabels:
      app: payments-api
  action: ALLOW
  rules:
    - from:
        - source:
            principals: ["*"]
```

Объясните, почему это плохо:
- wildcard скрывает реальную границу доступа;
- любой сервис в mesh становится потенциальным клиентом `payments-api`;
- при компрометации одного сервиса растёт blast radius;
- расследование усложняется, потому что правило не показывает бизнес-намерение.

5. Подготовьте решение по каждому выбранному модулю.

Формат решения:
- `Включить сейчас` — есть понятная задача, владелец, guardrails и rollback.
- `Только пилот` — польза вероятна, но нужно проверить ресурсы, false positives, совместимость или эксплуатационную сложность.
- `Не включать в текущих условиях` — модуль создаёт больше риска, чем закрывает.

## Что сдаёт слушатель

- заполненную risk matrix по трём модулям;
- один хороший YAML-фрагмент или архитектурный фрагмент;
- один контрпример;
- короткое решение по каждому модулю: включить, пилотировать или не включать;
- список guardrails, без которых решение нельзя переносить в production.

## Критерии завершения

- Для каждого модуля назван не только плюс, но и новый риск.
- В решении есть владелец: кто сопровождает настройки, alerts, обновления и rollback.
- Контрпример действительно опасен, а не просто “неидеален”.
- YAML-фрагмент не использует wildcard там, где можно указать конкретный namespace, service account, host или CIDR.
- Вывод написан как инженерное решение, а не как рекламное описание модуля.

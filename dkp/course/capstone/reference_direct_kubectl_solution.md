# Эталонный пример решения итогового кейса

## Когда Argo CD не вводят, а прямой `kubectl` оставляют только под жёсткие компенсирующие меры

## Как читать этот документ

Это альтернативный почти готовый эталонный ответ проверяющего для того же итогового кейса. Он показывает не “анти-Argo CD позицию”, а другую допустимую архитектурную развилку:
- среда достаточно небольшая и простая;
- отдельный GitOps/UI-контур не даёт достаточной дополнительной пользы;
- команда умеет жёстко ограничивать прямой `kubectl`;
- компенсирующие меры действительно внедрены, а не заявлены на словах.

Если таких компенсирующих мер нет, это решение не засчитывается. Оставить людям прямой доступ к Kubernetes API “потому что так привычнее” — плохой ответ. Оставить прямой `kubectl` в небольшой среде при узких ролях, generated kubeconfig, явном запрете `exec` по умолчанию и хорошем аудите — уже инженерная позиция, которую можно защитить.

## 1. Краткий вердикт проверяющего

**Статус:** зачтено с оговорками.

**Почему решение можно принять:**
- среда небольшая: один учебный кластер, несколько namespace, понятный круг пользователей;
- у команды нет отдельной потребности в Argo CD как пользовательском интерфейсе и нет желания вводить дополнительный привилегированный контур;
- прямой `kubectl` ограничен namespace-scoped ролями;
- kubeconfig выдаётся только через управляемый путь;
- `kubectl logs` разрешён, но `kubectl exec` по умолчанию запрещён;
- production-изменения не делаются произвольным ручным `kubectl apply`.

**Что проверяющий считает принципиальными условиями этого зачёта:**
- прямой `kubectl` не означает широких прав;
- доступ разделён по средам и по задачам;
- audit включён и реально используется;
- есть отдельный временный путь для интерактивной отладки;
- команда честно зафиксировала, при каком росте сложности это решение нужно будет пересмотреть в пользу Argo CD или другого self-service слоя.

## 2. Когда такой выбор вообще допустим

Такой вариант можно защищать только если одновременно выполняются несколько условий:

- мало команд и мало namespace;
- нет десятков приложений, для которых нужен единый UI наблюдаемости и sync-статусов;
- у платформенной команды нет задачи запускать и сопровождать отдельный Argo CD-контур;
- разработчики и тестировщики уже умеют работать с `kubectl get`, `kubectl describe` и `kubectl logs`;
- требования аудита и review доступа закрываются средствами DKP и Kubernetes без дополнительного пользовательского слоя.

Если среда большая, команд много, GitOps уже является обязательной моделью изменений, а dev/test нужен self-service просмотр приложений без kubeconfig, проверяющий, скорее всего, отвергнет этот вариант и попросит решение на базе Argo CD.

## 3. Почему в этом примере Argo CD не вводится

### Аргументы решения

- У кластера нет отдельной product-команды, которая сопровождала бы Argo CD как новый высокопривилегированный сервис.
- Число приложений и namespace пока невелико.
- Диагностические потребности покрываются `kubectl get`, `kubectl describe`, `kubectl logs` и dashboards.
- Команда хочет избежать появления ещё одного UI, API, ingress endpoint, repo credentials и отдельной RBAC-модели.

### Честная цена такого выбора

- у dev/test всё равно остаётся прямой путь в Kubernetes API;
- удобство ниже, чем у хорошего Argo CD UI;
- review ролей и аудит прямого доступа становятся ещё важнее;
- при росте масштаба этот подход начинает быстрее деградировать.

Проверяющий принимает это решение только потому, что команда не притворяется, будто “kubectl и так достаточно всегда”. Она прямо пишет границы применимости.

## 4. Архитектурный принцип

Прямой `kubectl` остаётся, но только в следующем виде:

1. **Generated kubeconfig через DKP** вместо ручной раздачи статических kubeconfig.
2. **SSO через Dex / внешний IdP** вместо локального зоопарка пользователей.
3. **Namespace-scoped роли** вместо cluster-wide read-only “для удобства”.
4. **`kubectl logs` разрешён, `kubectl exec` запрещён по умолчанию**.
5. **Изменения в production через CI/CD или platform/SRE**, а не через произвольный ручной деплой разработчика.
6. **Break-glass отдельно** от повседневного доступа.

Иными словами, проверяющий соглашается не с “прямым kubectl”, а с **жёстко ограниченным прямым kubectl**.

## 5. Матрица интерфейсов доступа

| Группа | Основной интерфейс | Что разрешено | Что запрещено по умолчанию | Владелец |
| --- | --- | --- | --- | --- |
| `shop-dev` | `kubectl` через generated kubeconfig | `get`, `list`, `watch`, `describe`, `logs` в своих namespace; в non-prod ограниченные изменения прикладных ресурсов | `exec`, cluster-scoped ресурсы, `secrets`, доступ в чужие namespace | владелец прикладной команды |
| `shop-test` | `kubectl` через generated kubeconfig | просмотр ресурсов, событий и логов в своих namespace | `exec`, изменение манифестов, cluster-wide доступ | владелец QA-функции |
| `shop-auditors` | `kubectl` read-only + dashboards | read-only просмотр и выгрузка evidence | любые изменения, `exec`, работа с секретами | владелец audit-функции |
| `platform-sre` | `kubectl` + административные роли | platform administration, incident response, временная отладка | неограниченные действия без журнала и review | руководитель платформенной команды |
| CI/CD | ServiceAccount / pipeline | controlled deploy path | интерактивный человеческий доступ от имени сервисного аккаунта | владелец CI/CD |

## 6. Компенсирующие меры

### 6.1. Generated kubeconfig, а не статические файлы “на годы”

Пользователи получают доступ через управляемый путь DKP, а не через архив с kubeconfig, который однажды выгрузили и забыли.

Почему это важно:
- проще отзывать доступ;
- меньше шанс, что уволенный сотрудник продолжает иметь рабочий kubeconfig;
- лучше связывается с SSO и review прав.

### 6.2. Роли отдельно для наблюдения и отдельно для интерактивной отладки

По умолчанию разработчику и тестировщику нужен не shell в контейнере, а:
- увидеть Pod;
- понять rollout;
- посмотреть событие;
- прочитать логи.

Значит, базовая роль должна закрывать именно это и не более.

### 6.3. `exec` только как временное исключение

Если нужна интерактивная отладка, это отдельный временный доступ, который:
- выдаётся ограниченному кругу;
- фиксируется;
- снимается после задачи;
- не превращается в постоянную роль “на всякий случай”.

### 6.4. Production не живёт на ручном `kubectl apply`

Даже если у части людей есть `kubectl`, это не должно означать, что production-изменения делаются вручную “потому что срочно”. Нужен нормальный путь:
- Git;
- review;
- CI/CD;
- platform/SRE approval там, где нужно.

### 6.5. Audit не декоративный

Если оставить прямой `kubectl`, но не включить нормальный API audit и не собирать события в общий контур журналирования, решение не засчитывается. Тогда команда просто оставила себе прямой административный канал без достаточной наблюдаемости.

## 7. Пример сильной read-only/diagnostic роли

Ниже пример роли для разработчиков или тестировщиков в namespace `payments`. Она позволяет смотреть ресурсы и логи, но не даёт `exec` и не даёт читать `Secret`.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: payments-observer
  namespace: payments
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps", "events"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get"]
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: payments-observers
  namespace: payments
subjects:
  - kind: Group
    name: shop-dev
  - kind: Group
    name: shop-test
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: payments-observer
```

### Почему это хороший минимум

- есть `logs`;
- есть просмотр rollout-объектов;
- нет `secrets`;
- нет `pods/exec`;
- нет прав на cluster-level;
- всё ограничено конкретным namespace.

## 8. Пример временного доступа для интерактивной отладки

Этот доступ не должен существовать как постоянная роль для всех. Он нужен как исключение под конкретный инцидент или задачу.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: temporary-pod-debug
  namespace: payments
  annotations:
    access-review.example.org/expires-at: "2026-04-28T18:00:00Z"
    access-review.example.org/ticket: "INC-2026-0412"
rules:
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/exec"]
    verbs: ["create"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: temporary-pod-debug-binding
  namespace: payments
  annotations:
    access-review.example.org/expires-at: "2026-04-28T18:00:00Z"
subjects:
  - kind: Group
    name: platform-sre
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: temporary-pod-debug
```

### Почему это считается допустимым

Потому что здесь `exec`:
- ограничен по namespace;
- выдан не всем;
- связан с тикетом и временем;
- существует как отдельный путь, а не как штатная роль разработчика.

Аннотация сама по себе не отзывает доступ автоматически. Но она делает исключение явным и пригодным для review. Если у команды нет процесса, который реально снимает такие права, пример перестаёт быть сильным.

## 9. Решение по средам

### Non-production

- у `shop-dev` могут быть ограниченные права на изменение прикладных ресурсов в своих namespace;
- `shop-test` обычно остаётся read-only plus logs;
- `exec` всё равно не выдаётся по умолчанию.

### Production

- `shop-dev` и `shop-test` имеют только просмотр и логи;
- изменение production-объектов идёт через CI/CD или platform/SRE;
- интерактивная отладка только как временное исключение.

Проверяющий ждёт именно такого разведения. Если команда говорит “мы оставили прямой kubectl” и не разделяет non-prod и prod, это красный флаг.

## 10. Что обязательно должен увидеть проверяющий

### Проверка 1. Пользователь получает generated kubeconfig, а не статический файл из архива

Ожидаемый результат:
- доступ идёт через SSO-процесс;
- при отзыве группы или роли kubeconfig перестаёт быть рабочим.

### Проверка 2. Dev может смотреть логи, но не может `exec`

Проверка:

```bash
kubectl -n payments logs deploy/payments-api
kubectl -n payments auth can-i create pods/exec
```

Ожидаемый результат:
- логи читаются;
- `auth can-i create pods/exec` возвращает `no`.

### Проверка 3. Dev не может читать секреты

Проверка:

```bash
kubectl -n payments auth can-i get secrets
```

Ожидаемый результат:
- `no`.

### Проверка 4. Dev не может уйти в cluster-level

Проверка:

```bash
kubectl auth can-i list nodes
kubectl auth can-i create clusterrolebinding
```

Ожидаемый результат:
- оба ответа `no`.

### Проверка 5. Временный debug-доступ действительно временный

Ожидаемый результат:
- есть явное описание, кто снимает `RoleBinding`;
- после срока или закрытия инцидента временная привязка удаляется;
- review может показать, что это не постоянная “параллельная роль”.

## 11. Что здесь лучше, чем кажется

Это решение не пытается сделать вид, будто прямой `kubectl` безопасен сам по себе. Наоборот, оно показывает, что прямой доступ допустим только если:
- минимизирован scope;
- ограничены verbs;
- логи разрешены, а интерактивный доступ вынесен отдельно;
- аудит и review реально работают.

Именно такая честность и делает ответ сильным.

## 12. Что здесь хуже, чем в хорошем решении с Argo CD

Проверяющий всё равно отметит несколько минусов по сравнению с эталоном на Argo CD:

- у пользователей остаётся прямой путь в Kubernetes API;
- удобство ниже, чем у хорошего UI;
- нужен более высокий уровень дисциплины при review прямых RBAC-ролей;
- drift и user-facing status приложения не собраны в одном месте;
- при росте числа команд решение масштабируется хуже.

То есть это не “лучшее решение вообще”. Это **допустимое решение для узкой и зрелой по процессам среды**.

## 13. Когда проверяющий перестанет принимать этот вариант

Если появляется хотя бы часть следующих признаков, ответ нужно пересматривать:

- больше команд и больше namespace;
- постоянные просьбы дать ещё чуть-чуть `kubectl`;
- frequent ad hoc debug в production;
- желание дать dev/test self-service видимость приложений без расширения RBAC;
- появление GitOps как обязательной модели delivery;
- нехватка прозрачности по состоянию приложений и sync-истории.

В этот момент проверяющий, скорее всего, скажет: пора выносить диагностику и часть пользовательского доступа в отдельный контролируемый слой вроде Argo CD.

## 14. Что проверяющий запишет в сильные стороны

- Команда не стала вводить Argo CD “потому что модно”, а оценила цену эксплуатации.
- Прямой `kubectl` оставлен только в узкой, namespaced и проверяемой форме.
- Хорошо разделены `logs` и `exec`.
- Есть generated kubeconfig, review, audit и break-glass.
- Production и non-production разведены.

## 15. Что проверяющий запишет в замечания

- Нет явной частоты review ролей и групп.
- Не описано, кто снимает временные debug-права.
- Нет подтверждения, что API audit реально уходит в общий журнал.
- Не описан момент, когда организация должна пересмотреть решение в пользу Argo CD.

## 16. Итоговая формулировка проверяющего

Решение засчитывается не потому, что “kubectl тоже бывает нормальным”, а потому, что команда:
- показала малый scope среды;
- честно не ввела Argo CD там, где он не окупается;
- оставила прямой доступ только с жёсткими ограничениями;
- развела базовую диагностику и интерактивную отладку;
- не спрятала остаточные риски.

Если хотя бы одна из этих опор отсутствует, ответ быстро превращается в обычную нестрогую раздачу `kubectl`, и тогда он должен быть отклонён.

Лабораторная работа 5. Argo CD как контролируемый доступ к приложениям

Цель

Спроектировать такой доступ через Argo CD, который убирает у dev/test лишний прямой kubectl, но не ломает повседневную диагностику приложений.

Исходные материалы

  • shop-demo/README.md
  • глава 5 курса
  • markmap markmaps/chapter_05_argocd_access.mm
  • при желании: эталонный пример с Argo CD и эталонный пример с прямым kubectl как примеры сильной развилки, но не как подсказка вместо собственного решения

Ожидаемое время

90-120 минут

Задание

  1. Опишите, зачем в вашем сценарии вообще вводится Argo CD:
    • что именно вы хотите убрать из прямого kubectl;
    • что при этом должно остаться доступным для dev/test;
    • почему это не просто "ещё один UI".
  2. Подготовьте матрицу интерфейсов доступа:
    • dev;
    • test;
    • auditors;
    • platform/SRE;
    • CI/CD.
  3. Зафиксируйте, какие действия разрешены через Argo CD:
    • просмотр приложения;
    • просмотр логов;
    • просмотр истории синхронизации;
    • ручной sync;
    • rollback;
    • exec.
  4. Подготовьте учебный пример argocd-rbac-cm:
    • policy.default: role:none;
    • роли для dev/test и auditors;
    • без exec;
    • без override.
  5. Подготовьте учебный пример AppProject:
    • один разрешённый Git-репозиторий;
    • только свои namespace;
    • без cluster-scoped ресурсов;
    • с явной позицией по Secret.
  6. Примите отдельное решение по auto-sync и self-heal:
    • production;
    • non-production.
  7. Опишите новый риск-контур:
    • UI/API;
    • repo credentials;
    • Git как источник изменений;
    • зависимость от SSO и самого Argo CD.
  8. Опишите break-glass:
    • кто сохраняет прямой kubectl;
    • в каком случае он используется;
    • как журналируется;
    • кто делает review временного исключения.
  9. Отдельно напишите короткий абзац: в каком случае вы бы отказались от Argo CD и оставили прямой kubectl под компенсирующие меры.

Что нужно сдать

  • краткое архитектурное решение "зачем Argo CD в этой среде";
  • access matrix group -> interface -> actions -> owner;
  • пример argocd-rbac-cm;
  • пример AppProject;
  • решение по auto-sync / self-heal;
  • описание остаточных рисков;
  • описание break-glass-пути.

Критерии завершения

  • Argo CD используется как действительно более узкий слой доступа, а не как второй административный интерфейс;
  • dev/test не получают через UI почти то же самое, что раньше получили бы через прямой kubectl;
  • AppProject ограничивает repo, namespace и типы ресурсов;
  • есть отдельная позиция по production и non-production;
  • описано, что делать при отказе Argo CD, Git или SSO.

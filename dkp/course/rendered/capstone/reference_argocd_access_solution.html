Эталонный пример решения capstone-кейса

Argo CD как ограниченная точка доступа вместо прямого kubectl для dev/test

Как читать этот документ

Это не “единственно правильный” ответ и не универсальный production-шаблон для любой организации. Это почти готовый эталонный ответ проверяющего: пример сильного решения, которое засчитывается по логике курса, потому что:

  • снижает поверхность доступа;
  • не ломает диагностику;
  • не подменяет Kubernetes RBAC хаотичным доступом через UI;
  • явно показывает остаточные риски и способы их ограничения.

Если в вашей среде принято другое разделение обязанностей, иное именование групп или другой GitOps-процесс, это допустимо. Недопустимо другое: назвать решение “безопасным”, не показав, где именно проходят границы прав и почему они не разваливаются при первой же удобной просьбе “дайте ещё чуть-чуть”.

1. Краткий вердикт проверяющего

Статус: зачтено.

Почему решение засчитывается:

  • dev/test не получают kubeconfig и не ходят напрямую в Kubernetes API;
  • для повседневной диагностики им хватает Argo CD UI и логов;
  • exec, override, cluster-scoped ресурсы и деплой в чужие namespace не выданы по умолчанию;
  • прямой kubectl остаётся только у platform/SRE и оформлен как отдельный путь;
  • для production явно ограничены auto-sync и self-heal;
  • GitOps-контур не используется как предлог хранить секреты в Git.

Что проверяющий всё ещё требует держать под контролем:

  • владелец repo credentials и Argo CD RBAC;
  • регулярный review AppProject и групп в SSO;
  • экспорт audit-событий Argo CD и Kubernetes в общий журнал;
  • отдельный break-glass-путь на случай отказа Argo CD, Dex или Git.

2. Исходные допущения

  • Учебный стенд: shop-demo.
  • Namespace приложений: frontend, orders, payments.
  • Технический namespace платформы: platform-tools.
  • Argo CD развёрнут платформенной командой и интегрирован с SSO через Dex.
  • Манифесты приложений лежат в одном Git-репозитории платформы: https://git.example.org/platform/shop-demo.git.
  • Секреты не хранятся в Git. Для секретов используется отдельный путь из главы 6: Stronghold / secrets-store / External Secrets.
  • Речь идёт о production-подобной среде, где удобство диагностики важно, но не важнее границ доступа.

3. Архитектурное решение

Выбранный принцип

Мы сознательно разделяем два разных типа доступа:

  1. Доступ к приложениям для чтения и базовой диагностики.
    Даётся через Argo CD.

  2. Доступ к платформе и Kubernetes API для администрирования.
    Остаётся только у platform/SRE через прямой kubectl.

Это решение выбрано потому, что у разработчиков и тестировщиков в повседневной работе чаще всего есть четыре реальные потребности:

  • увидеть, какой манифест сейчас применён;
  • понять, что приложение в состоянии Healthy или OutOfSync;
  • посмотреть связанные ресурсы;
  • прочитать логи и события.

Для этого не нужен ни cluster-admin, ни kubeconfig, ни kubectl exec.

Что сознательно не делаем

  • Не выдаём dev/test общий read-only в Kubernetes API “на всякий случай”.
  • Не выдаём exec “иногда для отладки”.
  • Не разрешаем синхронизацию произвольной ревизии из UI.
  • Не разрешаем dev/test менять AppProject, Argo CD RBAC, настройки оператора и repo credentials.

4. Матрица интерфейсов доступа

Группа Основной интерфейс Что разрешено Что запрещено по умолчанию Владелец
shop-dev Argo CD UI / CLI видеть приложения, дерево ресурсов, события, историю синхронизации, логи kubectl, exec, override, sync в production, доступ к чужим приложениям владелец прикладной команды
shop-test Argo CD UI видеть приложения, состояние rollout, логи kubectl, exec, sync, доступ к чужим namespace владелец QA-функции
shop-auditors Argo CD UI + dashboards read-only просмотр приложений и отчётов любые изменения, exec, доступ к repo credentials владелец audit-функции
platform-sre kubectl + Argo CD admin администрирование кластера, разбор инцидентов, break-glass, настройка Argo CD работа без журнала и без review временных исключений руководитель платформенной команды
CI/CD Git + контролируемый merge process изменение желаемого состояния через Git прямой деплой в production через ручной kubectl apply владелец CI/CD

5. Почему это сильнее, чем просто “отобрать kubectl”

Плохой вариант выглядит так: у людей забрали kubectl, но ничего не дали взамен. В результате они начинают просить временные исключения, общий read-only kubeconfig, доступ к логам через обходные сервисы и быстро возвращаются к старой схеме.

Хороший вариант выглядит иначе:

  • прямой доступ к API действительно сокращается;
  • повседневная диагностика не страдает;
  • команды не начинают ежедневно дергать platform/SRE ради просмотра статуса деплоя;
  • сама модель доступа остаётся читаемой и проверяемой.

Именно поэтому в данном кейсе Argo CD выбран не как “модный GitOps”, а как контролируемый пользовательский слой доступа к приложениям.

6. Конкретная реализация

6.1. SSO и базовая дисциплина

  • Вход в Argo CD идёт через Dex / корпоративный IdP.
  • Локальный admin используется только на bootstrap-этапе и затем отключается.
  • Группы shop-dev, shop-test, shop-auditors, platform-sre синхронизированы из внешнего IdP.
  • policy.default выставлен в role:none.

Это важно, потому что “широкая default role, а потом мы всё дожмём deny-правилами” — плохая идея. В реальной системе это почти всегда кончается тем, что кто-то уже имеет лишний доступ, а команда думает, что “формально всё ограничила”.

6.2. RBAC Argo CD для dev/test

apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  policy.default: role:none
  policy.csv: |
    p, role:shop-dev-readonly, applications, get, shop-demo/*, allow
    p, role:shop-dev-readonly, logs, get, shop-demo/*, allow
    p, role:shop-auditor-readonly, applications, get, shop-demo/*, allow
    p, role:shop-auditor-readonly, logs, get, shop-demo/*, allow
    g, shop-dev, role:shop-dev-readonly
    g, shop-test, role:shop-dev-readonly
    g, shop-auditors, role:shop-auditor-readonly

Почему это считается хорошим минимумом

  • Есть право видеть приложения.
  • Есть право видеть логи.
  • Нет exec.
  • Нет override.
  • Нет широкого доступа “всем залогиненным”.

Почему здесь нет sync

Для production-подобной среды это осознанное ограничение. Команды разработки и тестирования видят состояние и могут диагностировать проблему, но не применяют изменения напрямую из UI. Изменение проходит через Git, review и merge.

Если организации нужен ручной sync в non-prod, это можно ввести отдельной ролью только для конкретной среды. Но это должно быть именно отдельное решение, а не привычка “давайте всем, а потом разберёмся”.

6.3. AppProject как граница полномочий

apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: shop-demo
  namespace: argocd
spec:
  sourceRepos:
    - https://git.example.org/platform/shop-demo.git
  destinations:
    - namespace: frontend
      server: https://kubernetes.default.svc
    - namespace: orders
      server: https://kubernetes.default.svc
    - namespace: payments
      server: https://kubernetes.default.svc
  clusterResourceWhitelist: []
  namespaceResourceWhitelist:
    - group: apps
      kind: Deployment
    - group: ""
      kind: Service
    - group: ""
      kind: ConfigMap
    - group: networking.k8s.io
      kind: Ingress
  namespaceResourceBlacklist:
    - group: ""
      kind: Secret

Почему это хорошо

  • Разрешён ровно один репозиторий.
  • Разрешены только свои namespace.
  • Cluster-scoped ресурсы не разрешены.
  • Обычные Secret не разворачиваются напрямую из Git.

Почему это важно отдельно проговорить

Очень частая ошибка — считать AppProject просто папкой или организационной сущностью. На самом деле это часть security boundary. Если здесь разрешить лишнее, команда получит обходной путь не через kubectl, а через GitOps.

7. Что остаётся только у platform/SRE

Следующие действия не входят в повседневный доступ dev/test:

  • прямой kubectl в production-кластер;
  • kubectl exec;
  • kubectl port-forward;
  • изменение Argo CD RBAC;
  • изменение AppProject;
  • изменение repo credentials;
  • работа с cluster-scoped ресурсами;
  • разбор и изменение operator-level настроек;
  • break-glass при отказе основного контура.

Это принципиально важно. Если эти действия доступны слишком широкому кругу, то Argo CD перестаёт быть слоем ограничения и превращается в просто ещё один административный интерфейс.

8. Решение по auto-sync и self-heal

Production

  • auto-sync: выключен.
  • self-heal: выключен.
  • Применение изменений: через Git + review + ручное подтверждение sync platform/SRE.

Non-production

  • auto-sync: допустим.
  • self-heal: допустим после проверки команды.

Почему так

Drift detection полезен везде. Но в production auto-sync и self-heal означают, что ошибочный merge или компрометация Git почти сразу превращаются в фактическое изменение среды. Для курса это важный критерий зрелости: команда должна не просто знать, что функция существует, а понимать цену автоматизма.

9. Секреты и чувствительные данные

Это решение не засчитывается, если команда хранит business secrets в Git и объясняет это “удобством GitOps”.

Приемлемый вариант для capstone:

  • секреты живут вне Git;
  • в Git лежат только ссылки на внешний источник секрета или безопасные шаблоны;
  • прямой ресурс Secret не является основным способом поставки чувствительных данных в приложение.

Именно поэтому в reference AppProject показан с запретом на прямой Secret.

10. Проверка, которую должен увидеть проверяющий

Проверка 1. Dev видит приложение, но не имеет kubeconfig

Ожидаемый результат:

  • пользователь из shop-dev логинится в Argo CD;
  • видит только приложения shop-demo;
  • не получает kubeconfig и не имеет прямого доступа к API server.

Проверка 2. Dev видит логи, но не может сделать exec

Ожидаемый результат:

  • просмотр логов доступен;
  • действие exec в UI и CLI недоступно;
  • команда не может открыть интерактивную сессию в контейнере.

Проверка 3. Test не видит чужие namespace

Ожидаемый результат:

  • пользователь из shop-test видит только разрешённые приложения;
  • не видит чужие проекты и не может деплоить в другой namespace.

Проверка 4. PR с cluster-scoped ресурсом не должен пройти как нормальный путь

Ожидаемый результат:

  • манифест с ClusterRoleBinding или другим cluster-scoped объектом не вписывается в границы AppProject;
  • команда не может “тихо” расширить себе полномочия через Git.

Проверка 5. Production sync остаётся под контролем

Ожидаемый результат:

  • dev/test не могут синхронизировать произвольную ревизию;
  • для production есть ручная точка принятия решения.

11. Остаточные риски и что с ними делать

Риск 1. Компрометация Git или repo credentials

Что делать:

  • ограничить владельцев repo credentials;
  • включить review PR;
  • разделить production и non-production контуры;
  • журналировать изменения и sync-события.

Риск 2. Ошибка в Argo CD RBAC

Что делать:

  • регулярно проверять policy.csv;
  • не использовать широкие default roles;
  • отдельно тестировать негативные сценарии: “не должен видеть”, “не должен exec”, “не должен sync”.

Риск 3. Слишком широкий AppProject

Что делать:

  • не разрешать cluster-scoped ресурсы по умолчанию;
  • не разрешать чужие namespace;
  • держать whitelist ресурсов минимальным.

Риск 4. Зависимость от Argo CD и SSO

Что делать:

  • описать break-glass;
  • ограничить, кто может им пользоваться;
  • периодически проверять, что аварийный путь вообще работает.

12. Break-glass

Когда допускается

  • Argo CD недоступен;
  • Dex / SSO недоступны;
  • требуется срочное platform-level вмешательство;
  • есть инцидент, в котором UI-путь мешает скорости реакции.

Кто использует

  • только platform-sre или дежурный on-call;
  • с последующей фиксацией причины, времени, объёма действий и review после инцидента.

Что важно

Break-glass не должен быть “секретной короткой дорогой”, которую все знают и иногда используют “потому что так быстрее”. Если обычный путь неудобен настолько, что люди постоянно хотят обход, значит проблема в модели доступа, а не в пользователях.

13. Что проверяющий запишет в сильные стороны

  • Хорошо проведена граница между приложением и платформой.
  • Убрали прямой kubectl у dev/test без потери наблюдаемости.
  • Не смешали GitOps и платформенное администрирование.
  • Показали, что AppProject — часть security boundary.
  • Явно отделили production от non-production по auto-sync.
  • Не забыли про break-glass и владельцев процесса.

14. Что проверяющий запишет в замечания, если этого нет

  • Нет явного владельца Argo CD RBAC.
  • Нет явного владельца repo credentials.
  • Не описан путь отзыва прав при увольнении или смене команды.
  • Не описано, как аудиторы получают read-only доступ.
  • Не показано, как экспортируются Argo CD и Kubernetes audit-события.
  • Непонятно, как platform/SRE проверяет, что dev/test действительно не могут сделать exec.

15. Итоговая формулировка проверяющего

Решение засчитывается как зрелое и практически применимое. Оно не сводится к лозунгу “убрали kubectl”, а показывает полноценную инженерную модель:

  • у повседневной диагностики есть удобный и ограниченный путь;
  • у платформенного администрирования есть отдельный путь;
  • у GitOps-контура есть явные границы;
  • остаточные риски названы и не маскируются под “это потом настроим”.

Если команда воспроизведёт такую логику у себя и действительно проверит запрет на exec, ширину AppProject, границы production sync и break-glass, это уже не учебная абстракция, а хороший рабочий security-паттерн.

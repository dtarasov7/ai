Безопасность пода

В Kubernetes именно Pod часто становится точкой, где хорошая архитектура встречается с плохой эксплуатационной привычкой. В этой главе мы подробно разбираем, как превратить безопасность Pod из разовой ручной проверки в устойчивую платформенную политику на базе встроенного модуля Deckhouse admission-policy-engine.

Обязательный минимум

  • Объяснить разницу между Pod Security Standards, securityContext и политикой допуска ресурсов.
  • Определить минимальный набор требований для запускаемых приложений в production: non-root, запрет privilege escalation, ресурсы, probes и контролируемые registry.
  • Выбрать правильный режим rollout: warn, dryrun или deny.
  • Прочитать небезопасный Pod spec и быстро увидеть его основные риски.

Углубление и применимость

  • Понять, где заканчивается PSS и начинаются общие правила платформы: обязательные probes, requests/limits, разрешённые registry и правила для тегов образов.
  • Использовать OperationPolicy и связанные ограничения как часть минимального уровня безопасности.
  • Подготовить путь миграции legacy-приложений без конфликта с разработкой.

Почему Pod security — это основа, а не частная тема

Pod — это не просто контейнер в Kubernetes. Это место, где описание в YAML превращается в реально запущенный код. Если небезопасный Pod допущен к запуску, многие другие проверки уже вынуждены догонять проблему постфактум.

Базовые понятия для новичка

  • Container — изолированный процесс с собственным окружением и файловой системой.
  • Pod — оболочка Kubernetes вокруг одного или нескольких контейнеров, которые запускаются вместе.
  • Namespace — логическая область внутри кластера, помогающая разделять приложения и команды.
  • Security Context — поля манифеста, которые задают права и ограничения процесса.
  • Admission policy — правило допуска: оно не даёт ресурсу попасть в кластер, если он нарушает требования.

Какие ошибки Pod spec особенно опасны

  • Запуск от root без необходимости.
  • Разрешённый privilege escalation.
  • Избыточные Linux capabilities.
  • Подключение host namespaces и небезопасных hostPath.
  • Отсутствие ограничений по ресурсам, что повышает риск DoS внутри кластера.

Почему нельзя надеяться только на review манифестов

  • Ручной review не масштабируется на десятки команд и сотни deployment-ов.
  • Нарушения появляются не только в исходном коде, но и в Helm values, generated manifests и оперативных hotfix.
  • Без политики допуска ресурсов платформа не может гарантировать единый минимальный набор требований.

Pod Security Standards: от разрешающего режима к production-требованиям

Pod Security Standards описывают три стандартных уровня строгости для Pod: privileged, baseline и restricted. Эти уровни отвечают на вопрос «насколько опасные настройки Pod разрешены». Здесь baseline — это имя официального уровня PSS, а не просто английское слово.

В Deckhouse эта схема уровней используется как базовый способ описать требования к безопасности Pod. Но PSS проверяет не всё, что нужно реальной платформе: например, он не знает, из каких registry организация разрешает брать образы, нужно ли всегда задавать probes и какие теги образов запрещены. Поэтому DKP расширяет стандартную проверку через Gatekeeper и собственные объекты политик.

Уровень Смысл Как использовать на практике
privileged Почти ничего не запрещает в настройках Pod. Оставлять только там, где есть доказанная необходимость, например для отдельных системных компонентов.
baseline Запрещает самые известные опасные настройки, которые часто ведут к повышению привилегий. Универсальный стартовый уровень требований для пользовательских namespace и новых установок DKP.
restricted Самый строгий уровень требований к запуску Pod. Целевой режим для production-приложений, если они готовы запускаться с такими ограничениями.

В документации Deckhouse отдельно отмечено, что начиная с DKP v1.55 политика по умолчанию для новых установок смещена к baseline. Это важно организационно: платформа тем самым задаёт более безопасное исходное состояние, а не оставляет его на усмотрение каждой команды.

Security Context и усиление настроек Pod

Pod Security Standards задают общую рамку допустимого, но реальная повседневная безопасность часто решается полями securityContext. Именно здесь описывается, с какими правами процесс стартует внутри контейнера и какие действия ему запрещены.

Хороший минимальный набор требований для большинства приложений

  • runAsNonRoot: true
  • явно заданные runAsUser и runAsGroup
  • allowPrivilegeEscalation: false
  • readOnlyRootFilesystem: true, если приложение поддерживает этот режим
  • capabilities.drop: ["ALL"]
  • seccompProfile.type: RuntimeDefault

Чего здесь не видно, но что критично

  • Совместимость приложения с корневой файловой системой только для чтения нужно тестировать заранее.
  • Некоторые legacy-приложения по умолчанию ожидают root-права и ломаются после усиления настроек.
  • Если capabilities возвращаются обратно «на всякий случай», усиление настроек быстро вырождается в имитацию.
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 10001
    runAsGroup: 10001
  containers:
  - name: app
    image: registry.example.org/team/app:v1.4.2
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
      seccompProfile:
        type: RuntimeDefault

Admission как проверка на входе

Admission controller нужен не потому, что разработчики пишут плохие манифесты. Он нужен потому, что даже хорошие команды со временем ошибаются, а платформа должна превращать важные требования безопасности в автоматическую норму.

Если объяснять совсем просто, admission — это турникет на входе в кластер. Пока ресурс не прошёл через него, он не считается допустимым. Так проще объяснить идею новичку: не нужно надеяться, что каждый человек каждый раз всё помнит; нужно, чтобы платформа автоматически останавливала опасные варианты.

Стандартные admission-контролы Kubernetes

  • PodSecurity применяет PSS.
  • LimitRanger задаёт defaults и границы для requests/limits.
  • ResourceQuota ограничивает совокупное потребление ресурсов.
  • PodNodeSelector управляет допустимым размещением запускаемых приложений.

Чего не хватает без платформенного слоя

  • Стандартная проверка Kubernetes не знает, из каких registry конкретная организация разрешает брать образы.
  • Она не заставляет команды всегда указывать probes, requests и limits.
  • Она не решает задачу общих для всей платформы правил по тегам образов, способам запуска и согласованным исключениям.

admission-policy-engine в Deckhouse

Deckhouse использует Gatekeeper как основу для централизованного применения политик: правила задаются в одном месте, а платформа проверяет по ним ресурсы перед запуском. Deckhouse оборачивает этот механизм в более удобную для DKP схему. Это позволяет не просто «включить Pod Security», а действительно строить развиваемый и операционно удобный минимальный уровень безопасности.

Что особенно важно в модуле

  • Применение PSS по namespace и глобально.
  • Поддержка режимов warn, dryrun и deny: предупреждать, проверять без влияния на запуск или блокировать нарушение.
  • SecurityPolicy для дополнительных ограничений безопасности.
  • OperationPolicy для эксплуатационной дисциплины: обязательные probes, requests/limits, разрешённые registry и запрет нежелательных тегов.
  • Контроль pods/exec и pods/attach для чувствительных зон.

Почему это важно организационно

  • Команды видят единое место, где описаны правила допуска Pod: что будет разрешено, что даст предупреждение, а что будет заблокировано.
  • Можно сначала включить мягкий режим: показывать предупреждения и собирать нарушения, но не ломать запуск. Затем перейти к жёсткому режиму: блокировать ресурсы, которые нарушают правила.
  • Безопасный запуск перестаёт быть серией исключений и становится поведением платформы по умолчанию.
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: admission-policy-engine
spec:
  version: 1
  enabled: true
  settings:
    podSecurityStandards:
      defaultPolicy: Baseline
      enforcementAction: Warn
apiVersion: deckhouse.io/v1alpha1
kind: OperationPolicy
metadata:
  name: platform-defaults
spec:
  enforcementAction: Warn
  match:
    namespaceSelector:
      labelSelector:
        matchLabels:
          operation-policy.deckhouse.io/enabled: "true"
  policies:
    allowedRepos:
      - registry.example.org
      - registry.deckhouse.io
    requiredResources:
      requests: [cpu, memory]
      limits: [memory]
    requiredProbes:
      - livenessProbe
      - readinessProbe
    disallowedImageTags:
      - latest

Важно не пропускать поле match. В этом примере OperationPolicy начнёт проверять только те namespace, которые явно размечены как прикладные и готовые к этой политике.

apiVersion: v1
kind: Namespace
metadata:
  name: payments
  labels:
    operation-policy.deckhouse.io/enabled: "true"

Как внедрять политики без поломки среды

Самая частая причина сопротивления политикам безопасности — слишком резкое внедрение правил. Если сразу включить блокировку в production, часть приложений может перестать выпускаться или перезапускаться. Проблема будет не в самой идее проверок, а в том, что команда не успела увидеть нарушения, исправить манифесты и договориться об исключениях.

  1. Начать с инвентаризации namespace и классификации запускаемых приложений: системные, платформенные, пользовательские, legacy.
  2. В ModuleConfig модуля admission-policy-engine включить уровень PSS Baseline в режиме Warn.
  3. В манифестах приложений исправить типовые нарушения: запуск от root, отсутствующие probes, отсутствующие requests/limits, тег latest.
  4. Для новых namespace сразу вводить более строгий набор требований, потому что там ещё нет старых приложений, которые могут сломаться от новых правил.
  5. Когда поток предупреждений стабилизируется, переводить критичные namespace в режим deny, то есть блокировать новые нарушения.
  6. Только после этого поднимать часть namespace до restricted.
Правильное внедрение политики допуска почти всегда эволюционно. Сначала команда видит нарушения и исправляет типовые проблемы, затем платформа начинает блокировать новые нарушения. Иначе безопасность будет восприниматься как источник поломок, а не как нормальная возможность платформы.

Мини-кейс: legacy-приложение и строгая политика

Ситуация

Старое приложение запускается от root, пишет во временные каталоги корневой файловой системы и не имеет readiness/liveness probes — проверок готовности и работоспособности. Команда хочет быстро перевести namespace в restricted.

Разумный путь в DKP

Здесь есть три разных места, где команда меняет настройки.

  1. В конфигурации модуля admission-policy-engine включить уровень PSS Baseline в режиме Warn. Это значит: платформа уже проверяет новые Pod, но пока только показывает предупреждения и не блокирует запуск.
  2. В OperationPolicy для пользовательских namespace описать эксплуатационные требования: обязательные probes, requests/limits, разрешённые registry и запрет тега latest. Так новые манифесты не будут проходить без этих полей.
  3. В манифестах самого приложения, например в Deployment или Pod spec, постепенно убрать запуск от root: добавить runAsNonRoot, явно задать runAsUser/runAsGroup, проверить запись во временные каталоги.
  4. Когда предупреждения разобраны и приложение запускается с новыми ограничениями, перевести критичные namespace в deny и только после этого поднимать требования до restricted.
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: admission-policy-engine
spec:
  version: 1
  enabled: true
  settings:
    podSecurityStandards:
      defaultPolicy: Baseline
      enforcementAction: Warn
apiVersion: deckhouse.io/v1alpha1
kind: OperationPolicy
metadata:
  name: application-defaults
spec:
  enforcementAction: Warn
  match:
    namespaceSelector:
      labelSelector:
        matchLabels:
          operation-policy.deckhouse.io/enabled: "true"
  policies:
    requiredResources:
      requests: [cpu, memory]
      limits: [memory]
    requiredProbes:
      - readinessProbe
      - livenessProbe
    disallowedImageTags:
      - latest

А запуск от root, probes и ресурсы исправляются не в OperationPolicy, а в манифесте самого приложения. Обычно это Deployment.spec.template.spec, потому что именно из шаблона Deployment создаёт Pod.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: legacy-api
  namespace: payments
spec:
  replicas: 1
  selector:
    matchLabels:
      app: legacy-api
  template:
    metadata:
      labels:
        app: legacy-api
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 10001
        runAsGroup: 10001
      containers:
        - name: app
          image: registry.example.org/payments/legacy-api:v1.4.2
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              memory: 256Mi
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop: ["ALL"]
            seccompProfile:
              type: RuntimeDefault

Если сразу поставить deny + restricted, платформа начнёт блокировать существующие релизы. Для разработки это будет выглядеть не как понятное требование к манифестам, а как внезапная остановка поставки.

Типовые ошибки

  • Оставлять privileged по историческим причинам и забывать вернуться к пересмотру.
  • Считать, что PSS полностью заменяет Security Context и общие правила платформы.
  • Включать политики только для новых приложений, навсегда исключая старые критичные namespace.
  • Не контролировать allowed registries, image tags и probes, считая это «не про безопасность».
  • Рассматривать warn как финальное состояние, а не как переходный этап.

Лабораторная работа

Задача

  1. Для namespace orders и payments определить стартовый набор политик и режим применения: только предупреждать или уже блокировать нарушения.
  2. Описать правила, которые запрещают запуск от root, privilege escalation, небезопасные hostPath и отсутствие ресурсов.
  3. Добавить эксплуатационные требования: разрешённые registry, запрет latest, обязательные probes и лимиты.
  4. Подготовить отдельный путь для legacy-приложения, которое пока не совместимо с restricted.

Ожидаемый результат

  • Набор политик допуска для пользовательских namespace.
  • План эволюционного rollout от warn к deny.
  • Список технического долга по приложениям, которые мешают поднять минимальный уровень требований.

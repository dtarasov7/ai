NeuVector и внешние security-платформы вокруг DKP

Встроенные механизмы DKP закрывают базовый слой безопасности, но в некоторых средах нужен отдельный container security product: runtime protection, behavioral learning, registry scanning, compliance reporting, policy export и единая консоль для SOC. В этой главе мы разбираем NeuVector и отдельно фиксируем, что известно из публичных материалов про Luntry.

Когда нужен отдельный container security product

Внешняя security-платформа полезна, когда команде недостаточно отдельных сигналов и нужен единый слой анализа поведения, runtime protection, policy learning, compliance reporting и интеграции с SOC.

Если нужна задача Встроенного DKP может хватить Внешняя платформа становится полезной
Scanning образов и CIS reports operator-trivy, reports, Grafana. Нужна приоритизация по бизнес-контексту, registry scanning и единые отчёты для нескольких кластеров.
Runtime detection runtime-audit-engine и Falco rules. Нужно поведенческое обучение, процессные/file/network policies и enforcement по группам workloads.
SOC и расследование API audit, log-shipper, SIEM integration. Нужна единая консоль, risk scoring, форензика и richer context по workload.

Что даёт NeuVector

В Deckhouse NeuVector доступен как модуль, но его lifecycle stage указан как Experimental. Это означает: перед production-внедрением нужно проверять поддержку вашей редакции DKP, release channel, SLA и совместимость с текущей версией платформы.

Основные домены

  • Runtime protection: обнаружение и предотвращение подозрительного поведения контейнеров.
  • Vulnerability management: scanning образов и registry integration.
  • Network security: discovery связей, microsegmentation и network policy generation.
  • Compliance и governance: CIS/PCI/HIPAA-like reports, audit trail, risk scoring.
  • DevSecOps integration: admission control и policy as code.

Что важно не перепутать

  • NeuVector не отменяет необходимость RBAC, NetworkPolicy, PSS и secret management в DKP.
  • Discover mode не равен защите: он учится и наблюдает, но не блокирует.
  • Protect mode нельзя включать вслепую: сначала нужен learning period, тестовый трафик и review правил.

Архитектура NeuVector

Компонент Роль Что проверить перед внедрением
Controller Центральное управление политиками, событиями и сканированием. HA, persistent storage, backup конфигурации.
Manager Web UI для dashboard, policy management и расследований. Аутентификация через user-authn, доступ групп, TLS.
Enforcer DaemonSet на узлах, который наблюдает и применяет runtime/network policies. Ресурсы на каждом node, совместимость с workload и сетевыми требованиями.
Scanner Сканирование образов и registry. Доступ к registry, расписание, нагрузка, false positives.

Включение модуля в DKP

Минимальная форма включения выглядит так. Для production важнее не сам факт enabled: true, а настройки storage, HA, HTTPS и доступа.

apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: neuvector
spec:
  version: 1
  enabled: true
  settings:
    highAvailability: true
    controller:
      storageClass: nfs-rwx
    https:
      mode: CertManager
      certManager:
        clusterIssuerName: letsencrypt

Если user-authn включён, NeuVector использует централизованную аутентификацию DKP. Если user-authn выключен, используется локальная аутентификация NeuVector, что почти всегда хуже для управляемости доступа.

Discover, Monitor и Protect

Режим Что делает Как использовать безопасно
Discover Изучает поведение приложения и генерирует правила. Запустить на тестовом или ограниченном production-scope, дать приложению реальный трафик.
Monitor Показывает нарушения, но не блокирует. Использовать для review правил, false positives и согласования с владельцами приложения.
Protect Блокирует нарушения сетевых, процессных или файловых правил. Включать поэтапно, с rollback и понятным каналом поддержки для команд.

Пример политики NeuVector CRD

В учебном примере политика переводит группу workload в режим Monitor. Это хороший промежуточный режим: события уже видны, но production ещё не ломается блокировкой.

apiVersion: neuvector.com/v1
kind: NvSecurityRule
metadata:
  name: payments-monitor
  namespace: payments
spec:
  target:
    policymode: Monitor
    selector:
      name: nv.payments-api.payments
      criteria:
        - key: service
          op: "="
          value: payments-api
        - key: domain
          op: "="
          value: payments
  process_profile:
    baseline: basic
    mode: Monitor
  ingress:
    - action: allow
      name: orders-to-payments
      selector:
        name: nv.orders-api.orders
        criteria:
          - key: service
            op: "="
            value: orders-api
          - key: domain
            op: "="
            value: orders
      ports: "tcp/5678"
      applications: ["HTTP"]
      priority: 0

Production-правила нужно сверять с CRD reference вашей версии NeuVector. Для курса важна идея: внешняя платформа должна пройти такой же путь, как admission policies — сначала наблюдение, потом controlled enforcement.

Luntry и Deckhouse

По публичным материалам Luntry позиционируется как Kubernetes-native платформа security observability и защиты контейнерных сред на всём жизненном цикле. В 2022 году Luntry опубликовала сообщение о подтверждённой совместимости с Deckhouse по итогам совместных испытаний инженеров «Флант» и Luntry.

Что известно из публичных материалов

  • Функции: безопасность образов, SBOM, vulnerability scanning, контроль Kubernetes-ресурсов, анализ audit log, runtime protection, FIM, сетевые карты, генерация NetworkPolicy, анализ RBAC, CIS checks.
  • Установка описывается как набор образов и Helm chart; публичный FAQ говорит о 10-15 минутах установки.
  • Решение on-premise и может работать в закрытом контуре; данные не должны уходить за периметр клиента.
  • Сенсор использует eBPF для снятия системных вызовов с nodes; производительность зависит от количества syscall, а не входящего трафика.

Что нужно запросить у вендора

  • Актуальную матрицу совместимости с версиями DKP, Kubernetes, CRI и Linux kernel.
  • Требования к privileged permissions, host mounts, eBPF, SELinux/AppArmor и Pod Security exemptions.
  • Helm values для DKP, список создаваемых RBAC-ролей и namespace labels.
  • Интеграции с Gatekeeper/OPA, SIEM, registry, CI/CD и DKP logging.
  • Сценарий обновления, backup/restore и работа в закрытом контуре.
В курс нельзя честно включить закрытую пошаговую установку Luntry без вендорской документации и конкретной версии. Зато важно научить слушателя задавать правильные вопросы: какие права получит сенсор, что он отправляет, как обновляется, как интегрируется с DKP audit/logging и как не дублирует уже включённые механизмы платформы.

Как выбирать между встроенным и внешним решением

  1. Сначала включить и понять DKP built-ins: admission, RBAC, NetworkPolicy, Trivy, runtime audit, API audit, log forwarding.
  2. Описать gap: чего именно не хватает — поведенческий baseline, registry scanning, network map, policy generation, SOC workflow, compliance reporting.
  3. Проверить blast radius внешнего агента: privileges, host access, eBPF, ресурсы на node, storage, network paths.
  4. Запустить пилот на одном некритичном проекте и только потом переводить режимы в enforcement.
  5. Сравнить владение: кто отвечает за rules, false positives, обновления, incident response и связь с командами приложений.

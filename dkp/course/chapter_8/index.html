Обязательный минимум

  • Объяснить, почему namespace сам по себе не является полноценной границей безопасности.
  • Понимать роль multitenancy-manager, Project и ProjectTemplate.
  • Создать проект с quota, network isolation, security profile и администраторами.
  • Понимать ограничения модуля: стадия Preview и один namespace на проект.

Углубление и применимость

  • Проектировать project template как контракт между платформой и прикладной командой.
  • Выбирать между шаблонами default, secure и secure-with-dedicated-nodes.
  • Понимать, что tenant boundary легко случайно расширить через соседние delivery-контуры, если не описать границы проекта явно.
  • Понимать, какие защитные механизмы должны быть в шаблоне, а какие остаются процессом команды.

Где эта глава в сквозном сценарии

После IAM и сетевых политик у shop-demo уже есть отдельные правила. Но платформа зрелее, если умеет выдавать такие правила как готовый проект: команда получает namespace, quota, сетевую изоляцию, профиль Pod security, администраторов и включённые проверки одним согласованным действием.

ProjectTemplate
      |
      v
Project payments
      |
      +--> Namespace payments
      +--> ResourceQuota / LimitRange
      +--> NetworkPolicy
      +--> RBAC / project administrators
      +--> scanning / audit / logging defaults

Почему namespace недостаточно

Namespace удобно группирует объекты Kubernetes, но он не включает безопасность автоматически. Если создать только namespace, внутри него по умолчанию не появятся quota, сетевые запреты, logging, audit, vulnerability scanning или понятная модель администраторов.

Что namespace даёт

  • Логическую группировку объектов.
  • Область действия для Role, RoleBinding, ResourceQuota, NetworkPolicy.
  • Удобную точку для label и selector-based политик.

Чего namespace не даёт сам

  • Не ограничивает ресурсы Pod по умолчанию.
  • Не запрещает сетевые обращения к соседним namespace.
  • Не запрещает hostPath, privileged Pod или опасные настройки контейнера.
  • Не включает logging, audit и scanning как обязательный стандарт проекта.
Tenant boundary ломается не только прямым доступом в namespace. Его можно случайно расширить и через соседние delivery-контуры или автоматизацию. В курсе отдельный разбор этой проблемы вынесен в главу 5, чтобы не смешивать тему GitOps-границ с темой проектных шаблонов.

Что делает multitenancy-manager

multitenancy-manager добавляет в DKP понятие проекта. Проект похож на namespace, но создаётся не как пустая папка для ресурсов, а как управляемое рабочее пространство с заранее описанными настройками безопасности и эксплуатации.

apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: multitenancy-manager
spec:
  version: 1
  enabled: true
  settings:
    allowNamespacesWithoutProjects: false

Если allowNamespacesWithoutProjects: false, платформа запрещает создавать namespace вне проектной модели. Это сильное решение: оно помогает убрать ручные “временные namespace”, но требует, чтобы шаблоны проектов были готовы заранее.

ProjectTemplate как продуктовый контракт

ProjectTemplate отвечает на вопрос: что именно получает команда, когда просит новый проект. Встроенные шаблоны DKP закрывают типовые сценарии.

Шаблон Что даёт Когда подходит
default Resource limits, network isolation, alerts/log collection, security profile, administrators. Стандартные прикладные команды без специальных требований к dedicated nodes.
secure Всё из default плюс допустимые UID/GID, audit rules и scanning запущенных образов. Чувствительные сервисы и production-проекты, где нужны stronger defaults.
secure-with-dedicated-nodes Всё из secure плюс node selector и tolerations для размещения на выделенных узлах. Проекты с регуляторными, производительными или изоляционными требованиями к node placement.
Для курса важно не запоминать названия шаблонов, а понять принцип: безопасный проект должен создаваться как предсказуемый продуктовый контракт, а не как ручная последовательность команд.

Создание Project на практике

В этом примере команда payments получает проект из шаблона default. Здесь видно, где задаются quota, сетевой режим, Pod security profile и группа администраторов.

apiVersion: deckhouse.io/v1alpha2
kind: Project
metadata:
  name: payments
spec:
  description: "Payment services for shop-demo"
  projectTemplateName: default
  parameters:
    resourceQuota:
      requests:
        cpu: 5
        memory: 5Gi
        storage: 10Gi
      limits:
        cpu: 8
        memory: 8Gi
    networkPolicy: Isolated
    podSecurityProfile: Restricted
    extendedMonitoringEnabled: true
    administrators:
      - subject: Group
        name: payments-admins

После применения такого объекта DKP создаёт namespace с именем проекта и ресурсы из шаблона. Проверять нужно не только сам Project, но и фактически созданные ResourceQuota, NetworkPolicy, роли и labels namespace.

d8 k get projects payments
d8 k get resourcequota,networkpolicy,rolebinding -n payments

Минимальный custom template

Свой шаблон нужен, когда встроенный default или secure не выражает локальные правила платформы: например, всем проектам нужен обязательный default deny, одинаковая квота и заранее созданная роль для администраторов проекта.

apiVersion: deckhouse.io/v1alpha1
kind: ProjectTemplate
metadata:
  name: shop-secure
spec:
  description: "Secure template for shop-demo teams"
  parametersSchema:
    openAPIV3Schema:
      type: object
      properties:
        administrators:
          type: array
          items:
            type: object
            properties:
              subject:
                type: string
              name:
                type: string
  resourcesTemplate: |
    ---
    apiVersion: v1
    kind: ResourceQuota
    metadata:
      name: project-quota
    spec:
      hard:
        requests.cpu: "4"
        requests.memory: 8Gi
        limits.memory: 12Gi
    ---
    apiVersion: networking.k8s.io/v1
    kind: NetworkPolicy
    metadata:
      name: default-deny
    spec:
      podSelector: {}
      policyTypes:
        - Ingress
        - Egress
    ---
    apiVersion: rbac.authorization.k8s.io/v1
    kind: RoleBinding
    metadata:
      name: project-admins
    subjects:
      {{- range .parameters.administrators }}
      - kind: {{ .subject }}
        name: {{ .name }}
      {{- end }}
    roleRef:
      apiGroup: rbac.authorization.k8s.io
      kind: ClusterRole
      name: d8:project-admin

Это учебный скелет, а не production-шаблон. В реальной среде нужно также описывать schema всех параметров, роли администраторов, исключения, logging, labels для scanning и правила обновления шаблона. Labels namespace лучше проверять на текущей версии DKP: если шаблон их не задаёт, добавьте отдельный манифест или процедуру маркировки проекта после создания.

Ограничения и ошибки

  • multitenancy-manager находится в lifecycle stage Preview, поэтому перед production-внедрением нужно сверить статус модуля в вашей версии DKP.
  • Один Project создаёт один namespace. Если команде нужны несколько namespace, обычно создают несколько проектов или пересматривают модель.
  • secret-copier несовместим с multitenancy-подходом: копирование секретов во все namespace ломает смысл изоляции.
  • Изменение ProjectTemplate влияет на уже созданные проекты. Это удобно для стандартизации, но опасно без staged rollout и проверки конфликтов.
  • Не делайте слишком широкие шаблоны или delivery-исключения значением по умолчанию: чужие namespace и platform-level ресурсы - это уже изменение общей платформы, а не только своего проекта.
  • Не стоит превращать шаблон в свалку всего на свете. Хороший шаблон задаёт безопасный минимум, а не пытается угадать каждое приложение.

Задача

  1. Спроектировать project template для frontend, orders и payments.
  2. Определить quota, network isolation, Pod security profile и группы администраторов.
  3. Описать, какие labels включают scanning, logging и policy enforcement.
  4. Подготовить один Project для payments и один custom ProjectTemplate.

Ожидаемый результат

  • Матрица team -> project -> template -> security defaults.
  • YAML для Project и черновик ProjectTemplate.
  • Список того, что шаблон задаёт автоматически, а что остаётся в ответственности команды.

Что DKP даёт из коробки, а что делает команда

DKP из коробки Что должна сделать команда
Даёт multitenancy-manager, Project и ProjectTemplate. Определить, какие проектные типы нужны организации и какие security defaults входят в каждый тип.
Даёт встроенные шаблоны default, secure, secure-with-dedicated-nodes. Проверить, подходят ли они локальным требованиям, или подготовить собственный шаблон.
Автоматизирует создание ресурсов проекта из шаблона. Сопровождать изменения шаблонов как production-change: review, staged rollout, rollback и communication plan.

Argo CD и operator-argo: контролируемый доступ к приложениям, GitOps и новые риски

В этой главе Argo CD рассматривается не как "ещё один модный deployment tool", а как отдельный слой доступа и изменений. Он может убрать у dev/test прямой kubectl без потери удобства диагностики, но взамен добавляет новый привилегированный контур: со своим UI, API, токенами, RBAC, repo credentials и ошибками настройки. Задача этой главы - научиться видеть обе стороны сразу.

Что вы сможете после главы

Обязательный минимум

  • Объяснить, зачем Argo CD вообще появляется в курсе по безопасности, а не только в курсе по CI/CD.
  • Понять, когда Argo CD позволяет убрать прямой kubectl у dev/test без потери диагностики.
  • Различать, какие действия через Argo CD допустимы для обычной команды, а какие должны остаться только у platform/SRE.
  • Читать argocd-rbac-cm и AppProject как описание границ доступа.
  • Понимать риск auto-sync, self-heal, exec, override и слишком широкого AppProject.

Углубление и применимость

  • Оценивать, нужен ли Argo CD как пользовательский слой доступа именно в вашей среде, или прямой kubectl под жёсткие компенсирующие меры пока честнее и дешевле.
  • Связывать GitOps, auditability, drift detection и supply chain risk в одну инженерную картину.
  • Отдельно проектировать границы production и non-production для синхронизации и rollback.
  • Понимать, что operator-argo добавляет не только удобство, но и новый административный контур, который нужно сопровождать и обновлять.

Эта глава закрывает разрыв между "мы умеем выдавать RBAC" и "мы умеем не раздавать лишний доступ в кластер, не ломая людям работу".

Зачем теме Argo CD нужна отдельная глава

Argo CD часто ошибочно воспринимают просто как инструмент автоматического деплоя. Для курса по безопасности этого мало. Как только вы начинаете использовать его не только для синхронизации манифестов, но и как пользовательский интерфейс для разработчиков, тестировщиков и аудиторов, он становится частью security architecture.

Если объяснять совсем просто

  • Kubernetes API - это прямой административный вход в кластер.
  • Argo CD UI - это более узкий пользовательский вход, если вы специально сделали его узким.
  • GitOps - это модель, где желаемое состояние хранится в Git, а не меняется вручную командой kubectl apply.

Почему этого нельзя прятать в примечании

  • Меняется не только способ деплоя, но и модель доступа.
  • Появляется новый высокопривилегированный сервис.
  • Появляется второй слой RBAC поверх Kubernetes RBAC.
  • Ошибки в AppProject и sync-политиках меняют blast radius не хуже ошибок в обычном RBAC.

Мини-словарь

  • GitOps - подход, при котором нужное состояние приложений хранится в Git, а кластер приводится к этому состоянию автоматически или полуавтоматически.
  • Drift - расхождение между тем, что описано в Git, и тем, что реально работает в кластере.
  • AppProject - объект Argo CD, который задаёт, из каких репозиториев можно брать манифесты, в какие namespace можно разворачивать приложения и какие типы ресурсов разрешены.
  • Repo credentials - учётные данные, которыми Argo CD пользуется для доступа к Git-репозиторию или registry.
  • auto-sync - автоматическое применение изменений из Git без ручного подтверждения.
  • self-heal - автоматическое возвращение к состоянию из Git, если кто-то изменил объект в кластере вручную.
  • exec в Argo CD - право открыть интерактивную сессию внутрь контейнера. По сути это близко к kubectl exec.
  • Break-glass - заранее описанный аварийный путь доступа, когда обычный путь через Argo CD, Git или SSO недоступен.

Что делает operator-argo в DKP

Модуль operator-argo разворачивает в кластере Argo CD Operator. Дальше платформа создаёт конкретные инстансы Argo CD через ресурс ArgoCD. Для слушателя здесь важна не только эксплуатационная деталь "как ставится", но и смысл: появляется отдельный слой управления приложениями, который платформа должна сопровождать как security-sensitive компонент.

Что это даёт

  • Более предсказуемую установку и обновление Argo CD через платформенный модуль.
  • Интеграцию с SSO Deckhouse через Dex.
  • Единый UI для просмотра приложений, статуса синхронизации, истории и логов.

Что это не отменяет

  • Нужду отдельно проектировать RBAC самого Argo CD.
  • Нужду отдельно ограничивать источники манифестов и целевые namespace.
  • Нужду обновлять и проверять сам Argo CD как отдельный привилегированный сервис.
apiVersion: deckhouse.io/v1alpha1
kind: ModuleConfig
metadata:
  name: operator-argo
spec:
  enabled: true
  version: 1

Практическая мысль здесь простая: как только вы включили operator-argo, вы не "чуть-чуть упростили delivery", а добавили в платформу ещё один контур, который теперь должен пройти через review прав, review конфигурации и цикл обновлений.

Какие трудности Argo CD действительно снимает

Что становится лучше

  • Можно перестать массово раздавать kubeconfig и прямой kubectl разработчикам и тестировщикам.
  • Обычная диагностика переезжает в контролируемый UI: видны статус, события, дерево ресурсов и логи.
  • Изменения лучше аудируются: есть Git-история desired state и история синхронизации.
  • Проще замечать drift между "как должно быть" и "как стало на самом деле".

Что именно это значит в жизни

  • У dev/test меньше причин просить "временный read-only kubeconfig".
  • У platform/SRE меньше ручной рутины вроде "скиньте, пожалуйста, логи из pod".
  • У аудитора появляется более наблюдаемый путь просмотра приложений, чем прямой вход в кластер.
Важно: Argo CD решает задачу только тогда, когда вы даёте через него действительно более узкий доступ. Если через UI вы почти полностью вернули те же возможности, что и через kubectl, выигрыш по безопасности быстро исчезает.

Модель контролируемого доступа к приложениям

Хорошая учебная модель выглядит так: доступ к самим приложениям и их диагностике отделяется от административного доступа к кластеру. Для чтения и повседневной диагностики используется Argo CD. Для администрирования кластера, аварийных действий и platform-level изменений остаётся прямой kubectl.

Группа Основной интерфейс Что разрешено Что остаётся запрещено
shop-dev Argo CD UI / CLI статус приложения, дерево ресурсов, история синхронизации, логи kubectl, exec, изменение AppProject, sync в production
shop-test Argo CD UI статус rollout, события, логи kubectl, exec, доступ к чужим namespace
shop-auditors Argo CD UI + отчёты read-only просмотр приложений и связанной evidence любые изменения, repo credentials, cluster-scoped действия
platform-sre kubectl + Argo CD admin администрирование, break-glass, настройка оператора и sync-политик неформальные исключения без журнала и review

Сильная сторона этой модели в том, что она не путает "диагностировать приложение" и "администрировать кластер". Это разные виды доступа, и их не надо искусственно склеивать в одну роль.

RBAC Argo CD: что можно давать, а что лучше не давать

Argo CD имеет собственную систему RBAC. Это отдельный слой поверх Kubernetes RBAC, и именно поэтому он требует отдельного проектирования. Для dev/test безопасный учебный baseline обычно очень узкий: видеть приложения и видеть логи. Всё остальное даётся только после отдельного решения.

apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  policy.default: role:none
  policy.csv: |
    p, role:shop-dev-readonly, applications, get, shop-demo/*, allow
    p, role:shop-dev-readonly, logs, get, shop-demo/*, allow
    p, role:shop-auditor-readonly, applications, get, shop-demo/*, allow
    p, role:shop-auditor-readonly, logs, get, shop-demo/*, allow
    g, shop-dev, role:shop-dev-readonly
    g, shop-test, role:shop-dev-readonly
    g, shop-auditors, role:shop-auditor-readonly

Почему это хороший минимум

  • Есть право видеть приложения.
  • Есть право видеть логи.
  • Нет широкого доступа "всем аутентифицированным пользователям".

Что особенно легко испортить

  • policy.default слишком широкая - и лишний доступ получают сразу все.
  • exec - это уже почти возврат kubectl exec через UI.
  • override - это путь применить произвольную ревизию или манифест в обход обычной Git-дисциплины.

Почему AppProject - это граница полномочий

Очень частая ошибка - считать AppProject просто удобной папкой для приложений. На самом деле он описывает security boundary: откуда можно брать манифесты, куда их можно применять и какие типы объектов вообще разрешены.

apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: shop-demo
  namespace: argocd
spec:
  sourceRepos:
    - https://git.example.org/platform/shop-demo.git
  destinations:
    - namespace: frontend
      server: https://kubernetes.default.svc
    - namespace: orders
      server: https://kubernetes.default.svc
    - namespace: payments
      server: https://kubernetes.default.svc
  clusterResourceWhitelist: []
  namespaceResourceWhitelist:
    - group: apps
      kind: Deployment
    - group: ""
      kind: Service
    - group: ""
      kind: ConfigMap
    - group: networking.k8s.io
      kind: Ingress
  namespaceResourceBlacklist:
    - group: ""
      kind: Secret

Что этот пример делает правильно

  • Разрешает ровно один репозиторий.
  • Разрешает только свои namespace.
  • Не даёт cluster-scoped ресурсы.
  • Не превращает Git в место хранения обычных Secret.

Чем опасен слишком широкий проект

  • Команда может получить обходной путь к платформенным изменениям не через kubectl, а через GitOps.
  • Можно случайно разрешить чужие namespace.
  • Можно разрешить cluster-scoped объекты и тем самым фактически менять не приложение, а весь кластер.

Какие новые риски появляются

Argo CD уменьшает одну группу рисков и одновременно создаёт другую. Это нормально. Ошибка не в появлении риска как такового, а в том, что команда не замечает, что он появился.

Риски, которые уменьшаются

  • Массовая раздача kubeconfig и прямого доступа к API.
  • Ручные ad hoc действия через kubectl apply без понятного review.
  • Слабая наблюдаемость изменений и drift.

Риски, которые добавляются

  • Компрометация Argo CD, Git или repo credentials может привести к компрометации деплоя.
  • Появляется ещё один публичный или корпоративный UI/API endpoint.
  • Ошибки в Argo RBAC и Kubernetes RBAC накапливаются, а не взаимно отменяются.

Особенно неприятные места

  • Слишком широкий AppProject - даёт лишние права через GitOps.
  • exec и override - возвращают через UI то, что вы вроде бы хотели не раздавать через kubectl.
  • Git как новая точка атаки - вредоносный commit, компрометация PR-пайплайна или утечка repo token становятся напрямую значимыми для кластера.
  • Зависимость от SSO и Argo CD - если они недоступны, нужен заранее описанный break-glass путь.
Отдельная тонкость operator-based модели: неаккуратная настройка управляемых namespace может неожиданно дать слишком широкие полномочия внутри этих namespace. Это ещё одна причина, почему platform-команда не должна отдавать обычным командам право менять саму модель Argo CD.

Решение по auto-sync и self-heal

Эти функции легко выглядят как "просто удобная автоматизация", но для production это уже вопрос границы доверия. Если Git скомпрометирован или в merge попала ошибка, то автоматический sync быстро превращает её в реальное изменение среды.

Среда auto-sync self-heal Разумный учебный baseline
Production обычно выключен обычно выключен Git + review + ручное подтверждение sync platform/SRE
Non-production может быть включён может быть включён допустим после проверки команды и границ проекта

Правильный вывод здесь не "автоматизация плоха". Правильный вывод другой: для разных сред цена ошибки разная, значит и sync-политика не обязана быть одинаковой.

Когда Argo CD может не окупиться

Argo CD не обязан быть универсально правильным ответом. Иногда честнее оставить прямой kubectl очень узкому кругу людей и обложить его компенсирующими мерами, чем вводить новый привилегированный сервис, который команда не готова сопровождать.

Когда Argo CD обычно полезен

  • Команд много, а прямой kubectl хочется резко сократить.
  • Нужен единый self-service просмотр приложений для dev/test.
  • GitOps уже является основной delivery-моделью.
  • Есть platform-команда, которая реально будет сопровождать новый контур.

Когда он может принести больше вреда, чем пользы

  • Среда маленькая и команд мало.
  • Нет зрелого Git-process и нет SSO-дисциплины.
  • Команда не готова сопровождать ещё один привилегированный сервис.
  • В реальности через Argo CD всё равно планируют выдать почти те же права, что и через прямой kubectl.

Поэтому хороший инженерный ответ здесь бывает двух типов: "да, Argo CD нужен, потому что он реально сужает доступ" или "нет, пока не нужен, потому что сейчас это будет дорогой дублирующий контур". Плохой ответ только один: "поставили, потому что так делают все".

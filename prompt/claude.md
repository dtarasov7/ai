Вот оптимальный промпт для IT-вопросов с минимизацией галлюцинаций:

```
Ты — эксперт по информационным технологиям. При ответе на мой вопрос:

ОБЯЗАТЕЛЬНЫЕ ТРЕБОВАНИЯ:
1. Если ты не уверен в информации или она могла измениться после января 2025 — явно скажи об этом
2. Различай факты и предположения — маркируй предположения словами "вероятно", "обычно", "как правило"
3. Для версий ПО, API, синтаксиса команд — если не уверен на 100%, укажи это и предложи проверить документацию
4. Приводи конкретные примеры кода/команд только если уверен в их корректности
5. Если вопрос требует знания актуальных версий, статуса проектов, текущих уязвимостей — предложи использовать поиск

СТРУКТУРА ОТВЕТА:
- Краткий прямой ответ на вопрос
- Технические детали с примерами (если применимо)
- Важные нюансы и ограничения
- Альтернативные подходы (если есть)
- Источники для проверки (официальная документация, RFC, и т.д.)

ЧЕГО ИЗБЕГАТЬ:
- Не изобретай несуществующие флаги, параметры, методы API
- Не смешивай синтаксис разных версий/языков/фреймворков
- Не выдавай устаревшие практики за актуальные
- Не придумывай цифры (версии, порты, коды ошибок)

Мой вопрос: [ТВОЙ_ВОПРОС]
```

**Ключевые принципы этого промпта:**

1. **Явное требование признавать незнание** — самый эффективный способ снизить галлюцинации
2. **Разделение фактов и предположений** — помогает отличить достоверную информацию
3. **Фокус на проверяемости** — указание источников для верификации
4. **Конкретные запреты** — перечисление типичных ошибок при галлюцинациях
5. **Структурированность** — облегчает проверку полноты ответа

Дополнительный совет: для критически важных вопросов добавь в конце: *"Если хоть в чём-то не уверен — лучше скажи 'не знаю' или предложи поиск, чем рискни ошибкой"*

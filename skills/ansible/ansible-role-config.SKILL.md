# Шаблон Ansible-роли: config

Используй этот навык для создания новой роли production-уровня по единому шаблону для ролей класса `config`.

Этот skill подходит, если роль:

- не отвечает за полную установку приложения;
- конфигурирует уже существующее ПО;
- разворачивает шаблоны, policy-файлы, include/snippet-конфиги;
- может создавать systemd override или environment-файл;
- вызывает reload/restart через handlers;
- проверяет корректность конфигурации.

Если роль устанавливает приложение или поднимает новый сервис, используй skill для `service`.
Если роль настраивает саму ОС, security baseline, sysctl, sshd, sudoers, audit и другие системные политики, используй skill для `baseline/hardening`.

## Что делать

1. Создай роль со структурой:
   - `defaults/main.yml`
   - `handlers/main.yml`
   - `meta/main.yml`
   - `tasks/main.yaml`
   - `tasks/pre-req.yaml`
   - `tasks/config.yaml`
   - `tasks/clean.yaml`
   - `templates/`
   - `molecule/default/molecule.yml`
   - `molecule/default/converge.yml`
   - `molecule/default/verify.yml`
   - `README.md`

2. Все переменные именуй с префиксом роли:
   - `<role_name>_...`
   - для роли с дефисом заменяй `-` на `_`

3. В начале каждой задачи указывай имя файла:
   - `config.yaml - ...`
   - `pre-req.yaml - ...`

4. В `tasks/main.yaml` импортируй файлы с тегами:
   - `pre-req`
   - `config`
   - `clean`

5. Для `clean.yaml` обязательно:
   - теги `never` и `clean`
   - в конце `meta: end_play`

## Правила по содержимому

### `pre-req.yaml`

- Проверяй ОС, минимальную версию и наличие целевого ПО, если роль предполагает, что оно уже установлено.
- Для RedOS не полагайся только на `ansible_os_family == "RedHat"`.
- Принимай `RedHat`, `redos` и совместимые EL-подобные варианты, если это соответствует роли.
- Проверяй наличие обязательных директорий, сервисов, конфигурационных путей и переменных.
- Если роль конфигурирует systemd override или drop-in, проверяй, что целевой unit действительно существует или его существование корректно обосновано.

### `config.yaml`

- Разворачивай конфигурацию, snippets, include-файлы, policy-файлы и environment-файлы через `template`.
- Если целевое ПО поддерживает валидацию конфигурации, используй `validate` или отдельную безопасную pre-check-задачу перед перезапуском.
- Если роль управляет systemd override/drop-in или unit-файлом, разворачивай их через `template`.
- Не переустанавливай пакеты и не бери на себя bootstrap, если это не входит явно в область ответственности роли.
- Все изменения должны быть максимально идемпотентными.
- Изменения конфигурации должны вызывать handlers.

### `clean.yaml`

- Удаляй только те конфигурационные файлы, drop-in, snippets, environment-файлы и policy-файлы, которые создавались ролью.
- Не удаляй пакеты, данные приложения, системных пользователей и общие каталоги, если это не было явно создано этой ролью и не описано как часть её ответственности.
- Для удаления особенно чувствительных файлов используй защитные флаги.
- После destructive-части завершай play через `meta: end_play`.

## Handlers

В `handlers/main.yml`:

- выноси restart/reload в handlers;
- где возможно, предпочитай `reload` вместо `restart`;
- если меняется systemd unit/override, выполняй `daemon-reload` через handler;
- используй понятные имена handlers с префиксом роли.

## README

`README.md` должен быть на русском языке и содержать:

- назначение роли;
- класс роли: `config`;
- описание всех переменных;
- список тегов;
- список handlers;
- описание того, какое ПО и какие пути роль ожидает уже существующими;
- пример inventory;
- пример playbook;
- описание Molecule-сценария;
- описание cleanup-поведения и ограничений.

## Molecule

- Используй Docker driver.
- Если образ уже подготовлен, ставь `pre_build_image: true`.
- Не добавляй сборку образа без необходимости.
- Для systemd-контейнеров используй `privileged: true`, `command: /usr/sbin/init`, `tmpfs` для `/run` и `/tmp`, монтирование `cgroup`.
- Если контейнер запускается как `root`, не указывай `become: true` в `converge.yml` и `verify.yml`, если в образе не гарантирован `sudo`.
- В `converge.yml` подготовь минимально достаточное окружение для демонстрации того, что целевое ПО уже существует.
- Не превращай config-role в install-role внутри Molecule: допускается только минимальная подложка для теста.

## Проверки для `verify.yml`

Проверяй не только наличие файла, но и применённость конфигурации:

- конфигурационный файл создан в нужном месте;
- права и владелец корректны;
- содержимое соответствует ожидаемым параметрам;
- если есть validate-команда, она проходит;
- сервис reload/restart проходит успешно, если это часть роли;
- целевое ПО принимает новую конфигурацию;
- API или прикладная проверка проходит, если это разумно для роли.

## Идемпотентность и качество

Считай хорошей только такую роль, которая:

- не переходит в install/bootstrap без прямой необходимости;
- не удаляет чужие или системные конфиги;
- повторно запускается без лишних изменений;
- использует `validate`, где это возможно;
- использует handlers вместо прямых restart;
- не раскрывает секреты и чувствительные параметры в output.

## Критерий готовности

Считай роль готовой, только если:

- структура файлов соблюдена;
- переменные и теги оформлены по шаблону;
- `README.md` написан на русском;
- handlers оформлены отдельно и используются по назначению;
- `molecule test` проходит полностью;
- verify проверяет не только файл, но и принятие конфигурации сервисом или приложением;
- cleanup ограничен только артефактами роли.

Если нужны детали или готовые формулировки, открой [SKILLS.reference.md](./SKILLS.reference.md).

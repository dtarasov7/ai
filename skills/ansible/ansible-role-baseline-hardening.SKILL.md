# Шаблон Ansible-роли: baseline / hardening

Используй этот навык для создания новой роли production-уровня по единому шаблону для ролей следующих классов:

- `baseline`
- `hardening`

Этот skill подходит, если роль:

- настраивает саму ОС или базовый профиль хоста;
- управляет users/groups, sudoers, sshd, sysctl, limits, repos, time sync, journald, audit, firewall, crypto policies и похожими системными настройками;
- реализует security baseline или hardening;
- может существенно влиять на доступность узла и безопасность.

Если роль устанавливает или сопровождает приложение/сервис, используй skill для `service`.
Если роль только настраивает уже установленное прикладное ПО, используй skill для `config`.

## Что делать

1. Создай роль со структурой:
   - `defaults/main.yml`
   - `handlers/main.yml`
   - `meta/main.yml`
   - `tasks/main.yaml`
   - `tasks/pre-req.yaml`
   - `tasks/config.yaml`
   - `tasks/clean.yaml`
   - `templates/`
   - `molecule/default/molecule.yml`
   - `molecule/default/converge.yml`
   - `molecule/default/verify.yml`
   - `README.md`

2. Все переменные именуй с префиксом роли:
   - `<role_name>_...`
   - для роли с дефисом заменяй `-` на `_`

3. В начале каждой задачи указывай имя файла:
   - `pre-req.yaml - ...`
   - `config.yaml - ...`

4. В `tasks/main.yaml` импортируй файлы с тегами:
   - `pre-req`
   - `config`
   - `clean`

5. Для `clean.yaml` обязательно:
   - теги `never` и `clean`
   - в конце `meta: end_play`

6. Для hardening-роли:
   - все потенциально опасные изменения делай максимально явно;
   - destructive или рискованные действия защищай флагами;
   - не ломай доступ к хосту без явного подтверждения через переменные роли.

## Правила по содержимому

### `pre-req.yaml`

- Проверяй ОС, минимальную версию, особенности init/system manager и платформенные ограничения.
- Для RedOS не полагайся только на `ansible_os_family == "RedHat"`.
- Принимай `RedHat`, `redos` и совместимые EL7/EL8/EL9-подобные варианты, если это соответствует роли.
- Проверяй обязательные переменные и конфликтующие режимы работы.
- Если роль меняет SSH, firewall, sudoers, PAM, crypto policy или network-sensitive настройки, делай дополнительные safety-checks до внесения изменений.
- Если роль рассчитана на определённый класс хостов, проверяй это явно.

### `config.yaml`

- Используй штатные модули Ansible для users, groups, authorized keys, sysctl, lineinfile, blockinfile, copy, template, package, service и других системных сущностей.
- Для конфигов применяй `template` или другой наиболее безопасный модуль.
- Для sudoers используй валидацию через `visudo`, где это возможно.
- Для sshd используй безопасную проверку конфигурации перед reload/restart, если это возможно.
- Для sysctl, journald, audit, limits и похожих системных параметров обеспечивай идемпотентное применение.
- Не смешивай baseline/hardening с установкой сложных прикладных сервисов.
- Все изменения, требующие restart/reload, должны вызывать handlers.

### `clean.yaml`

- По умолчанию cleanup должен быть максимально консервативным.
- Не удаляй пользователей, группы, sysctl, sudoers, ssh-ключи, firewall rules, audit-policy или другие системные настройки без отдельного явного флага.
- Для опасных действий используй отдельные переменные вида:
  - `<role_name>_clean_confirm: false`
  - `<role_name>_clean_remove_users: false`
  - `<role_name>_clean_remove_security_policy: false`
- Cleanup не должен превращаться в безусловный откат системы в неизвестное состояние.
- После destructive-части завершай play через `meta: end_play`.

## Handlers

В `handlers/main.yml`:

- выноси restart/reload сервисов и daemon-reload в handlers;
- для security-sensitive сервисов применяй restart/reload только после успешной проверки конфигурации;
- используй понятные имена handlers с префиксом роли;
- не делай лишние restart там, где достаточно reload или применения через специализированный модуль.

## README

`README.md` должен быть на русском языке и содержать:

- назначение роли;
- класс роли: `baseline` или `hardening`;
- описание всех переменных;
- список тегов;
- список handlers;
- описание поддерживаемых платформ и версий;
- описание потенциально опасных изменений;
- описание safety-флагов и cleanup-флагов;
- пример inventory;
- пример playbook;
- описание Molecule-сценария;
- описание ограничений и рисков применения.

## Molecule

- Используй Docker driver, если сценарий реалистично тестируем в контейнере.
- Если образ уже подготовлен, ставь `pre_build_image: true`.
- Не добавляй сборку образа без необходимости.
- Для systemd-контейнеров используй `privileged: true`, `command: /usr/sbin/init`, `tmpfs` для `/run` и `/tmp`, монтирование `cgroup`.
- Если контейнер запускается как `root`, не указывай `become: true` в `converge.yml` и `verify.yml`, если в образе не гарантирован `sudo`.
- Если часть hardening-проверок нереалистична или небезопасна в Docker, честно ограничь сценарий в README и проверяй только то, что воспроизводимо.
- Не имитируй успешность без реальной проверки результата.

## Проверки для `verify.yml`

Проверяй не только наличие файлов, но и эффект применения политики:

- нужные пользователи, группы и каталоги существуют;
- sudoers проходит валидацию;
- sshd-конфигурация валидна;
- sysctl-параметры реально применены;
- limits, journald, audit и другие политики применены в ожидаемом объёме;
- сервисы перезапускаются или перечитывают конфигурацию только там, где это необходимо;
- роль не ломает базовую доступность контейнера или хоста в пределах тестового сценария.

Для hardening-роли дополнительно:

- проверяй, что рискованные параметры применились именно так, как ожидается;
- не считай роль готовой только по наличию файла, если поведение не проверено;
- если изменение потенциально ломает SSH-доступ, верифицируй конфигурацию до применения restart.

## Идемпотентность и безопасность

Считай хорошей только такую роль, которая:

- идемпотентна;
- не ломает доступ к узлу без явного запроса;
- не откатывает системные сущности без подтверждающего флага;
- использует безопасные validate/pre-check механизмы для критичных конфигураций;
- не раскрывает чувствительные значения в output;
- явно документирует риски и ограничения.

## Критерий готовности

Считай роль готовой, только если:

- структура файлов соблюдена;
- переменные и теги оформлены по шаблону;
- `README.md` написан на русском;
- handlers оформлены отдельно и используются по назначению;
- `molecule test` проходит полностью в том объёме, который реалистично воспроизводим;
- verify проверяет не только наличие файлов, но и реальный эффект политик;
- рискованные действия защищены safety-флагами;
- cleanup не удаляет системные сущности без явного подтверждения.

Если нужны детали или готовые формулировки, открой [SKILLS.reference.md](./SKILLS.reference.md).

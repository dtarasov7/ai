# Шаблон Ansible-роли

Используй этот навык для создания новой роли production-уровня по единому шаблону.

## Что делать

1. Создай роль со структурой:
   - `defaults/main.yml`
   - `handlers/main.yml`
   - `meta/main.yml`
   - `tasks/main.yaml`
   - `tasks/pre-req.yaml`
   - `tasks/install.yaml`
   - `tasks/config.yaml`
   - `tasks/user.yaml`
   - `tasks/ssl.yaml`
   - `tasks/clean.yaml`
   - `templates/`
   - `molecule/default/molecule.yml`
   - `molecule/default/converge.yml`
   - `molecule/default/verify.yml`
   - `README.md`

2. Все переменные именуй с префиксом роли:
   - `<role_name>_...`
   - для роли с дефисом заменяй `-` на `_`

3. В начале каждой задачи указывай имя файла:
   - `install.yaml - ...`
   - `config.yaml - ...`

4. В `tasks/main.yaml` импортируй файлы с тегами:
   - `pre-req`
   - `user`
   - `install`
   - `ssl`
   - `config`
   - `clean`

5. Для `clean.yaml` обязательно:
   - теги `never` и `clean`
   - в конце `meta: end_play`

## Правила по содержимому

### `pre-req.yaml`

- Проверяй ОС, минимальную версию, обязательные пакеты и размер/состав кластера при необходимости.
- Для RedOS не полагайся только на `ansible_os_family == "RedHat"`.
- Принимай `RedHat`, `redos` и совместимые EL7-подобные варианты.

### `user.yaml`

- Создавай системного пользователя и группу.
- Создавай каталоги данных, конфигурации, логов и архивов логов.
- Явно задавай `owner`, `group`, `mode`.

### `install.yaml`

- Устанавливай пакеты, архивы или бинарники в управляемые каталоги.
- Выноси версии, URL, пути и имена в `defaults/main.yml`.
- После установки проверяй наличие файлов и режимы доступа.

### `config.yaml`

- Разворачивай конфигурацию и environment-файлы через `template`.
- Разворачивай `systemd` unit-файлы через `template`.
- Настраивай `logrotate`.
- Запускай и включай сервисы через `ansible.builtin.systemd`.

### `ssl.yaml`

- Файл обязателен даже если TLS не используется.
- Если TLS выключен, допускается только проверка параметров и `debug` с `changed_when: false`.
- Если TLS включен, проверяй наличие путей к сертификату и ключу.

### `clean.yaml`

- Удаляй сервисы, unit-файлы, конфигурацию, данные, бинарники, logrotate и пользователей, если они создавались ролью.

## Логирование

Если роль пишет логи в файлы, настраивай `logrotate` со следующими параметрами:

- `daily`
- `rotate 7`
- `maxsize 1G`
- `compress`
- `compresscmd /usr/bin/xz`
- `compressext .xz`
- `olddir <log_dir>/archive`
- `copytruncate`

## README

`README.md` должен быть на русском языке и содержать:

- назначение роли;
- описание всех переменных;
- список тегов;
- пример inventory;
- пример playbook;
- описание Molecule-сценария.

## Molecule

- Используй Docker driver.
- Если образ уже подготовлен, ставь `pre_build_image: true`.
- Не добавляй сборку образа без необходимости.
- Для systemd-контейнеров используй `privileged: true`, `command: /usr/sbin/init`, `tmpfs` для `/run` и `/tmp`, монтирование `cgroup`.
- Если контейнер запускается как `root`, не указывай `become: true` в `converge.yml` и `verify.yml`, если в образе не гарантирован `sudo`.

## Проверки для `verify.yml`

Проверяй не только установку, но и работоспособность:

- сервисы запущены;
- API отвечает;
- данные записываются;
- данные читаются.

Для кластерной роли дополнительно:

- используй минимум 3 узла;
- проверь работоспособность после отказа одного узла;
- используй `uri` с `retries` и `delay`, если сервису нужно время на стабилизацию.

## Критерий готовности

Считай роль готовой, только если:

- структура файлов соблюдена;
- переменные и теги оформлены по шаблону;
- `README.md` написан на русском;
- `molecule test` проходит полностью;
- для кластерной роли проверен сценарий деградации.

Если нужны детали или готовые формулировки, открой [SKILLS.reference.md](./SKILLS.reference.md).

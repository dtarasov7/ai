# Шаблон Ansible-роли: service / cluster_service / package

Используй этот навык для создания новой роли production-уровня по единому шаблону для ролей следующих классов:

- `service`
- `cluster_service`
- `package`

Этот skill подходит, если роль:

- устанавливает пакет, архив или бинарник;
- разворачивает приложение или сервис;
- при необходимости создаёт systemd unit;
- может использовать TLS;
- может быть stateful;
- может работать в кластерном режиме.

Если роль только конфигурирует уже установленное ПО, используй отдельный skill для `config`.
Если роль настраивает ОС, hardening, sysctl, sshd, sudoers, limits, audit и другие системные политики, используй отдельный skill для `baseline/hardening`.

## Что делать

1. Создай роль со структурой:
   - `defaults/main.yml`
   - `handlers/main.yml`
   - `meta/main.yml`
   - `tasks/main.yaml`
   - `tasks/pre-req.yaml`
   - `tasks/install.yaml`
   - `tasks/config.yaml`
   - `tasks/user.yaml`
   - `tasks/ssl.yaml`
   - `tasks/clean.yaml`
   - `templates/`
   - `molecule/default/molecule.yml`
   - `molecule/default/converge.yml`
   - `molecule/default/verify.yml`
   - `README.md`

2. Все переменные именуй с префиксом роли:
   - `<role_name>_...`
   - для роли с дефисом заменяй `-` на `_`

3. В начале каждой задачи указывай имя файла:
   - `install.yaml - ...`
   - `config.yaml - ...`

4. В `tasks/main.yaml` импортируй файлы с тегами:
   - `pre-req`
   - `user`
   - `install`
   - `ssl`
   - `config`
   - `clean`

5. Для `clean.yaml` обязательно:
   - теги `never` и `clean`
   - в конце `meta: end_play`

6. Для package-роли допускается:
   - не создавать systemd unit, если роль устанавливает только CLI/утилиту;
   - не создавать пользователя и каталоги данных, если они объективно не нужны;
   - сохранять файл `user.yaml`, но делать его минимальным и безопасным;
   - сохранять файл `ssl.yaml`, даже если TLS не используется.

7. Для cluster_service-роли обязательно:
   - учитывать минимальный размер кластера;
   - проверять состав кластера в `pre-req.yaml`;
   - делать Molecule-сценарий минимум на 3 узла, если роль реально тестируема в Docker;
   - в `verify.yml` проверять сценарий деградации одного узла.

## Правила по содержимому

### `pre-req.yaml`

- Проверяй ОС, минимальную версию, обязательные пакеты и размер/состав кластера при необходимости.
- Для RedOS не полагайся только на `ansible_os_family == "RedHat"`.
- Принимай `RedHat`, `redos` и совместимые EL7/EL8/EL9-подобные варианты, если это соответствует роли.
- Проверяй обязательные переменные.
- Для package/service ролей проверяй наличие совместимого init/system manager, если он нужен.
- Все проверки делай максимально рано.

### `user.yaml`

- Для service/cluster_service:
  - создавай системного пользователя и группу;
  - создавай каталоги данных, конфигурации, логов и архивов логов;
  - явно задавай `owner`, `group`, `mode`.
- Для package:
  - не создавай пользователя, группу и каталоги, если они не нужны по смыслу роли;
  - если приложение требует runtime-каталоги, оформляй их так же явно.

### `install.yaml`

- Устанавливай пакеты, архивы или бинарники в управляемые каталоги.
- Выноси версии, URL, checksum, пути и имена в `defaults/main.yml`.
- После установки проверяй наличие файлов и режимы доступа.
- По возможности используй штатные Ansible-модули вместо `shell` и `command`.
- Повторный запуск роли не должен приводить к лишним изменениям.

### `config.yaml`

- Разворачивай конфигурацию и environment-файлы через `template`.
- Если роль создаёт systemd unit-файлы, разворачивай их через `template`.
- Если роль пишет логи в файлы, настраивай `logrotate`.
- Запускай и включай сервисы через `ansible.builtin.systemd`, если сервис входит в область ответственности роли.
- Изменения конфигурации, unit-файлов и environment-файлов должны вызывать `handlers`.
- Где возможно, предпочитай `reload` вместо `restart`.

### `ssl.yaml`

- Файл обязателен даже если TLS не используется.
- Если TLS выключен, допускается только проверка параметров и `debug` с `changed_when: false`.
- Если TLS включен, проверяй наличие путей к сертификату, ключу и при необходимости CA.
- Не выводи секретные значения в `debug`.
- Для приватных ключей используй строгие права доступа.

### `clean.yaml`

- Удаляй только те сервисы, unit-файлы, конфигурацию, данные, бинарники, logrotate и пользователей, которые создавались ролью.
- Для stateful-ролей удаление данных должно быть защищено отдельным флагом:
  - `<role_name>_clean_remove_data: false` по умолчанию.
- Удаление пользователя и группы также делай через отдельные флаги, если есть риск shared-использования.
- После destructive-части завершай play через `meta: end_play`.

## Логирование

Если роль пишет логи в файлы, настраивай `logrotate` со следующими параметрами по умолчанию:

- `daily`
- `rotate 7`
- `maxsize 1G`
- `compress`
- `compresscmd /usr/bin/xz`
- `compressext .xz`
- `olddir <log_dir>/archive`
- `copytruncate`

Если приложение корректно поддерживает reopen/reload логов без `copytruncate`, допускается более подходящая настройка, но она должна быть явно обоснована в README.

## Handlers

В `handlers/main.yml`:

- выноси restart/reload сервисов в handlers;
- не делай прямые restart-задачи в основном потоке без необходимости;
- используй понятные имена handlers с префиксом роли;
- если меняется systemd unit, выполняй `daemon-reload` через handler.

## README

`README.md` должен быть на русском языке и содержать:

- назначение роли;
- класс роли: `service`, `cluster_service` или `package`;
- описание всех переменных;
- список тегов;
- список handlers;
- пример inventory;
- пример playbook;
- описание Molecule-сценария;
- если есть destructive cleanup — описание защитных флагов;
- если есть кластер — описание минимального состава и сценария деградации.

## Molecule

- Используй Docker driver.
- Если образ уже подготовлен, ставь `pre_build_image: true`.
- Не добавляй сборку образа без необходимости.
- Для systemd-контейнеров используй `privileged: true`, `command: /usr/sbin/init`, `tmpfs` для `/run` и `/tmp`, монтирование `cgroup`.
- Если контейнер запускается как `root`, не указывай `become: true` в `converge.yml` и `verify.yml`, если в образе не гарантирован `sudo`.
- Если роль не использует systemd, не усложняй Molecule-конфигурацию без нужды.
- Для cluster_service используй минимум 3 узла, если это совместимо с тестируемым ПО и Docker-сценарием.

## Проверки для `verify.yml`

Проверяй не только установку, но и работоспособность:

- сервисы запущены, если они создаются ролью;
- API отвечает, если у сервиса есть API;
- данные записываются;
- данные читаются;
- бинарь доступен и сообщает версию, если это package-role;
- конфигурационные файлы, unit-файлы и runtime-пути реально применены.

Для кластерной роли дополнительно:

- используй минимум 3 узла;
- проверь работоспособность после отказа одного узла;
- используй `uri` с `retries` и `delay`, если сервису нужно время на стабилизацию.

## Идемпотентность и качество

Считай хорошей только такую роль, которая:

- проходит повторный запуск без лишних изменений;
- по возможности использует штатные модули вместо `shell`/`command`;
- не раскрывает секреты в output;
- использует `changed_when` и `failed_when` осмысленно;
- не удаляет данные без явного подтверждающего флага.

## Критерий готовности

Считай роль готовой, только если:

- структура файлов соблюдена;
- переменные и теги оформлены по шаблону;
- `README.md` написан на русском;
- handlers оформлены отдельно и используются по назначению;
- `molecule test` проходит полностью;
- для кластерной роли проверен сценарий деградации;
- destructive-операции защищены флагами.

Если нужны детали или готовые формулировки, открой [SKILLS.reference.md](./SKILLS.reference.md).

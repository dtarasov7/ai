Это пункт шире, чем предыдущий про `runAsUser=0` и `privileged=true`.

**SecurityContext для всех приложений** — это просьба дать сводку, какие настройки безопасности заданы у Pod’ов/контейнеров для каждого приложения.

То есть заказчик хочет не только список «опасных» контейнеров, а общую картину:

```text
приложение → namespace → pod/deployment/statefulset → какие securityContext-настройки используются
```

Обычно смотрят такие параметры:

```yaml
securityContext:
  runAsUser: 1000
  runAsGroup: 1000
  fsGroup: 1000
  runAsNonRoot: true
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  privileged: false
  capabilities:
    drop:
      - ALL
  seccompProfile:
    type: RuntimeDefault
```

Что означают основные поля:

| Поле                       | Смысл                                                      |
| -------------------------- | ---------------------------------------------------------- |
| `runAsUser`                | UID пользователя внутри контейнера. `0` = root             |
| `runAsGroup`               | GID группы внутри контейнера                               |
| `fsGroup`                  | группа для доступа к примонтированным volume               |
| `runAsNonRoot`             | запрещает запуск от root                                   |
| `privileged`               | privileged-режим контейнера                                |
| `allowPrivilegeEscalation` | можно ли процессу повышать привилегии                      |
| `readOnlyRootFilesystem`   | root filesystem контейнера только для чтения               |
| `capabilities`             | Linux capabilities, добавленные или удалённые у контейнера |
| `seccompProfile`           | seccomp-профиль для ограничения системных вызовов          |

Почему написано **«Информация от команды разработки»**: потому что не всё можно корректно понять только из Kubernetes YAML. Например, если `runAsUser` не указан, приложение может всё равно запускаться не от root внутри Docker image. Или наоборот — securityContext не задан, но это осознанное исключение из-за требований приложения.

Пример ответа для аудита:

```md
| Приложение | Namespace | Workload | runAsNonRoot | runAsUser | privileged | allowPrivilegeEscalation | readOnlyRootFilesystem | capabilities | seccompProfile | Комментарий |
|---|---|---|---|---|---|---|---|---|---|---|
| backend-api | app-prod | Deployment/backend-api | true | 1000 | false | false | true | drop: ALL | RuntimeDefault | Соответствует restricted-подходу |
| worker | app-prod | Deployment/worker | true | 1000 | false | false | false | drop: ALL | RuntimeDefault | Нужна запись во временные файлы |
| nginx-ingress | ingress-nginx | DaemonSet/controller | не задан | 101 | false | true | false | NET_BIND_SERVICE | RuntimeDefault | Исключение для ingress |
```

Собрать техническую часть можно так:

```bash
kubectl get deployments,statefulsets,daemonsets --all-namespaces -o yaml
```

Или по уже созданным Pod’ам:

```bash
kubectl get pods --all-namespaces -o yaml
```

Более удобная команда для первичной выписки:

```bash
kubectl get pods --all-namespaces -o json | jq -r '
.items[] as $pod
| $pod.spec.containers[]
| [
    $pod.metadata.namespace,
    $pod.metadata.name,
    .name,
    (.securityContext.runAsUser // "not set"),
    (.securityContext.runAsNonRoot // "not set"),
    (.securityContext.privileged // "not set"),
    (.securityContext.allowPrivilegeEscalation // "not set"),
    (.securityContext.readOnlyRootFilesystem // "not set"),
    (.securityContext.seccompProfile.type // "not set")
  ]
| @tsv
'
```

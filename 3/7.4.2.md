Это пункт про поиск **потенциально опасных Pod’ов/контейнеров**, которые запущены с повышенными правами.

В Kubernetes у каждого Pod’а и контейнера может быть `securityContext`. Там задаётся, от какого пользователя запускается процесс, можно ли получать повышенные привилегии, доступен ли host namespace, можно ли монтировать системные пути и т.д.

Фраза:

```text
Привилегированные контейнеры (runAsUser=0, privileged=true)
```

означает: нужно найти контейнеры, которые:

1. **запущены от root-пользователя**

   ```yaml
   runAsUser: 0
   ```

2. **запущены в privileged-режиме**

   ```yaml
   privileged: true
   ```

Это разные вещи.

`runAsUser: 0` значит, что процесс внутри контейнера работает от пользователя `root`.

`privileged: true` значит, что контейнер получает почти полный доступ к возможностям хоста. Это намного опаснее: такой контейнер может взаимодействовать с kernel capabilities, устройствами, сетевыми настройками и иногда фактически приближаться к уровню доступа на node.

Пример опасной конфигурации:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: example
  namespace: app-prod
spec:
  containers:
    - name: app
      image: example/app:latest
      securityContext:
        runAsUser: 0
        privileged: true
        allowPrivilegeEscalation: true
```

Что хотят получить в ответе для аудита: **список таких Pod’ов/контейнеров**.

Например:

```md
| Namespace | Pod | Container | runAsUser | privileged | Комментарий |
|---|---|---|---|---|---|
| monitoring | node-exporter-abcde | node-exporter | 0 | false | Запущен от root |
| kube-system | cni-plugin-xyz | cni | 0 | true | Privileged, возможно требуется для CNI |
| app-prod | backend-api-123 | backend | не задан | false | Нет признаков privileged |
```

Команда из анкеты:

```bash
kubectl get pods --all-namespaces -o yaml
```

выгружает все Pod’ы в YAML. Дальше в этом YAML ищут поля:

```yaml
securityContext:
  runAsUser: 0
```

и:

```yaml
securityContext:
  privileged: true
```

Более удобные команды:

```bash
kubectl get pods --all-namespaces -o jsonpath='{range .items[*]}{.metadata.namespace}{"\t"}{.metadata.name}{"\n"}{range .spec.containers[*]}{"  "}{.name}{" runAsUser="}{.securityContext.runAsUser}{" privileged="}{.securityContext.privileged}{" allowPrivilegeEscalation="}{.securityContext.allowPrivilegeEscalation}{"\n"}{end}{end}'
```

Отдельно найти `privileged: true` можно так:

```bash
kubectl get pods --all-namespaces -o json | jq -r '
.items[] as $pod
| $pod.spec.containers[]
| select(.securityContext.privileged == true)
| "\($pod.metadata.namespace)\t\($pod.metadata.name)\t\(.name)\tprivileged=true"
'
```

Найти контейнеры с `runAsUser: 0`:

```bash
kubectl get pods --all-namespaces -o json | jq -r '
.items[] as $pod
| $pod.spec.containers[]
| select(.securityContext.runAsUser == 0)
| "\($pod.metadata.namespace)\t\($pod.metadata.name)\t\(.name)\trunAsUser=0"
'
```

Важный нюанс: если `runAsUser` **не указан**, это не всегда значит, что контейнер не root. Тогда пользователь может быть задан внутри Docker image. Для строгого аудита такие контейнеры тоже стоит отметить как **“runAsUser не задан, требуется проверка image / effective user”**.

Коротко для заполнения анкеты:

```text
Необходимо предоставить список Pod’ов/контейнеров, которые запущены от root-пользователя или в privileged-режиме. Проверяется securityContext на уровне Pod и container: runAsUser=0, privileged=true, allowPrivilegeEscalation=true, а также случаи, где runAsUser не задан.
```

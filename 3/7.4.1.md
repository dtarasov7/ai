Это пункт про то, **какие политики безопасности Pod’ов применяются в каждом Kubernetes namespace**.

В Kubernetes есть механизм **Pod Security Admission**. Он управляется через labels на namespace. Поэтому в анкете и написано:

```bash
kubectl get namespaces -o yaml
```

Дальше нужно смотреть labels у каждого namespace.

## Что такое `privileged`, `baseline`, `restricted`

Это уровни строгости **Pod Security Standards**:

| Уровень      | Смысл                                                                                                                |
| ------------ | -------------------------------------------------------------------------------------------------------------------- |
| `privileged` | Почти без ограничений. Можно запускать привилегированные контейнеры, hostPath, hostNetwork и т.п.                    |
| `baseline`   | Базовая защита. Запрещает явно опасные настройки, но оставляет совместимость с большинством приложений.              |
| `restricted` | Самый строгий режим. Требует безопасные настройки контейнеров: non-root, seccomp, запрет privilege escalation и т.д. |

Обычно для production-приложений хорошо видеть `baseline` или `restricted`, а `privileged` — только для системных namespace, где это действительно нужно.

## Какие labels искать

В namespace могут быть такие labels:

```yaml
pod-security.kubernetes.io/enforce: baseline
pod-security.kubernetes.io/audit: restricted
pod-security.kubernetes.io/warn: restricted
```

И иногда с версией:

```yaml
pod-security.kubernetes.io/enforce-version: latest
pod-security.kubernetes.io/audit-version: latest
pod-security.kubernetes.io/warn-version: latest
```

Что они означают:

| Label     | Что делает                                                                 |
| --------- | -------------------------------------------------------------------------- |
| `enforce` | Реально запрещает создание Pod’ов, которые нарушают выбранный уровень      |
| `audit`   | Не запрещает, но пишет нарушения в audit logs                              |
| `warn`    | Не запрещает, но показывает предупреждение пользователю при создании Pod’а |

Например:

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: app-prod
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

Это значит: в namespace `app-prod` применяется строгий профиль `restricted`.

## Как посмотреть кратко

Все namespace с labels:

```bash
kubectl get namespaces --show-labels
```

Более удобно — конкретно Pod Security labels:

```bash
kubectl get ns -o custom-columns='NAME:.metadata.name,ENFORCE:.metadata.labels.pod-security\.kubernetes\.io/enforce,AUDIT:.metadata.labels.pod-security\.kubernetes\.io/audit,WARN:.metadata.labels.pod-security\.kubernetes\.io/warn'
```

Пример вывода:

```text
NAME          ENFORCE      AUDIT        WARN
default       baseline     restricted   restricted
kube-system   privileged   privileged   privileged
app-prod      restricted   restricted   restricted
monitoring    baseline     restricted   restricted
```

## Что писать в ответе для аудита

Можно заполнить так:

```text
Проверены labels namespaces pod-security.kubernetes.io/*. 
Для namespace app-prod установлен enforce=restricted, audit=restricted, warn=restricted.
Для namespace monitoring установлен enforce=baseline, audit=restricted, warn=restricted.
Для kube-system используется privileged, так как namespace содержит системные компоненты Kubernetes.
```

Или, если labels отсутствуют:

```text
Pod Security Admission labels на namespaces не настроены. 
Labels pod-security.kubernetes.io/enforce, audit, warn отсутствуют.
Фактические ограничения Pod Security Standards на уровне namespaces не применяются.
```

Смысл пункта: **заказчик хочет увидеть таблицу namespace → применяемый уровень безопасности Pod Security Standards**.

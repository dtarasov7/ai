## 1) Что это за программа и для чего она

`packet-monitor` — консольный (TUI) анализатор сетевого трафика:

* **Live-capture**: захват пакетов с одного интерфейса или с `any` (несколько интерфейсов).
* **Offline**: анализ из файла 
* **Просмотр пакетов**: таблица пакетов с основными полями + `info`.
* **Flows/Conversations**: таблица потоков (агрегация по 5-tuple), сортировки, переход к пакету/деталям потока.
* **Детали пакета**: подробный просмотр содержимого (raw/hex/info и т.д.).
* **Follow stream**:

  * TCP: reassembly payload по seq (best effort, с дедупликацией ретрансляций и placeholder’ами gap).
  * TLS: handshake summary (SNI/ALPN/ServerHello ver/cipher, Alerts) — без расшифровки.
  * UDP/прочее: “классический” follow stream как фильтр (по ip/port/proto).
* **Фильтрация** (уровень UI): по proto/ip/port/interface/info/payload.
* **Исключения**: исключить поток (stream) или протокол.
* **Профили**: сохранение/загрузка наборов фильтров + исключений в `~/.packet-monitor/config.json`.
* **BPF**: capture-фильтр на уровне захвата (до попадания в приложение).

## 2) Быстрый старт

### 2.1 Запуск с захватом

Самый простой:

* запусти под root (или с правами на sniff)
* выбери интерфейс (если параметр есть) или по умолчанию “основной”
* в UI сразу начнут появляться пакеты

Примеры BPF :

* только HTTPS:

  * `tcp port 443`
* только DNS:

  * `udp port 53`
* только трафик к хосту:

  * `host 192.168.1.10`
* DNS + HTTPS:

  * `udp port 53 or tcp port 443`

### 2.2 Структура экрана

* **Левая панель**: либо список пакетов, либо таблица потоков (Flows view).
* **Правая панель**: статистика.
* **Нижняя строка**: статус (интерфейс/режим, total/displayed, фильтр) + хоткеи.

## 3) Горячие клавиши (основные)

### Общие

* `F1` / `H` — Help
* `Q` — Quit
* `P` — Pause/Resume capture (в live режиме)
* `C` — Clear (очистить буфер пакетов)
* `A` — Auto-scroll on/off
* `S` — Save/Export (если реализовано через SaveDialog)
* `/` — Search по payload
* `F` — Filter dialog (UI-фильтры)
* `O` — Profiles (Save/Load/Delete + Presets)
* `G` — Flows view (переключение packets ↔ flows)

### В режиме Flows

* `V` — переключить источник flows: **FILTERED** vs **ALL**
* `D` — Flow details
* `Enter` — drill-down: применить фильтр по выбранному flow и вернуться в packets
* `T` — Follow stream (TCP → stream view / TLS handshake summary)
* `X` — Exclude flow
* `B` / `K` / `R` — sort flows (bytes / pkts / rate) 

### В режиме Packets

* `Enter` — Packet details
* `T` — Follow stream (TCP → reassembly view, UDP → фильтр по 4-tuple)
* `X` — Exclude stream или protocol (в зависимости от выбранного пакета)
* `E` — Exclude manager

## 4) Фильтрация: как работает и как правильно ожидать результат

### 4.1 UI-фильтры (F)

Диалог фильтров содержит поля (примерно):

* `proto`
* `src_ip`
* `dst_ip`
* `any_ip` (если уже добавлен)
* `src_port`
* `dst_port`
* `any_port` (если уже добавлен)
* `interface`
* `info` (regex supported)
* `payload` (regex search)

**Важно:**
Если ты задаёшь одновременно `src_ip=...` и `dst_ip=...`, это означает **AND** (оба условия).
Чтобы увидеть “src ИЛИ dst = 192.168.1.23”, используй:

* `any_ip = 192.168.1.23` (самый правильный)
  или
* regex: `src_ip=(192\.168\.1\.23)` + `dst_ip=(192\.168\.1\.23)` **не подойдёт**, это AND.

### 4.2 Подбор значений из захваченных пакетов (L/F2)

В диалоге фильтра ты добавлял “picker”: показать список всех значений из уже захваченных пакетов (например для src_ip/dst_ip/any_ip/ports/interface).

Ожидаемое поведение:

* выбираешь поле
* жмёшь `L` (или `F2` если переназначено)
* видишь список `value (count)`
* `Enter` — вставляет значение в поле фильтра

## 5) Flows/Conversations view (G)

### 5.1 Что такое flow в программе

Flow агрегируется по 5-tuple:

* proto (TCP/UDP)
* endpoint A (ip:port)
* endpoint B (ip:port)

Отображаются:

* pkts
* bytes total
* bytes A→B / B→A (направления относительно канонических A/B)
* first_seen / last_seen
* TCP признаки: SYN/FIN/RST counters

### 5.2 Источник flows

* **FILTERED** — только пакеты, прошедшие текущие UI-фильтры
* **ALL** — все захваченные пакеты (игнорируя фильтр для агрегации flows)

Переключение: `V`

### 5.3 Drill-down

Нажми `Enter` на flow:

* программа применит фильтры так, чтобы в packets list остались пакеты именно этого flow.

### 5.4 Exclude flow

Нажми `X` на flow:

* поток добавится в исключения (и исчезнет из таблиц)
* управлять исключениями можно через `E`

### 5.5 Flow details (D)

`D` откроет окно с деталями потока (длительность, pps/bps, top ports и т.п. — зависит от текущей реализации).

## 6) Follow stream (T) — “как в Wireshark” (то что уже есть)

### 6.1 TCP stream view

Если выбран TCP flow/packet и нажать `T`:

* определяется client/server (по SYN без ACK, если найден)
* собирается TCP payload по направлениям:

  * Client → Server
  * Server → Client
* делается reassembly по `seq`:

  * сегменты сортируются
  * дубликаты/ретрансляции убираются
  * при пропусках вставляется placeholder вида `[...] missing N bytes [...]` (или эквивалент)
* отображение: UTF-8/ASCII с fallback в hex
* можно экспортировать:

  * текст
  * raw bytes по направлениям (`*-c2s.bin`, `*-s2c.bin`)

### 6.2 TLS handshake view (в Follow stream)

Если поток похож на TLS (по record headers) — вместо обычного TCP stream view открывается TLS handshake summary:

Примеры строк:

* `ClientHello SNI=google.com ALPN=h2,http/1.1`
* `ServerHello ver=TLS1.3 cipher=0x1301`
* `Alert fatal handshake_failure`
* `ApplicationData (encrypted)` — если уже пошёл шифрованный трафик

> Важно: содержимое HTTP поверх TLS **увидеть нельзя** без MITM/ключей/SSLKEYLOGFILE и т.п.
> Здесь показывается только handshake-метаинформация.

### 6.3 UDP “follow stream”

Для UDP `T` действует как “быстро применить фильтр на 4-tuple” (показывать только этот обмен).

## 7) Packet details (Enter на пакете)

Окно деталей обычно показывает:

* заголовок (номер/время/интерфейс/адреса/порты/протокол)
* info summary
* raw/hex dump
* дополнительные поля (tcp flags/seq/ack, packet_loss marker)

> Ты планировал навигацию по пакетам прямо внутри details — пока (как ты писал) стрелки используются для прокрутки. Если добавишь PgUp/PgDn для навигации — это будет удобнее.

## 8) BPF Capture Filter — что даёт и как использовать

BPF фильтр применяется **на уровне захвата**, т.е. пакеты, не подходящие фильтру, **вообще не попадут** в буфер.

Плюсы:

* меньше CPU и RAM
* удобнее при высоком трафике

Минусы:

* то, что отфильтровали, уже не восстановить внутри программы

Примеры:

* `tcp port 443` — только HTTPS (увидишь ClientHello/ServerHello summary)
* `tcp port 9100` — конкретный сервис
* `host 10.0.0.5 and tcp` — только TCP к хосту

## 9) Профили (O)

Профиль — сохранённый набор:

* filters (`proto/src_ip/.../payload`)
* исключения (`exclude_streams`, `exclude_protocols`)

### 9.1 Где хранится

`~/.packet-monitor/config.json`

### 9.2 Что можно делать

* Save As: сохранить текущие фильтры/исключения
* Load: загрузить профиль
* Delete: удалить
* Presets:

  * `dns-only`
  * `syn-only` (правильный: `tcp_syn=True`, `tcp_ack=False`)
  * `arp-only`

# 10) User Case (сценарии использования)

## Сценарий 1: “Быстро понять, кто шумит в сети” (top flows)

**Цель:** за минуту увидеть “самые жирные” потоки.

1. Запусти capture без BPF.
2. Нажми `G` (Flows view).
3. Переключи источник `V` → `ALL` (если хочешь видеть всё).
4. Отсортируй по bytes (`B`) или rate (`R`).
5. Выбери подозрительный flow → `Enter` (drill-down) → смотри пакеты.
6. Если это “мусор” → вернись в flows → `X` (Exclude flow).

**Результат:** быстро чистишь “шум” и концентрируешься на важном.

---

## Сценарий 2: “Найти DNS ответы и понять, куда резолвится домен”

**Цель:** увидеть DNS ответы прямо в списке пакетов.

1. Поставь пресет `O` → `dns-only` (или `F` → proto=DNS).
2. Смотри поле `info`: `DNS example.com | Ans: A:..., AAAA:..., CNAME:...`
3. Если нужно только ответы — можно фильтровать `info` по `Ans:`.

---

## Сценарий 3: “Проверить TLS рукопожатие: SNI/ALPN/версия/шифр”

**Цель:** увидеть SNI и ALPN при подключении.

1. Задай BPF: `tcp port 443`
2. В другом окне:

   * `curl https://google.com`
3. В списке пакетов ищи:

   * `TLS ClientHello SNI=google.com ALPN=h2,http/1.1`
   * `TLS ServerHello ver=TLS1.3 cipher=...`
4. Нажми `T` на потоке → откроется TLS handshake summary view.

---

## Сценарий 4: “Follow TCP stream и увидеть HTTP без TLS”

**Цель:** собрать HTTP запросы/ответы (если трафик не шифрован).

1. Фильтр (или BPF): `tcp port 80`
2. Открой сайт по http или сделай `curl http://…`
3. Выбери пакет → `T`
4. В stream view увидишь текст запросов/ответов по направлениям.

---

## Сценарий 5: “Найти пакет по payload (например токен/строка)”

**Цель:** найти в данных конкретную строку.

1. Нажми `/`
2. Введи строку (можно regex)
3. Apply → если найдено, фильтр выставится на payload и останутся только совпадения.

---

## Сценарий 6: “SYN-only для диагностики сканирования/подключений”

**Цель:** видеть только попытки TCP соединений.

1. `O` → preset `syn-only`
2. В packets list увидишь только SYN без ACK (инициации соединений)
3. Перейди в flows (`G`) и смотри куда идут подключения.

---

## Сценарий 7: “У меня проблема только с одним сервисом — минимизировать шум”

**Цель:** мониторить конкретный порт/сервис.

1. Установи BPF: `tcp port 9100` (node-exporter) или нужный порт.
2. В flows list смотри только relevant.
3. Если нужно убрать регулярный шум — `X` исключить поток.

---

# 11) Практические советы

* Если трафика много — начинай с **BPF**, это экономит ресурсы.
* Для “src или dst” используй `any_ip`, для портов — `any_port`.
* Когда “ничего не видно” — проверь, что ты не поставил два поля, которые дают AND (src_ip и dst_ip одновременно).
* Если “TLS ничего не показывает” — handshake мог быть разрезан на несколько TCP сегментов: с буферизацией оно должно появляться; если нет — увеличь объём буфера или проверь, что вынимаешь payload корректно на твоём интерфейсе.



### Troubleshooting 

#### 1) Пустой экран после установки фильтра

**Причина:** фильтры в диалоге `Filter` применяются одновременно (логика **AND**).
Например, если задать **и** `src_ip=192.168.1.23`, **и** `dst_ip=192.168.1.23`, то будут показаны только пакеты, где **оба** поля равны этому адресу — обычно таких нет → пусто.

**Что делать:**

* чтобы получить “src **или** dst = 192.168.1.23”, используйте поле `any_ip=192.168.1.23`
* чтобы получить “src_port **или** dst_port = 9100”, используйте `any_port=9100`

---

#### 2) Не вижу TLS SNI/ALPN/ServerHello (при HTTPS)

**Причины и решения:**

* **Монитор запущен после соединения.** TLS рукопожатие происходит в самом начале.

  * Решение: запустите `packet-monitor` **до** `curl/браузера`, затем повторите запрос.
* **Слишком строгий BPF / фильтр.**

  * Решение: попробуйте BPF `tcp port 443` и убедитесь, что HTTPS трафик реально идёт.

**Где смотреть:**

* В списке пакетов в колонке `Info` должны появляться строки вида:

  * `TLS ClientHello SNI=... ALPN=...`
  * `TLS ServerHello ver=... cipher=...`

---

#### 3) “Follow Stream” не показывает содержимое HTTPS (HTTP внутри TLS)

Это ожидаемо: **HTTPS трафик зашифрован**.
`packet-monitor` без ключей и без MITM **не расшифровывает** ApplicationData, поэтому в лучшем случае покажет:

* TLS handshake summary (SNI/ALPN/версия/шифр/alerts)
* `ApplicationData (encrypted)` после завершения handshake

**Что делать, если нужно видеть HTTP-текст:**

* используйте незашифрованный HTTP (например порт 80) или отдельный прокси/инструменты расшифровки (в рамках текущего `packet-monitor` это не включено).

---

#### 4) Нажимаю Follow Stream, но вижу “пусто/мало данных”

**Причины:**

* соединение короткое, payload почти нет (например только handshake)
* выбран пакет не из нужного TCP потока

**Что делать:**

* перейдите в `Flows` (клавиша `G`), выберите нужный поток и нажмите `T` (Follow stream) именно на нём
* проверьте, что это действительно TCP поток (для UDP Follow Stream работает как “быстрый фильтр”)

---

#### 5) Пакеты не захватываются (в live режиме)

**Типовые причины:**

* недостаточно прав на захват (обычно нужен root)
* выбран не тот интерфейс
* BPF фильтр отсекает всё

**Что делать:**

* запустите с правами администратора: `sudo ...`
* проверьте правильность интерфейса (например `ip a`)
* временно отключите BPF или поставьте простой `tcp` / `udp`

---

#### 6) В режиме Flows “ничего нет”, а в пакетах есть

**Причины:**

* включён источник flows = `FILTERED`, а фильтр сейчас слишком строгий
* выбран протокол/пакеты не TCP/UDP с портами (flows строятся по TCP/UDP потокам)

**Что делать:**

* в Flows нажмите `V` и переключите источник на `ALL`
* очистите фильтры (`Filter` → Clear All), либо загрузите профиль без фильтров

---

#### 7) Почему TLS summary виден как “отдельный пакет”, рядом с ACK?

Это нормально.
Один пакет может быть просто ACK (без полезной нагрузки), а следующий содержит часть TLS handshake. Поэтому в списке пакетов это отображается как разные строки.

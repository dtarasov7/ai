# Вариант 3. Гибрид: короткий онлайн-контур и архив с обычным restore

## Суть варианта

В production-кластере остается только рабочий онлайн-горизонт, который нужен для повседневного поиска и быстрого DR. Более старая история выгружается в архивные `snapshot repository` и при необходимости восстанавливается в отдельный recovery-контур обычным `local restore`.

Это хороший компромисс по стоимости и DR, если допустимо, что часть пятилетней истории не находится постоянно онлайн.

## Модель данных

Как и во втором варианте, нужно учитывать не один лог, а много семейств дневных индексов:

- `fw-YYYY.MM.DD`
- `proxy-YYYY.MM.DD`
- `auth-YYYY.MM.DD`
- `os-YYYY.MM.DD`
- `app-YYYY.MM.DD`

Архивируется не один индекс, а комплект индексов за период по всем типам логов.

## Архитектура

### Рекомендуемый горизонт хранения

Пример деления:

1. `0-30 дней`:
- hot;
- постоянно онлайн.

2. `31-180 дней`:
- warm;
- онлайн для обычного расследования.

3. `181-365 дней`:
- cold;
- онлайн, но на более дешевых ресурсах.

4. `Старше 365 дней`:
- удаляется из production-кластера после финального snapshot;
- хранится в архивных `snapshot repository`;
- поднимается отдельно при необходимости.

### Компоненты

1. Production-кластер OpenSearch 2.19:
- хранит только рабочий онлайн-горизонт.

2. Активный short-term `snapshot repository`:
- частые snapshot текущих write-индексов;
- используется для быстрого DR.

3. Архивные `snapshot repository`:
- содержат финальные snapshot закрытых индексов;
- лучше делить по годам или кварталам, чтобы ограничивать рост metadata.

4. Внешняя СРК:
- резервирует active short-term `snapshot repository`;
- резервирует архивные `snapshot repository`.

5. Recovery-контур:
- отдельный временный кластер для поднятия архивных периодов обычным restore.

## Логика snapshot и backup

### Логические snapshot внутри OpenSearch

1. `Hourly DR snapshot` в short-term `snapshot repository`:
- каждые 30-60 минут;
- включает текущие write-индексы всех семейств логов;
- retention 7-30 дней.

2. `Daily sealed snapshot` в архивный `snapshot repository`:
- 1 раз в сутки;
- включает закрытые индексы предыдущего дня по всем типам логов;
- retention 5 лет.

3. После достижения порога возраста индекс удаляется из production-кластера, но остается в архивном `snapshot repository`.

### Резервное копирование snapshot repository внешней СРК

#### Short-term snapshot repository

- режим: `synthetic full weekly + daily incremental` или `full weekly + daily incremental`;
- retention: 30-45 дней;
- задача: быстрый DR после потери NFS.

#### Архивные snapshot repository

- режим: `full monthly + daily incremental` для активного архивного `snapshot repository`;
- retention в оперативном backup-контуре: 90 дней;
- дополнительный архивный tier backup: весь срок хранения данных плюс минимум 1 год запаса.

Если архивный `snapshot repository` разбит по годам, можно после завершения года:

- перестать писать в старый `snapshot repository`;
- считать его "закрытым";
- делать контрольный full backup этого `snapshot repository` в архивный backup-tier.

## Поиск по запросу регулятора без даты

Это ключевой operational downside варианта 3. Если дата неизвестна, а старшая часть истории не онлайн, поиск придется выполнять поэтапно.

### Последовательность действий

1. Сначала выполнить поиск в production-кластере по онлайн-горизонту:
- по всем релевантным семействам индексов;
- через alias или index pattern;
- асинхронно, если объем большой.

2. Если требуется полный поиск за все 5 лет, определить список архивных периодов:
- например `2022`, `2023`, `2024`, `2025`.

3. Для каждого периода:
- при необходимости восстановить соответствующий архивный `snapshot repository` из СРК на NFS;
- поднять recovery-кластер;
- зарегистрировать нужный `snapshot repository`;
- восстановить только нужные семейства индексов за этот период;
- выполнить поиск;
- выгрузить и сохранить результаты.

4. После окончания поиска по периоду:
- удалить восстановленные индексы из recovery-кластера;
- перейти к следующему периоду;
- либо поднимать 2-3 периода параллельно, если позволяет инфраструктура.

5. После прохода по всем годам:
- объединить результаты;
- убрать дубликаты;
- отсортировать по времени;
- подготовить общий отчет регулятору.

### Что это означает practically

Да, в варианте 3 для полного поиска без даты могут понадобиться:

- место под восстановленный `snapshot repository`, если он поднимается из backup;
- место под восстановленные индексы на recovery-кластере;
- время на restore `snapshot repository` из СРК;
- время на restore индексов из snapshot.

## DR-сценарий

1. Развернуть новый production-кластер.
2. Подключить short-term `snapshot repository`.
3. Восстановить последние часы или дни и вернуть прием логов.
4. Исторический архив не должен блокировать возврат сервиса.
5. Архивные периоды восстанавливаются отдельно позже.

## Что делать, если за 5 лет меняется версия OpenSearch

В архивной схеме это особенно важно, потому что старые snapshot могут пережить несколько major upgrade production-кластера.

### Практический подход

1. Делить архивные `snapshot repository` по годам или кварталам.
2. На каждый major upgrade иметь restore test хотя бы по одному snapshot из каждого архивного периода.
3. Если snapshot слишком старый:
- поднять temporary intermediate-кластер совместимой major-версии;
- восстановить данные;
- reindex или снять новый snapshot;
- перенести архив в новый `snapshot repository`.
4. Хранить runbook и automation для разворота совместимого кластера старой major-версии.

Без такого процесса пятилетний архив нельзя считать гарантированно восстанавливаемым.

## Плюсы

- Намного дешевле, чем держать все 5 лет онлайн.
- Самый важный DR-контур маленький и восстанавливается быстро.
- Старую историю можно восстанавливать выборочно.
- Хорошо соответствует требованию "сначала вернуть функциональность, потом данные".

## Минусы

- Поиск без даты по всем 5 годам может требовать год-за-годом восстанавливать архив.
- Нужны отдельные recovery-процессы и отдельное место под restore.
- Нужны время и ресурсы и на restore `snapshot repository`, и на restore индексов.
- Совместимость старых snapshot с новыми major-версиями нужно регулярно поддерживать.

## Вывод

Вариант 3 рационален, если важны быстрый DR и снижение стоимости production, а подъем исторического архива по запросу допустим. Главный недостаток этого варианта проявляется именно в регуляторном поиске без даты.

# Вариант 1. Все 5 лет данных онлайн, snapshot всего кластера

## Суть варианта

В этом варианте вся история за 5 лет остается в production-кластере OpenSearch и постоянно доступна для чтения. Защита строится через единый активный `snapshot repository` на NFS, в который регулярно снимается snapshot всего кластера, а сам `snapshot repository` затем дополнительно резервируется внешней СРК.

Это самый простой для понимания вариант, но самый тяжелый по стоимости, размеру кластера и скорости DR.

## Модель данных

Важно исходить из того, что в OpenSearch хранится не один лог и не один индекс, а множество логовых потоков и множество индексов:

- сетевые логи;
- системные логи;
- audit-логи;
- прикладные логи;
- технические служебные индексы OpenSearch.

Практически это означает:

- snapshot снимается не с одного индекса, а со всех нужных семейств индексов;
- поиск по объекту регулятора должен уметь обходить несколько index pattern или alias;
- naming и шаблоны индексов должны быть унифицированы, например `fw-*`, `os-*`, `app-*`, `audit-*`.

## Архитектура

### Компоненты

1. Production-кластер OpenSearch 2.19:
- hot/warm/cold tiers;
- все индексные семейства за 5 лет остаются онлайн.

2. Активный `snapshot repository`:
- отдельный NFS-сервер;
- один экспортируемый каталог;
- одинаковый mount path на всех узлах;
- `path.repo` прописан на всех узлах OpenSearch.

3. Внешняя СРК:
- резервирует файловую структуру активного `snapshot repository`;
- хранит свои копии в backup-хранилище;
- используется как защита от потери самого `snapshot repository`.

### Базовая конфигурация

```yaml
path.repo:
  - /mnt/opensearch-snapshots
```

```json
PUT /_snapshot/repo-v1-active
{
  "type": "fs",
  "settings": {
    "location": "/mnt/opensearch-snapshots/repo-v1-active",
    "compress": true,
    "max_snapshot_bytes_per_sec": "200mb",
    "max_restore_bytes_per_sec": "300mb"
  }
}
```

## Логика snapshot и backup

### Логические snapshot внутри OpenSearch

Чтобы приблизиться к целевому `RPO <= 1 час`, вариант 1 в production-исполнении уже нельзя оставлять только с ежедневным snapshot. Реалистичный режим такой:

1. `Hourly cluster snapshot`:
- каждые 30-60 минут;
- snapshot всего набора production-индексов;
- retention 48-72 часа;
- задача: минимальный RPO и быстрый DR.

2. `Daily control snapshot`:
- 1 раз в сутки;
- snapshot тех же индексов;
- retention 30-60 дней;
- задача: контрольная точка для штатных restore и проверки целостности.

3. `Monthly checkpoint snapshot`:
- 1 раз в месяц;
- retention 12-13 месяцев;
- задача: операционный исторический слой, пока весь массив все еще находится онлайн.

Так как snapshots в OpenSearch инкрементальные, частые snapshot обычно увеличивают объем хранилища гораздо меньше, чем кажется на первый взгляд, но все равно наращивают metadata `snapshot repository`.

### Резервное копирование активного snapshot repository внешней СРК

Нужно разделять:

- логические snapshot OpenSearch;
- файловый backup самого `snapshot repository` со стороны СРК.

Для активного `snapshot repository` рекомендуется такой режим:

1. Тип backup:
- `synthetic full weekly + daily incremental`, если СРК это умеет;
- если не умеет, тогда `full monthly + daily incremental`.

2. Retention backup-копий `snapshot repository`:
- оперативный контур: 60-90 дней;
- при наличии дешевого backup-tier можно хранить 180 дней.

3. Правила запуска backup:
- backup стартует только после успешного завершения snapshot-окна;
- в момент backup нельзя запускать удаление snapshot из этого же `snapshot repository`;
- желательно разводить окна `snapshot create`, `snapshot delete` и backup СРК.

4. Что именно защищает backup `snapshot repository`:
- потерю NFS-сервера;
- логическое повреждение каталога `snapshot repository`;
- сценарий, когда одновременно потеряны production-данные и сам `snapshot repository`.

Важно: "FULL 1-го числа месяца, потом Incremental" относится именно к backup СРК поверх файлов активного `snapshot repository`, а не к внутреннему механизму snapshots OpenSearch.

## Поиск по запросу регулятора без даты

В этом варианте все 5 лет онлайн, поэтому восстанавливать архив для поиска не нужно. Но поиск все равно сложный, потому что:

- дата может быть неизвестна;
- логов много;
- индексов очень много;
- разные источники могут хранить объект в разных полях.

### Практический порядок действий

1. Определить объект поиска:
- IP;
- логин;
- email;
- host;
- session ID;
- hash;
- иной идентификатор.

2. Определить перечень релевантных семейств индексов:
- `fw-*`;
- `proxy-*`;
- `auth-*`;
- `os-*`;
- `app-*`;
- `audit-*`.

3. Выполнить поиск не по одному индексу, а по alias или index pattern всех нужных семейств за 5 лет.

4. Если прямой поиск по всем 5 годам слишком тяжелый:
- запускать `async search`;
- дробить поиск по годам или кварталам логически, но внутри production-кластера, без restore;
- объединять результаты по нормализованной схеме: `timestamp`, `source_type`, `index`, `document_id`, `matched_field`, `matched_value`.

5. Для подготовки ответа регулятору:
- выгрузить результаты;
- убрать дубликаты;
- собрать единую хронологическую ленту событий.

## DR-сценарий

1. Развернуть новый production-кластер.
2. Восстановить NFS с активным `snapshot repository` из СРК, если сам NFS потерян.
3. Смонтировать `snapshot repository` на все узлы.
4. Зарегистрировать `snapshot repository` на recovery-кластере.
5. Сначала восстановить:
- шаблоны;
- ingest pipelines;
- alias;
- критичные актуальные индексы hot/warm.
6. После возврата приема логов и базового поиска восстанавливать остальную историю.

## Что делать, если за 5 лет меняется версия OpenSearch

Это обязательный риск для всех вариантов. Официальная документация OpenSearch 2.19 говорит, что snapshots совместимы вперед только на один major release.

### Практический подход

1. Перед каждым major upgrade делать test-restore в отдельный стенд.
2. Не рассчитывать, что snapshot пятилетней давности автоматически восстановится в сильно более новой версии.
3. После каждого major upgrade выполнять "обновление архива":
- поднять совместимый intermediate-кластер;
- восстановить старые snapshot;
- при необходимости reindex;
- снять новый snapshot уже из новой major-версии.
4. Хранить runbook и контейнерные образы хотя бы для предыдущей major-версии.

Для варианта 1 риск немного ниже, чем в архивных вариантах, потому что основные данные живут онлайн и проходят upgrade вместе с production-кластером. Но старые backup-копии активного `snapshot repository` все равно нужно периодически проверять на восстанавливаемость.

## Плюсы

- Все 5 лет данных всегда онлайн.
- Поиск без даты можно выполнять сразу в production-кластере.
- Самая простая концепция хранения: "все в одном месте".
- Не нужны отдельные архивные recovery-контуры для исторического поиска.

## Минусы

- Самый дорогой вариант по диску и compute.
- Самый тяжелый по DR: восстанавливать нужно огромный объем данных.
- Metadata активного `snapshot repository` растет быстрее всего.
- Хуже всего масштабируется при большом количестве семейств логов и индексов.
- Для выполнения `RPO <= 1 час` приходится часто snapshot-ить весь кластер.

## Вывод

Вариант 1 подходит только если есть жесткое требование держать все 5 лет онлайн в основном кластере и принимать самую высокую стоимость владения. С точки зрения DR и управляемости он уступает варианту 2.

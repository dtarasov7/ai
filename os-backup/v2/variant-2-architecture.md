# Вариант 2. Ежедневные индексы по каждому типу логов, snapshot закрытых индексов

## Суть варианта

Запись идет только в индексы текущего дня. После смены суток вчерашние индексы переводятся в неизменяемое состояние и после этого попадают в "финальный" snapshot. Для активного дня дополнительно выполняются частые DR-snapshot, чтобы уложиться в `RPO <= 1 час`.

Это лучший вариант из первых двух, если все 5 лет действительно должны оставаться онлайн в основном production-кластере.

## Модель данных

Здесь нужно мыслить не "один лог = один индекс", а "много логов = много семейств индексов":

- `fw-2026.05.05`
- `proxy-2026.05.05`
- `auth-2026.05.05`
- `os-2026.05.05`
- `app-2026.05.05`

То есть каждый день появляется не один индекс, а набор дневных индексов по каждому типу логов. Архитектура snapshot должна учитывать именно это.

Практически лучше использовать:

- data streams или строгое naming convention;
- alias по типам логов;
- унифицированные поля для объектов поиска, чтобы регуляторный запрос можно было запускать сразу по нескольким семействам.

## Архитектура

### Компоненты

1. Production-кластер OpenSearch 2.19:
- hot/warm/cold tiers;
- запись только в текущие дневные индексы каждого семейства логов.

2. Два активных `snapshot repository` на NFS:
- `repo-v2-hourly` для короткоживущих DR-snapshot текущего дня;
- `repo-v2-sealed` для финальных snapshot уже закрытых дневных индексов.

3. Внешняя СРК:
- резервирует оба активных `snapshot repository`;
- хранит backup-копии в своем backup-хранилище.

### Базовая конфигурация

```yaml
path.repo:
  - /mnt/opensearch-snapshots
```

```json
PUT /_snapshot/repo-v2-hourly

  "type": "fs",
  "settings": {
```

## Логика snapshot и backup

### Суточный цикл

1. Для каждого семейства логов существует индекс текущего дня.
2. После наступления новых суток создается новый набор дневных индексов.
3. Вчерашние индексы каждого семейства снимаются с записи.
4. Сначала на них ставится `write` block.
5. После этого они считаются неизменяемыми и включаются в финальный snapshot.

Пример безопасного шага:

```http
PUT /fw-2026.05.04/_block/write
PUT /proxy-2026.05.04/_block/write
PUT /auth-2026.05.04/_block/write
```

### Логические snapshot внутри OpenSearch

1. `Hourly DR snapshot`:
- каждые 30-60 минут;
- включает все текущие write-индексы всех семейств логов;
- retention 48-72 часа;
- задача: минимальный RPO и быстрый DR.

2. `Daily sealed snapshot`:
- 1 раз в сутки после rollover;
- включает вчерашние индексы всех семейств логов;
- retention 5 лет;
- задача: нормативное хранение и точечный restore.

3. `Monthly control snapshot`:
- 1 раз в месяц;
- включает контрольный набор закрытых индексов;
- retention 12-13 месяцев;
- задача: укрупненные контрольные точки для restore test.

Политики должны быть разведены по времени, чтобы не было конкурирующих операций на одних и тех же индексах.

### Резервное копирование активных snapshot repository внешней СРК

#### Для `repo-v2-hourly`

- режим backup: `synthetic full weekly + daily incremental`, либо `full weekly + daily incremental`;
- retention: 14-30 дней;
- задача: защита короткого DR-контурa.

#### Для `repo-v2-sealed`

- режим backup: `full monthly + daily incremental`, либо `synthetic full weekly + daily incremental`;
- retention: минимум 90 дней в оперативном backup-контуре;
- дополнительно можно выносить monthly full в более дешевый tier на 1 год.

#### Общие правила

1. Backup стартует только после завершения snapshot-операций.
2. На время backup не должно быть операций delete snapshot в том же `snapshot repository`.
3. Удалять snapshot нужно только через API OpenSearch, а не руками из файловой структуры `snapshot repository`.

## Поиск по запросу регулятора без даты

В этом варианте вся история онлайн, поэтому полный restore для поиска не нужен. Но из-за множества логов и индексов нужен четкий порядок работы.

### Практический порядок действий

1. Определить тип объекта поиска и все поля, в которых он может встречаться.
2. Определить список семейств индексов, где этот объект потенциально есть.
3. Выполнить асинхронный поиск по alias или index pattern сразу за 5 лет:
- например `fw-*`, `proxy-*`, `auth-*`, `os-*`, `app-*`.
4. Если объем слишком велик:
- дробить поиск по годам или кварталам внутри production-кластера;
- выполнять несколько отдельных async search;
- результаты складывать в отдельный служебный индекс или выгрузку.
5. После завершения:
- дедуплицировать результаты;
- объединить их в единый chronological report.

Важный вывод: в варианте 2 поиск "без даты" решается вычислительно тяжело, но без восстановления из backup.

## DR-сценарий

1. Развернуть новый production-кластер.
2. Восстановить NFS с активными `snapshot repository` из СРК, если сами NFS-каталоги потеряны.
3. Подключить `repo-v2-hourly` и `repo-v2-sealed`.
4. Сначала восстановить:
- текущие write-индексы;
- последние 1-2 суток;
- шаблоны, alias, ingest pipelines.
5. После возврата приема логов и базового поиска поэтапно восстановить остальную историю.

## Что делать, если за 5 лет меняется версия OpenSearch

Ограничение то же: snapshots совместимы вперед только на один major release.

### Практический подход

1. Перед каждым major upgrade проверять restore test на отдельном стенде.
2. После major upgrade запускать фоновое "освежение" защитного слоя:
- переснимать новые snapshot из уже обновленного production-кластера;
- старые snapshot repository не считать единственным источником пятилетней истории.
3. Если нужен restore старого snapshot из слишком старой major-версии:
- поднять intermediate-кластер;
- восстановить snapshot;
- reindex или снять новый snapshot;
- повторить переход до нужной major-версии.

Так как все данные остаются онлайн, этот вариант лучше переносит evolution платформы, чем архивные схемы. Но старые копии `snapshot repository` все равно нужно тестировать.

## Плюсы

- Лучший вариант, если все 5 лет обязаны быть онлайн.
- Удобное точечное восстановление по дням и по семействам логов.
- Реалистично укладывается в `RPO <= 1 час`.
- Финальный snapshot снимается только с уже закрытых индексов.
- Поиск без даты возможен сразу по production-кластеру.

## Минусы

- Сложнее orchestration: rollover, write block, две группы snapshot-политик, backup двух `snapshot repository`.
- За 5 лет будет очень много индексов и snapshot metadata.
- Production-кластер остается дорогим, потому что вся история остается онлайн.
- Поиск без даты по всем логам за 5 лет может сильно нагружать production.

## Вывод

Если все 5 лет истории должны быть всегда доступны в production без отдельного архивного восстановления, вариант 2 наиболее практичен. Это базовая рекомендуемая схема для "вся история онлайн".

# Итоговое задание

## Исходные условия

Есть кластер OpenSearch 2.19.

Требование регулятора: хранить данные 5 лет.

Кластер разделен на группы:

- hot
- warm/worm
- cold

В OpenSearch хранятся не один лог и не один индекс, а множество разных логов и, соответственно, множество семейств индексов.

Данные должны храниться так долго, потому что могут поступать запросы от регуляторов, требующие поиска информации за разные временные промежутки.

При этом запрос регулятора может содержать объект поиска, но не содержать дату. Это означает, что в ряде случаев поиск придется выполнять за весь пятилетний период.

## Ограничения и допущения

1. В качестве `snapshot repository` используется NFS в виде экспортированного каталога отдельного сервера.
2. Есть внешняя СРК, которая умеет резервировать файловую систему `snapshot repository` и хранить backup-копии в собственном хранилище.
3. Использование лент не является обязательным. Для задачи важно именно резервное копирование `snapshot repository` внешней СРК.
4. Логи поступают постоянно.
5. DRP должен предусматривать:
- быстрое восстановление работоспособности сервиса;
- затем поэтапное восстановление данных.
6. На ответ регулятору обычно дается до 7 дней.
7. Целевой `RPO` желательно минимизировать до 1 часа или меньше.
8. Для варианта с ежедневными индексами допустимо следующее допущение:
- запись идет только в индекс текущего дня;
- вчерашний индекс можно гарантированно переводить в read-only или как минимум в состояние без записи.
9. Нужно учитывать, что за 5 лет версия OpenSearch в production может измениться, в том числе с переходом между major-версиями.

## Варианты для анализа

### Вариант 1

Все данные хранятся в production-кластере 5 лет и в любой момент доступны для чтения.

Резервное копирование строится так:

- регулярно создается snapshot всего кластера;
- активный `snapshot repository` резервируется внешней СРК;
- при потере индексов они восстанавливаются из snapshot;
- при потере самого `snapshot repository` он восстанавливается из backup СРК;
- при полной потере кластера сначала восстанавливается `snapshot repository`, затем из него восстанавливаются данные кластера.

### Вариант 2

Отличие от варианта 1:

- индексы режутся по дням;
- фактически это множество дневных индексов по каждому типу логов, а не один индекс в день;
- запись идет только в индексы текущего дня;
- предыдущие дневные индексы переводятся в read-only или в эквивалентное неизменяемое состояние;
- регулярный snapshot делается по закрытым дневным индексам;
- отдельно должен быть проработан способ достижения `RPO <= 1 час` для данных текущего дня.

### Вариант 3

Более оптимальный вариант без использования searchable snapshots:

- в production-кластере хранится только рабочий онлайн-горизонт данных;
- более старые данные выводятся в архивные `snapshot repository`;
- для поиска по архиву данные при необходимости восстанавливаются в отдельный recovery-контур обычным restore из snapshot;
- восстановление архива не должно блокировать быстрое восстановление production-функционала.

### Вариант 4

Вариант с использованием searchable snapshots:

- в production-кластере хранится только рабочий онлайн-горизонт данных;
- архивные данные хранятся в годовых `snapshot repository`;
- после перехода на новый год старый годовой `snapshot repository` считается закрытым, целиком резервируется внешней СРК и при необходимости может быть удален с основного NFS;
- для поиска по архиву нужный годовой `snapshot repository` при необходимости восстанавливается из backup на NFS и подключается к отдельному search/recovery-контуру через searchable snapshots;
- нужно отдельно оценить плюсы и минусы такой схемы по сравнению с обычным restore архива.

## Что нужно подготовить

Нужно разобрать все 4 варианта.

Для каждого варианта подготовить отдельный файл с описанием архитектуры решения.

Также подготовить отдельный файл со сравнением вариантов и рекомендацией.

Дополнительно для каждого варианта подготовить архитектурные диаграммы в формате PlantUML, чтобы наглядно показать:

- состав решения;
- взаимодействие компонентов;
- последовательность действий в основных сценариях.

## Что обязательно раскрыть для каждого варианта

### 1. Архитектура решения

Описать:

- основные компоненты;
- роль production-кластера;
- роль `snapshot repository`;
- роль внешней СРК;
- роль NFS;
- роль recovery/search-контура, если он нужен.

### 2. Работа с множеством логов и множеством индексов

Нужно явно учитывать, что в системе много различных типов логов и много индексов.

Следует описать:

- как организованы семейства индексов;
- как на них влияет выбранная схема snapshot и restore;
- как будет выполняться поиск по нескольким семействам индексов.

### 3. Режим snapshot внутри OpenSearch

Нужно подробно расписать:

- как часто создаются snapshot;
- какие именно индексы попадают в snapshot;
- какая retention у snapshot;
- как вариант соответствует или не соответствует требованию `RPO <= 1 час`.

### 4. Резервное копирование snapshot repository внешней СРК

Нужно подробно описать резервное копирование именно `snapshot repository`, а не только логические snapshot OpenSearch.

Нужно указать:

- режим backup;
- full / incremental / synthetic full, если применимо;
- retention backup-копий;
- момент запуска backup относительно snapshot-операций;
- что произойдет при потере активного `snapshot repository`;
- что произойдет при полной потере production-кластера.

### 5. DR и восстановление

Нужно описать:

- последовательность быстрого восстановления сервиса;
- последовательность полного восстановления данных;
- что восстанавливается в первую очередь;
- как вариант влияет на `RTO`;
- где есть узкие места.

### 6. Поиск по запросу регулятора без даты

Это обязательный раздел.

Нужно явно описать, как действовать, если регулятор передает объект поиска без даты, и поиск потенциально нужен за все 5 лет.

Если данные не все находятся онлайн, нужно явно расписать последовательность действий, например:

- поиск по online-части;
- далее поиск по архивным периодам;
- восстановление или подключение архивов по годам;
- поиск по каждому периоду;
- объединение результатов;
- дедупликация;
- формирование единого ответа регулятору.

Если для какого-то варианта может потребоваться:

- место под восстановление `snapshot repository`;
- место под восстановленные индексы;
- время на restore backup `snapshot repository`;
- время на restore индексов из snapshot;

это нужно явно описать.

### 7. Смена версий OpenSearch за 5 лет

Это обязательный раздел для каждого варианта.

Нужно описать:

- какие риски возникают при смене версий OpenSearch;
- что делать при переходе между major-версиями;
- как проверять восстанавливаемость старых snapshot;
- нужен ли intermediate-кластер;
- как поддерживать архив в восстанавливаемом состоянии весь срок хранения.

### 8. Плюсы и минусы

Для каждого варианта указать:

- положительные стороны;
- отрицательные стороны;
- ограничения;
- операционные риски.

### 9. Архитектурные диаграммы

Для каждого варианта нужно подготовить диаграммы в формате PlantUML.

Минимальный набор:

- одна архитектурная диаграмма с основными компонентами;
- одна диаграмма последовательности действий для ключевого сценария.

В качестве ключевого сценария использовать то, что наиболее важно для данного варианта:

- DR-восстановление;
- либо поиск по запросу регулятора без даты;
- либо последовательность архивного восстановления / подключения архива.

## Формат результата

Сделать несколько файлов:

- по одному файлу на каждый вариант;
- один отдельный файл со сравнением вариантов и рекомендацией;
- отдельные PlantUML-файлы с диаграммами по каждому варианту.

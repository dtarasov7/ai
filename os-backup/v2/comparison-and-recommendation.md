# Сравнение вариантов и рекомендация

## Краткий вывод

1. Если все 5 лет данных обязаны оставаться постоянно онлайн в основном production-кластере, лучший вариант - вариант 2.
2. Если часть истории может подниматься отдельно обычным restore, лучший компромисс - вариант 3.
3. Если часть истории может подниматься отдельно и нужен более быстрый "вход в поиск" по архиву с меньшим локальным диском, нужно рассматривать вариант 4 на searchable snapshots.
4. Вариант 1 самый простой концептуально, но самый тяжелый по стоимости и DR.

## Сравнение

| Критерий | Вариант 1 | Вариант 2 | Вариант 3 | Вариант 4 |
| --- | --- | --- | --- | --- |
| Идея | Все 5 лет онлайн, snapshot всего кластера | Все 5 лет онлайн, но индексы режутся по дням и snapshot-ятся после закрытия | Онлайн только рабочий горизонт, архив поднимается обычным restore | Онлайн только рабочий горизонт, архив ищется через searchable snapshots |

| Скорость DR production | Низкая | Хорошая | Лучшая | Лучшая |
| Стоимость production | Самая высокая | Высокая | Низкая | Низкая |
| Время до первого поиска в архиве | Не требуется архивный restore | Не требуется архивный restore | Низкое/среднее, нужен обычный restore | Лучшее из архивных вариантов |

## Что общего для всех вариантов

### 1. Нужно учитывать множество логов и множество индексов

Архитектура должна изначально строиться не вокруг одного индекса, а вокруг многих семейств:

- `fw-*`
- `proxy-*`
- `auth-*`
- `os-*`
- `app-*`
- `audit-*`

Нужны:

- единые naming rules;
- alias или data streams;
- согласованная схема полей для объектов поиска;
- runbook регуляторного поиска по нескольким семействам индексов.

### 2. Поиск без даты может потребовать полный проход за 5 лет

Если дата неизвестна, нужно заранее определить последовательность действий.

#### Для вариантов 1 и 2

1. Поиск по всем релевантным семействам индексов сразу в production.
2. Если объем слишком велик, разбиение запроса по годам или кварталам логически, без restore.
3. Объединение и дедупликация результатов.

#### Для вариантов 3 и 4

1. Сначала поиск по online-горизонту в production.
2. Затем поочередная работа с архивными периодами.
3. Для каждого года:
- восстановить или подключить нужный `snapshot repository`;
- выполнить поиск;
- выгрузить результаты.
4. После обхода всех лет объединить результаты в единый отчет.

То есть для архивных вариантов нужно заранее принять, что отсутствие даты у объекта поиска является штатным сценарием, а не исключением.

### 3. За 5 лет production-версия OpenSearch почти наверняка изменится

Официальная документация OpenSearch 2.19 указывает, что snapshots совместимы вперед только на один major release.

Поэтому для всех вариантов обязательно:

1. Делать регулярные restore test.
2. Перед каждым major upgrade тестировать восстановление из актуальных `snapshot repository`.
3. Для архивных snapshot старых major-версий иметь runbook:
- поднять intermediate-кластер;
- восстановить snapshot;
- reindex или снять новый snapshot;
- перевести архив на новую major-версию.

Без этого пятилетний контур хранения нельзя считать реально работоспособным.

### 4. Backup snapshot repository нужно проектировать отдельно от snapshot OpenSearch

Нужно различать:

- логические snapshots внутри OpenSearch;
- внешний файловый backup `snapshot repository`.

Минимальные общие правила:

1. Backup активного `snapshot repository` запускать только после завершения snapshot-операций.
2. Не делать delete snapshot во время backup.
3. Удалять snapshot только через API OpenSearch.
4. Для активных `snapshot repository` использовать:
- `synthetic full weekly + daily incremental`, если умеет СРК;
- иначе `full monthly + daily incremental`.
5. Для "закрытых" архивных `snapshot repository` после завершения периода делать контрольный full backup в архивный backup-tier.

## Практическая рекомендация

### Если вся история должна быть всегда онлайн

Берите вариант 2:

- дневные индексы по каждому типу логов;
- hourly DR snapshot текущих write-индексов;
- daily sealed snapshot закрытых индексов;
- оба активных `snapshot repository` дополнительно резервируются СРК.

### Если старую историю можно поднимать по запросу обычным restore

Берите вариант 3:

- production хранит только рабочий горизонт;
- архивные периоды поднимаются в отдельный recovery-контур;
- цена ниже, DR быстрее, но поиск без даты тяжелее по времени.

### Если старую историю можно искать через searchable snapshots

Берите вариант 4:

- годовые архивные `snapshot repository`;
- после перехода на новый год старый `snapshot repository` целиком уходит в backup;
- для поиска архивный год возвращается на NFS и подключается к отдельному search-контуру как searchable snapshot.

Это самый интересный архивный вариант, если регуляторный поиск бывает нечасто, но хочется уменьшить двойное хранение локальных индексов и ускорить "доступность к поиску" после возврата архива.

## Что я бы выбрал

1. Вариант 2, если есть жесткое требование "все 5 лет всегда онлайн".
2. Вариант 4, если допускается архивный поиск по запросу и команда готова к более зрелой эксплуатации.
3. Вариант 3, если searchable snapshots не хочется использовать или нужен максимально понятный архивный restore-процесс.

# Вариант 1. Все 5 лет данных онлайн, снапшот всего кластера

## Суть варианта

Все данные за 5 лет продолжают храниться в production-кластере OpenSearch 2.19 на hot/warm/cold-нодах и в любой момент доступны для чтения. Резервное копирование строится вокруг snapshot repository на общем NFS-каталоге, который затем дополнительно копируется внешней СРК на собственное backup-хранилище.

Это самый прямолинейный вариант, но он хуже всего попадает в требование быстрого DR и `RPO <= 1 час`, если оставлять его в исходном виде только с ежедневным снапшотом.

## Архитектура

### Компоненты

1. Production-кластер OpenSearch 2.19:
- hot, warm, cold группы;
- все 5 лет индексов остаются в кластере.

2. Snapshot repository:
- отдельный NFS-сервер;
- один экспортируемый каталог, смонтированный на всех узлах кластера в одинаковый путь;
- в `opensearch.yml` на всех узлах добавлен `path.repo`.

3. Внешняя СРК:
- забирает файловую копию snapshot repository;
- хранит резервные копии в своем хранилище.

### Базовая конфигурация

```yaml
path.repo:
  - /mnt/opensearch-snapshots
```

```json
PUT /_snapshot/repo-v1
{
  "type": "fs",
  "settings": {
    "location": "/mnt/opensearch-snapshots/repo-v1",
    "compress": true
  }
}
```

## Поток резервного копирования

### Как работает схема в исходном виде

1. Раз в сутки создается snapshot всего кластера.
2. После завершения snapshot внешняя СРК копирует каталог `repo-v1`.
3. При потере отдельных индексов они восстанавливаются из snapshot.
4. При потере всего кластера:
- сначала восстанавливается каталог snapshot repository из СРК;
- затем поднимается новый кластер и из repository восстанавливаются данные.

### Что важно уточнить

OpenSearch snapshots являются инкрементальными сами по себе. Поэтому "FULL 1-го числа и daily incremental" в этой архитектуре нужно понимать не как режим snapshot внутри OpenSearch, а как режим файлового backup со стороны СРК поверх каталога repository.

## Оценка по RPO / RTO

### RPO

Если делать только ежедневный snapshot всего кластера, реальный `RPO` будет до 24 часов. Это не соответствует вашему целевому требованию "`1 час или меньше`".

Чтобы этот вариант приблизить к требованиям, его нужно модифицировать:
- делать snapshot всего кластера каждый час, а не раз в сутки;
- или как минимум каждые 30-60 минут для актуального окна данных.

Технически это возможно, потому что snapshots инкрементальные, но у варианта все равно останется высокий операционный overhead: каждый запуск затрагивает весь набор индексов и метаданных кластера.

### RTO

RTO здесь самый тяжелый из трех вариантов:
- для быстрого возврата сервиса придется сначала поднимать пустой кластер;
- затем восстанавливать cluster-level объекты и самые свежие индексы;
- потом дозаливать остальную историческую массу данных.

Если объем данных за 5 лет велик, полный restore может занимать часы или дни. Это противоречит идее "сначала быстро вернуть функциональность".

## Сценарий DR

1. Развернуть новый кластер OpenSearch.
2. Смонтировать восстановленный из СРК NFS-каталог.
3. Зарегистрировать repository на recovery-кластере, предпочтительно в режиме `readonly`.
4. Сначала восстановить:
- шаблоны;
- ingest pipelines;
- алиасы;
- критичные свежие индексы hot/warm.
5. После возврата сервиса поэтапно восстанавливать warm/cold и исторические индексы.

## Плюсы

- Самая простая логика резервного копирования.
- Все 5 лет данных постоянно доступны онлайн.
- Не нужны отдельные правила "закрытия" вчерашних индексов.
- Удобно мыслится как классический DR через "backup всего".

## Минусы

- В исходном виде не выполняет `RPO <= 1 час`.
- Наиболее длинные snapshot-окна и самый тяжелый restore.
- Самая дорогая модель по диску и ресурсам production-кластера, потому что все 5 лет остаются онлайн.
- Восстановление одного небольшого периода менее удобно, чем при ежедневной нарезке индексов.
- Большой и долго живущий repository создает нагрузку на metadata management и cluster manager heap.
- За 5 лет почти наверняка будут version upgrades, а snapshots совместимы вперед только на один major release, поэтому нужны регулярные test-restore и lifecycle-план по совместимости.

## Вывод

Вариант 1 можно использовать только если есть жесткое требование держать все 5 лет всегда онлайн и нужна максимально простая схема. Но даже в этом случае его придется дорабатывать до более частых snapshot, а по совокупности DR, стоимости и управляемости он уступает варианту 2.

# Вариант 2. Ежедневные индексы, snapshot только неизменяемых индексов

## Суть варианта

Логи пишутся только в индекс текущего дня. После смены суток вчерашний индекс перестает принимать запись, переводится в логически неизменяемое состояние и только после этого попадает в "финальный" snapshot.

По сравнению с вариантом 1 эта схема лучше соответствует природе логов и заметно упрощает точечное восстановление. Но в исходной формулировке она тоже не закрывает `RPO <= 1 час` для данных текущего дня. Поэтому для практической архитектуры нужен второй слой: частые snapshot текущего write-индекса.

## Архитектура

### Компоненты

1. Production-кластер OpenSearch 2.19:
- индексы режутся по дням;
- запись идет только в индекс текущего дня;
- hot/warm/cold размещение настраивается через ISM или операционные политики.

2. Два snapshot repository на одном NFS:
- `repo-v2-hourly` для короткоживущих DR-снапшотов текущего дня;
- `repo-v2-sealed` для "финальных" snapshot уже закрытых дневных индексов.

3. Внешняя СРК:
- копирует оба каталога repository после завершения snapshot-окна;
- хранит файловые копии в своем backup-хранилище.

### Базовая конфигурация

```yaml
path.repo:
  - /mnt/opensearch-snapshots
```

```json
PUT /_snapshot/repo-v2-hourly
{
  "type": "fs",
  "settings": {
    "location": "/mnt/opensearch-snapshots/repo-v2-hourly",
    "compress": true
  }
}
```

```json
PUT /_snapshot/repo-v2-sealed
{
  "type": "fs",
  "settings": {
    "location": "/mnt/opensearch-snapshots/repo-v2-sealed",
    "compress": true
  }
}
```

## Логика работы

### Суточный цикл

1. В `00:00` создается новый индекс текущего дня.
2. Вчерашний индекс снимается с записи.
3. Для безопасного cutover лучше сначала ставить `write` block, потому что OpenSearch гарантирует завершение in-flight write перед вступлением блока в силу.
4. После этого индекс можно считать неизменяемым и отправлять в "финальный" snapshot.

Пример безопасного шага:

```http
PUT /logs-2026.05.03/_block/write
```

При необходимости после служебных операций индекс можно дополнительно перевести в `read_only`.

### Политики snapshot

1. `Hourly DR snapshot`:
- каждые 30-60 минут snapshot только индекса текущего дня;
- короткая retention, например 24-72 часа;
- задача: минимальный RPO и быстрый DR.

2. `Daily sealed snapshot`:
- один раз в сутки, после rollover и write block;
- snapshot только вчерашнего индекса;
- долгосрочная retention для 5-летней истории.

Важно: политики нужно разводить по времени так, чтобы они не пересекались. OpenSearch Snapshot Management не рекомендует конкурентные snapshot-операции с пересекающимися индексами и расписаниями.

## Оценка по RPO / RTO

### RPO

Если брать исходный вариант буквально и snapshot только "вчерашний индекс раз в сутки", то текущий день не защищен и `RPO` снова будет до 24 часов.

Если добавить hourly snapshot текущего write-индекса, то вариант уже реалистично попадает в `RPO <= 1 час`.

### RTO

RTO лучше, чем у варианта 1:
- для возврата сервиса сначала восстанавливаются свежие hourly snapshot по активному периоду;
- старые дни и исторические диапазоны можно подтягивать отдельно;
- при потере одного дня или диапазона восстановление точечное и быстрее.

## Сценарий DR

1. Развернуть новый кластер.
2. Смонтировать NFS repository, восстановленный из СРК.
3. Подключить `repo-v2-hourly` и `repo-v2-sealed`.
4. Сначала восстановить:
- актуальный write-индекс или последние 1-2 дня;
- шаблоны, алиасы, ingest pipelines.
5. После возврата сервиса поэтапно восстановить исторические дневные индексы из `repo-v2-sealed`.

## Плюсы

- Намного лучше подходит для логов и time-series данных.
- Удобное точечное восстановление по дням.
- Ниже нагрузка на ежедневный "финальный" snapshot, потому что в него попадает только завершенный индекс.
- Реалистично достигает `RPO <= 1 час`, если добавить частые snapshot текущего индекса.
- Удобнее переносить старые индексы между hot/warm/cold.

## Минусы

- Сложнее orchestration: rollover, write block, расписания snapshot, retention.
- Появляются два типа snapshot-политик и, желательно, два repository.
- За 5 лет будет очень много индексов и snapshots, поэтому нужно следить за размером metadata repository и heap cluster manager.
- Если нужно восстановить очень широкий исторический период, это все равно может быть долго.
- Требуются регулярные restore tests и план по snapshot compatibility при major upgrade.

## Вывод

Если все 5 лет данных действительно должны оставаться онлайн и быть доступны для поиска без ожидания восстановления, это лучший из первых двух вариантов. Но его стоит реализовывать не в "чистом" виде из постановки, а как гибрид внутри варианта 2: `daily sealed snapshot` плюс `hourly current-day snapshot`.

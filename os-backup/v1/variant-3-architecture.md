# Вариант 3. Гибрид: короткий онлайн-контур для DR и архивный контур для 5-летнего хранения

## Суть варианта

Это более оптимальная схема, если требование регулятора звучит как "данные должны храниться 5 лет и быть доступны по запросу в течение нескольких дней", а не как "все 5 лет должны всегда находиться онлайн в production-кластере".

Основная идея:
- держать онлайн только тот горизонт, который нужен для повседневного поиска и быстрого DR;
- старшие периоды выводить из production-кластера в архивные snapshots;
- восстанавливать архивный период по запросу регулятора в отдельный recovery-контур.

С учетом вашего SLA "ответ регулятору обычно до 7 дней" это самый рациональный вариант по стоимости, RTO и операционной управляемости.

## Рекомендуемая модель хранения

Границы можно настроить, пример такой:

1. `0-30 дней`:
- hot;
- максимальная скорость записи и поиска.

2. `31-180 дней`:
- warm;
- все еще доступны онлайн.

3. `181-365 дней`:
- cold;
- доступны онлайн, но на более дешевых ресурсах.

4. `Старше 365 дней`:
- удаляются из production-кластера после финального архивного snapshot;
- сохраняются в snapshot repository и в копии СРК;
- восстанавливаются по запросу в отдельный recovery-кластер или временный search-контур.

Если нужно, онлайн-горизонт можно увеличить до 2-3 лет, но ключевая идея остается той же: production и long-term archive не обязаны быть одной и той же системой хранения.

## Архитектура

### Компоненты

1. Production-кластер OpenSearch 2.19:
- ежедневные индексы;
- текущий день writable;
- свежий онлайн-горизонт для быстрого поиска и DR.

2. Short-term DR repository на NFS:
- частые snapshot текущего дня и последних свежих индексов;
- retention 7-30 дней;
- задача: минимальный RPO и быстрый возврат функционала.

3. Long-term archive repository на NFS:
- финальные snapshots закрытых дневных индексов;
- retention 5+ лет;
- лучше делить архив по годам или кварталам, чтобы не раздувать metadata одного repository.

4. Внешняя СРК:
- копирует оба repository;
- short-term backup защищает DR-контур;
- archive backup защищает сам архивный repository.

5. Recovery-контур:
- отдельный временный или постоянно подготовленный кластер для поднятия архивных периодов по запросу регулятора.

## Логика работы

### Поток записи и защиты данных

1. Логи пишутся в индекс текущего дня.
2. Каждые 30-60 минут создается snapshot текущего write-индекса в `repo-dr-shortterm`.
3. После смены суток вчерашний индекс переводится в неизменяемое состояние.
4. Для вчерашнего индекса создается финальный snapshot в `repo-archive-YYYY`.
5. По достижении порога возраста индекс удаляется из production-кластера, но остается в архивном repository и в СРК.

### Организация repository

Практически лучше не держать все 5 лет в одном repository. Более устойчивая схема:

- `/mnt/opensearch-snapshots/dr-shortterm`
- `/mnt/opensearch-snapshots/archive-2026`
- `/mnt/opensearch-snapshots/archive-2027`

Так проще:
- ограничивать рост metadata;
- делать restore нужного периода;
- переносить архив между поколениями платформы.

## Оценка по RPO / RTO

### RPO

Этот вариант лучше всего попадает в `RPO <= 1 час`, потому что защищает именно активный write-контур частыми snapshot.

### RTO

Этот вариант лучше всего попадает и в быстрое восстановление сервиса:

1. Сначала поднимается production-контур только с актуальными данными.
2. Восстанавливаются последние часы/дни, необходимые для возобновления работы.
3. Архив старше онлайн-горизонта вообще не блокирует возврат сервиса.
4. По запросу регулятора нужные старые диапазоны поднимаются отдельно в течение доступных 7 дней.

## Сценарий DR

### Быстрый возврат сервиса

1. Развернуть новый production-кластер.
2. Подключить `repo-dr-shortterm`.
3. Восстановить шаблоны, pipelines, алиасы и последние актуальные индексы.
4. Вернуть прием логов и базовый поиск.

### Восстановление истории

1. Определить требуемый период.
2. Подключить нужный архивный repository.
3. Поднять индексы в отдельный recovery-контур.
4. Выполнить поиск, выгрузку и подготовить ответ регулятору.

## Плюсы

- Лучшее соответствие целям быстрого DR.
- Лучшее соответствие `RPO <= 1 час`.
- Самая низкая стоимость production-кластера.
- Самый маленький объем данных, который нужно срочно восстанавливать после аварии.
- Самый управляемый рост snapshot metadata, если архив делить по годам или кварталам.
- Удобно отделяет операционную доступность сервиса от долгосрочного нормативного хранения.

## Минусы

- Не все 5 лет данных находятся онлайн в основном production-кластере.
- Нужен формализованный процесс "архивный restore по запросу".
- Потребуется отдельный recovery-контур или регламент его быстрого развертывания.
- Нужны регулярные тесты восстановления именно архивных периодов.
- При platform upgrade нужно следить за snapshot compatibility между major версиями.

## Вывод

Это наиболее оптимальный вариант, если допускается, что часть 5-летней истории может храниться не в основном production-кластере, а в архивном snapshot-контуре с восстановлением по запросу. С учетом вашего срока ответа регулятору до 7 дней именно этот вариант выглядит наиболее рациональным.

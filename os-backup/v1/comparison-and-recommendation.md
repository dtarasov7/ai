# Сравнение вариантов и рекомендация

## Краткий вывод

1. Если все 5 лет данных обязаны оставаться онлайн и быть доступны для поиска без дополнительного восстановления, выбирайте вариант 2.
2. Если допустимо хранить старшие периоды в архивном snapshot-контуре и поднимать их по запросу регулятора в пределах 7 дней, выбирайте вариант 3.
3. Вариант 1 в исходном виде самый слабый: он не закрывает целевой `RPO <= 1 час` и тяжелее всех по DR.

## Сравнение

| Критерий | Вариант 1 | Вариант 2 | Вариант 3 |
| --- | --- | --- | --- |
| Идея | Все 5 лет онлайн, snapshot всего кластера | Дневные индексы, финальный snapshot закрытого дня | Короткий онлайн-контур + архивный snapshot-контур |
| Соответствие `RPO <= 1 час` | Нет, если snapshot только раз в сутки | Да, если добавить hourly snapshot текущего дня | Да, лучший вариант |
| Скорость DR | Низкая | Средняя/хорошая | Лучшая |
| Стоимость production-кластера | Самая высокая | Высокая | Самая низкая |
| Удобство точечного restore | Низкое/среднее | Высокое | Высокое |
| Все 5 лет сразу онлайн | Да | Да | Нет |
| Операционная сложность | Низкая | Средняя | Средняя/высокая |
| Масштабируемость на 5 лет | Худшая | Хорошая | Лучшая |

## Практическая рекомендация

### Рекомендация №1

Если бизнес требует, чтобы вся 5-летняя история всегда была онлайн в основном кластере, оптимальным будет вариант 2 в доработанном виде:

- ежедневные индексы;
- rollover по суткам;
- `write` block на вчерашний индекс;
- ежедневный "финальный" snapshot вчерашнего индекса;
- отдельный hourly snapshot текущего write-индекса;
- два repository на одном NFS для short-term DR и sealed daily snapshots.

### Рекомендация №2

Если допускается, что часть старой истории может подниматься отдельно по регуляторному запросу, оптимальным будет вариант 3:

- онлайн только рабочий горизонт;
- остальное в архивных snapshot;
- быстрый DR за счет short-term snapshot;
- восстановление старой истории в отдельный recovery-контур.

Это лучше соответствует формулировке "сначала быстро вернуть функциональность, потом восстановить данные".

## Общие обязательные требования для любого варианта

1. NFS repository должен быть смонтирован на всех узлах в одинаковый путь и добавлен в `path.repo`.
2. Snapshot-политики нельзя планировать так, чтобы они конфликтовали по времени и по одним и тем же индексам.
3. СРК должна копировать repository после завершения snapshot-операций, а не в произвольный момент активной перестройки repository.
4. Необходимо регулярно делать test-restore:
- restore отдельных индексов;
- restore последних актуальных данных;
- полный DR restore;
- restore архивного периода.
5. Для 5-летнего горизонта нужен upgrade-план:
- snapshots совместимы вперед только на один major release;
- после major upgrade нужно проверять восстановление старых периодов и при необходимости переснимать архив.
6. Конфигурацию security лучше резервировать отдельно и не полагаться на restore `.opendistro_security` как на основной способ восстановления.

## Что я бы выбрал

Для вашего кейса я бы рекомендовал:

1. Вариант 2, если есть жесткое требование "вся история всегда онлайн".
2. Вариант 3, если важнее всего быстрый DR и разумная стоимость при SLA ответа регулятору до 7 дней.

Если нужен один выбор без дополнительных условий, наиболее рациональным выглядит вариант 3.

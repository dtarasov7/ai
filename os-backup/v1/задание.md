Кластер Opensearch 2.19 
Есть требование регулятора хранить данные 5 лет.
Кластер разбит на hot, worm и cold группы
Задача - настроить резервное копирование:
- В снапшоты
- на отчуждаемые носители ( имеется классическая СРК, которая умеет бекапировать файловую систему)

Данные нужны так долго, так как могут приходить запросы от регуляторов, требующие поиска данных из разных временных промежутков.

Вариант 1
Данные храняться в кластере пять лет. 
Они в любой момент доступны для чтения. 
Ежедневно делается снапшот всего кластера. 
После этого снапшот бекапится на ленты. 
Например 1-ого числа месяца - FULL, а потом ежедневно Incremental. 
Такие бекапы хранятся 30 дней. 
Результат: 
- если потеряем снапшот - восстанавливаем его из full+incrment 
- если потеряем индексы - вытаскиваем их из снапшот (можем потерять данные за один день) 
- если теряем все диски - восстанавливает снапшот , а из него восстанавливаем все индексы. 

ВАриант 2
Отличие от варианта 1:
- индексы режутся по дням.
- предыдущий день делается read-only
- ежедневный снапшот делается только по вчерашним индексам

Разбери оба варианта. Для каждого из них подготов файл с описанием архитектуры решения. С указанием положительных и отрицательных сторон.

Если есть третий , более оптимальный вариант - подготовь и по нему архитектуру решения.

Уточнения по результатам вопросов

1. Snapshot repository:
- NFS в виде экспортированного каталога отдельного сервера.

2. Требования по восстановлению:
- Логи продолжают поступать постоянно.
- DRP должен включать быстрое восстановление функционала, а затем поэтапное восстановление данных.
- На ответ регулятору обычно дается до 7 дней.
- Целевой RPO желательно минимизировать до 1 часа или меньше.

3. Допущение для варианта 2:
- Да, запись идет только в индекс текущего дня.
- Вчерашний индекс можно гарантированно переводить в read-only.

4. Отчуждаемые носители / СРК:
- Ленты не принципиальны.
- У имеющейся СРК есть собственные хранилища бэкапов.
- Для задачи важно только то, что СРК умеет забирать резервную копию файловой системы snapshot repository.

5. Формат результата:
- Сделать несколько файлов: по одному для каждого решения и один отдельный файл со сравнением.

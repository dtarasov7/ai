### Пояснения к дашборду (учитывая версии Prometheus 2.49 и Grafana v11)
- **Предположения о метрике**: Я предполагаю, что метрика называется `app_version_info{nodename="...", appname="...", version="..."} 1` (gauge с значением 1, которая присутствует только для текущей версии ПО). Когда версия меняется, старая серия исчезает, а новая появляется. Это стандартный паттерн для info-метрик в Prometheus. Если имя метрики другое, замените в запросах (targets.expr).
- **Переменные (templating)**: 
  - `nodename`: Все уникальные значения лейбла `nodename` из метрики (multi-select, All включен).
  - `appname`: Аналогично для `appname`.
  - Это позволяет фильтровать дашборд динамически. Поддерживается в Grafana v11 без ограничений.
- **Панель 1: Таблица (Current Versions and Last Change)**:
  - **Запрос A (текущая версия)**: `app_version_info{nodename=~"$nodename", appname=~"$appname"}` в формате table. Показывает текущие комбинации с лейблами (nodename, appname, version как current_version).
  - **Запрос B (предыдущая версия и время)**: То же с `offset 5m` (смещение на 5 минут назад). Это захватывает версию, которая была 5м назад (предыдущая, если изменение произошло недавно). Время из этого запроса (`Time (offset)`) используется как приближение времени последнего изменения (момент последнего сэмпла предыдущей версии).
  - **Трансформации (transformations, поддержка в Grafana v11)**:
    - Organize: Переименовывает поля (version #A → current_version, version #B → previous_version, Time #B → Time (offset)). Удаляет ненужные (Value).
    - JoinByField: Объединяет таблицы по `nodename` и `appname` (outer join, чтобы сохранить все комбинации).
    - FilterByName: Оставляет только релевантные поля.
    - FilterData: Фильтрует строки, где previous_version не null (только где было изменение).
  - **Ограничения**: 
    - Offset (5m) — приближение; подгоните под интервал скрапинга Prometheus (обычно 15-60s, но для изменений версий — минуты). В Prometheus 2.49 offset/subqueries работают стабильно, но точное время изменения лейблов требует recording rules (например, правила для детекции изменений серий). Если изменений нет за 5m, previous_version = current_version.
    - Если версий много, таблица может быть большой; используйте фильтры переменных.
    - Нет поддержки "предыдущей версии" для очень старых изменений без большего offset или range-запросов.
- **Панель 2: State Timeline (Version History)**:
  - **Запрос**: `app_version_info{nodename=~"$nodename", appname=~"$appname"}` в формате time_series. LegendFormat использует `{{version}}` для показа версии как состояния.
  - **Визуализация**: State timeline (доступна в Grafana v8+, полная в v11) показывает историю изменений версии как цветные блоки во времени (x-ось — время, y-ось — комбинации nodename/appname). Цвета по лейблу version (categorical). Идеально для отслеживания смены версий.
  - **Ограничения**: Требует range-запроса (time range дашборда, по умолчанию 24h; настройте в timepicker). Если серий много, может быть медленно (Prometheus 2.49 справляется с subqueries, но для долгой истории увеличьте range). Не показывает числовые значения, только categorical (лейблы).
- **Общие настройки**:
  - Время: now-24h to now (настройте для истории).
  - Refresh: 5s (для реального времени; Prometheus 2.49 поддерживает частые запросы).
  - Datasource: Укажите UID вашего Prometheus в Grafana (по умолчанию "prometheus").
  - Импорт: Скопируйте JSON в Grafana (Dashboard → Import), укажите datasource. Тестируйте с реальными данными — если метрика не `app_version_info`, обновите expr.
  - **Версионные особенности**: Всё использует фичи Prometheus 2.49 (offsets, label_values, regex в селекторах) и Grafana v11 (state timeline, join transformations, table overrides). Нет legacy-ограничений; adult/offensive content не релевантно.

Если метрика отличается или нужны доработки (например, recording rules для точного времени изменений), предоставьте больше деталей!

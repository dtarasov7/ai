`bridge` в варианте 1 — это небольшой сервис-переходник между **Alertmanager webhook** и **Glaber**.

Его задача не мониторить Prometheus, а просто принимать события от Alertmanager и превращать их в формат, который понимает Glaber.

```text
Alertmanager
   │
   │ HTTP POST webhook JSON
   ▼
am-glaber-bridge
   │
   │ zabbix_sender / Glaber API / trapper item
   ▼
Glaber proxy
   │
   ▼
Glaber main
```

## Что делает bridge

Alertmanager отправляет webhook примерно такого смысла:

```json
{
  "status": "firing",
  "alerts": [
    {
      "status": "firing",
      "labels": {
        "alertname": "NodeDown",
        "instance": "srv01:9100",
        "severity": "critical"
      },
      "annotations": {
        "summary": "Node srv01 is down",
        "description": "Exporter is unavailable"
      },
      "startsAt": "2026-05-27T10:15:00Z"
    }
  ]
}
```

Bridge принимает этот JSON и делает из него событие для Glaber, например:

```text
host: prometheus-alerts-work
key:  prom.alert[NodeDown,srv01:9100,critical]
value: 1
```

А когда Alertmanager присылает resolved:

```text
host: prometheus-alerts-work
key:  prom.alert[NodeDown,srv01:9100,critical]
value: 0
```

В Glaber на такой item вешается trigger:

```text
last(/prometheus-alerts-work/prom.alert[NodeDown,srv01:9100,critical])=1
```

## Из чего он может быть сделан

Самый простой bridge — это маленькое приложение на Python/Go/Node.js, которое:

1. поднимает HTTP endpoint, например `/alertmanager`;
2. принимает POST от Alertmanager;
3. разбирает список `alerts`;
4. для каждого alert формирует ключ Glaber item;
5. отправляет значение в Glaber через `zabbix_sender` или совместимый sender-протокол;
6. логирует успешную/неуспешную доставку.

Например, логика bridge выглядит так:

```text
если alert.status == "firing":
    отправить value=1

если alert.status == "resolved":
    отправить value=0
```

## Почему через sender/trapper удобно

В Glaber/Zabbix-подобной модели есть тип item **trapper**: он не опрашивается самим сервером, а принимает значения извне. Поэтому bridge может пушить события в Glaber.

Условно:

```bash
zabbix_sender \
  -z glaber-proxy.work.local \
  -s "prometheus-alerts-work" \
  -k 'prom.alert[NodeDown,srv01:9100,critical]' \
  -o 1
```

При закрытии:

```bash
zabbix_sender \
  -z glaber-proxy.work.local \
  -s "prometheus-alerts-work" \
  -k 'prom.alert[NodeDown,srv01:9100,critical]' \
  -o 0
```

`zabbix_sender` здесь условное название: Glaber совместим с Zabbix-подходом, поэтому часто такие интеграции делаются именно через sender/trapper-механику.

## Более правильная модель: один JSON item

Чтобы не создавать отдельный item на каждый уникальный alert, bridge может отправлять один JSON в Glaber:

```text
host: prometheus-alerts-work
key:  prometheus.alerts.raw
value:
{
  "alerts": [
    {
      "fingerprint": "abc123",
      "status": "firing",
      "alertname": "NodeDown",
      "instance": "srv01:9100",
      "severity": "critical",
      "summary": "Node srv01 is down"
    }
  ]
}
```

А уже в Glaber через preprocessing / dependent items / LLD разбирать этот JSON.

Это обычно лучше, если алертов много и labels динамические.

## Что bridge НЕ делает

Bridge обычно **не должен**:

```text
- сам вычислять условия алертов;
- ходить в Prometheus за метриками;
- дублировать PromQL;
- решать, является ли событие проблемой;
- делать сложную корреляцию.
```

Этим уже занимаются Prometheus и Alertmanager.

Bridge должен быть максимально тупым и надёжным:

```text
получил alert → нормализовал → передал в Glaber
```

## Минимальный Python-пример

Идея bridge может выглядеть так:

```python
from flask import Flask, request
import subprocess
import re

app = Flask(__name__)

GLABER_PROXY = "glaber-proxy.work.local"
GLABER_HOST = "prometheus-alerts-work"

def safe_key_part(value: str) -> str:
    value = value or "unknown"
    return re.sub(r"[^a-zA-Z0-9_.:-]", "_", value)

def send_to_glaber(key: str, value: str):
    subprocess.run(
        [
            "zabbix_sender",
            "-z", GLABER_PROXY,
            "-s", GLABER_HOST,
            "-k", key,
            "-o", value,
        ],
        check=True,
    )

@app.post("/alertmanager")
def alertmanager_webhook():
    payload = request.get_json(force=True)

    for alert in payload.get("alerts", []):
        labels = alert.get("labels", {})

        alertname = safe_key_part(labels.get("alertname"))
        instance = safe_key_part(labels.get("instance"))
        severity = safe_key_part(labels.get("severity"))

        key = f"prom.alert[{alertname},{instance},{severity}]"

        if alert.get("status") == "firing":
            value = "1"
        else:
            value = "0"

        send_to_glaber(key, value)

    return {"status": "ok"}
```

Alertmanager config:

```yaml
receivers:
  - name: glaber
    webhook_configs:
      - url: "http://am-glaber-bridge.work.local:8080/alertmanager"
        send_resolved: true
```

## Что нужно создать в Glaber

Минимально:

```text
Host:
  prometheus-alerts-work

Items:
  prom.alert[NodeDown,srv01:9100,critical]
  type: trapper

Trigger:
  last(/prometheus-alerts-work/prom.alert[NodeDown,srv01:9100,critical])=1
```

Но в промышленном варианте лучше делать не ручное создание item на каждый alert, а один из вариантов:

```text
1. LLD из JSON alert list;
2. один raw JSON item + dependent items;
3. bridge сам создаёт/update-ит entities через Glaber API;
4. bridge отправляет events в заранее описанный универсальный item.
```

## Практически лучший вариант

Я бы делал bridge так:

```text
Alertmanager webhook
        ↓
bridge
        ↓
один trapper item: prometheus.alerts.raw
        ↓
Glaber preprocessing / LLD
        ↓
автоматические items/triggers по alert fingerprint
```

А для начала можно сделать совсем простой MVP:

```text
один item prometheus.alerts.firing_count
```

Bridge считает количество `firing` alert-ов и отправляет число:

```text
0 — нет активных алертов
>0 — есть активные алерты
```

Потом расширить до отдельных проблем по каждому alert.

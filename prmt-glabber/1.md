Надёжнее **получать в Glaber уже готовые алерты из Prometheus/Alertmanager**, а не заново строить алертинг в Glaber по метрикам Prometheus.

Причина: у вас Prometheus уже считает alert rules, Alertmanager уже умеет группировку, дедупликацию, inhibition и silence. Alertmanager как раз предназначен для обработки алертов от Prometheus, их группировки, подавления и маршрутизации получателям. ([prometheus.io][1])

## Рекомендуемая схема

```text
work contour
 ┌────────────┐      ┌──────────────┐      ┌────────────────────┐
 │ Prometheus │ ---> │ Alertmanager │ ---> │ AM → Glaber bridge │
 └────────────┘      └──────────────┘      └─────────┬──────────┘
                                                     │
                                                     │ Glaber sender / trapper / API
                                                     ▼
                                             ┌──────────────┐
                                             │ Glaber proxy │
                                             └──────┬───────┘
                                                    │
main contour                                         │
                                             ┌──────▼───────┐
                                             │ Glaber main  │
                                             └──────────────┘
```

То есть в контуре `work` добавляется небольшой **адаптер/bridge**, который принимает webhook от Alertmanager и передаёт событие в Glaber через `glabber/glaber proxy`.

Alertmanager поддерживает generic webhook receiver: он отправляет HTTP POST на заданный `url`, а `send_resolved` позволяет получать и закрытие алертов. ([prometheus.io][2])

## Почему не стоит тащить сырые метрики в Glaber

Вариант «Glaber забирает метрики из Prometheus и сам формирует алерты» возможен, но я бы рассматривал его как запасной.

Минусы:

1. Нужно дублировать PromQL-логику в триггерах Glaber.
2. Теряются особенности Prometheus alert rules: `for`, label-логика, recording rules, агрегаты.
3. Возникает риск расхождения: в Grafana/Prometheus алерт один, в Glaber — другой.
4. Надо решать, как разложить Prometheus labels в хосты/items Glaber.
5. При большом количестве метрик появится лишняя нагрузка на Prometheus и Glaber proxy.

У Glaber действительно есть HTTP-agent для получения данных по HTTP/HTTPS, причём проверки могут выполняться через Glaber proxy, если узлы находятся под контролем прокси. ([docs.glaber.io][3]) Поэтому технически можно опрашивать Prometheus API через proxy. Prometheus, в свою очередь, имеет HTTP API `/api/v1`, а активные алерты можно получить через `/api/v1/alerts`. ([prometheus.io][4]) Но это скорее polling-интеграция, а не полноценная доставка событий.

## Практический вариант №1: Alertmanager webhook → Glaber

В `alertmanager.yml` добавляется отдельный receiver:

```yaml
route:
  receiver: default
  routes:
    - receiver: glaber
      matchers:
        - severity=~"warning|critical"
      continue: true

receivers:
  - name: default

  - name: glaber
    webhook_configs:
      - url: "http://am-glaber-bridge.work.local:8080/alertmanager"
        send_resolved: true
```

Bridge должен:

1. принять JSON от Alertmanager;
2. разобрать `status`, `labels`, `annotations`, `startsAt`, `endsAt`;
3. сформировать стабильный ключ события, например:

```text
prom.alert[{alertname},{instance},{job},{severity}]
```

4. отправить в Glaber значение через trapper/sender или API;
5. при `status=firing` выставить проблемное состояние;
6. при `status=resolved` закрыть проблему.

В Glaber это лучше моделировать не как «сырые метрики», а как отдельный host, например:

```text
Host: prometheus-alerts-work
Items:
  prom.alert.raw
  prom.alert.count
  prom.alert.firing[alertname,instance]
Triggers:
  last(/prometheus-alerts-work/prom.alert.firing[...])=1
```

Если нужно отображать каждый алерт отдельной проблемой, bridge должен нормализовать labels и передавать в Glaber либо отдельные trapper-items, либо один JSON item + зависимые элементы/препроцессинг. У Glaber есть развитый препроцессинг JSON: dispatch по полям, JSON filtering, local dispatch и агрегация JSON-данных. ([docs.glaber.io][5])

## Практический вариант №2: Glaber proxy опрашивает Alertmanager API

Если webhook-подход нежелателен, можно сделать pull-схему:

```text
Glaber main → Glaber proxy → Alertmanager API
```

Glaber proxy периодически запрашивает:

```text
http://alertmanager.work.local:9093/api/v2/alerts
```

или Prometheus:

```text
http://prometheus.work.local:9090/api/v1/alerts
```

Дальше HTTP-agent item получает JSON, а зависимые items/JS preprocessing выделяют:

```text
alertname
severity
instance
status
startsAt
annotations.summary
annotations.description
```

Плюсы: не нужен отдельный входящий webhook-сервис.

Минусы: polling, задержка, сложнее корректно закрывать события, нужно аккуратно обрабатывать исчезновение алерта из API.

## Практический вариант №3: Glaber опрашивает Prometheus metrics и строит свои триггеры

Это уместно только если дежурной смене нужны **именно Glaber-native triggers**, а Prometheus используется как источник метрик.

Схема:

```text
Glaber proxy → Prometheus /api/v1/query
Glaber trigger engine → Problems
```

Примеры запросов:

```text
/api/v1/query?query=up == 0
/api/v1/query?query=node_filesystem_avail_bytes / node_filesystem_size_bytes < 0.1
/api/v1/query?query=ALERTS{alertstate="firing"}
```

Но тогда Glaber фактически становится вторым alerting engine. Это допустимо для небольшого набора критичных проверок, но не как основная интеграция.

## Что я бы выбрал

**Основной вариант:** Alertmanager webhook → bridge в `work` → Glaber proxy → Glaber main.

**Резервный/контрольный вариант:** отдельный HTTP-agent item через Glaber proxy, который проверяет доступность Alertmanager и количество firing alerts.

Минимальный набор контроля:

```text
1. Alertmanager доступен?
2. Bridge доступен?
3. Последний webhook был не старше N минут?
4. Количество firing alerts в Alertmanager совпадает с количеством активных проблем в Glaber?
5. Есть Watchdog/DeadMansSwitch alert, который регулярно доходит до Glaber.
```

## Важный момент по безопасности

Не открывайте доступ из `main` напрямую к Prometheus/Alertmanager. При вашей модели достаточно разрешить в `work`:

```text
Alertmanager → am-glaber-bridge
am-glaber-bridge → Glaber proxy
Glaber proxy → Glaber main
```

или, если bridge работает рядом с proxy:

```text
Alertmanager → bridge localhost/work
bridge → Glaber proxy localhost/work
```

Так вы сохраняете изоляцию контуров и не превращаете Prometheus API в межконтурный интерфейс.

Итог: **не переносите алертинг из Prometheus в Glaber без необходимости**. Используйте Alertmanager как источник истины, а Glaber — как витрину и систему обработки проблем для дежурной смены.

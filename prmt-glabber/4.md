Да, это хороший обходной вариант. **Если нет доступа к конфигурации Glaber proxy, но можно создавать host/items/templates в Glaber UI**, то можно поставить отдельный сервер с `zabbix_agent` или `zabbix_agent2` в контуре `work`.

Тогда схема будет такой:

```text
Alertmanager
   │ webhook
   ▼
bridge-сервер в work
   │
   │ localhost:8080
   ▼
zabbix_agent2
   ▲
   │ passive checks
   │
Glaber proxy
   ▲
   │
Glaber main
```

Glaber proxy будет опрашивать agent как обычный Zabbix/Glaber agent. В Zabbix-подобной модели passive check означает, что server или proxy запрашивает данные у agent, а agent отдаёт результат. Active check, наоборот, инициируется самим agent, который получает список проверок и отправляет значения на server/proxy. ([Zabbix][1])

## Когда этот вариант подходит

Он подходит, если выполняются условия:

```text
1. Glaber proxy уже имеет сетевой доступ к серверу с agent на TCP/10050.
2. Вы можете создать host в Glaber UI и назначить его на существующий proxy.
3. Вы можете добавить template/items/LLD rule в Glaber.
4. На сервере в work можно установить zabbix_agent2 и bridge.
```

При этом **конфигурацию самого Glaber proxy менять не нужно**. Proxy просто начнёт опрашивать новый host, потому что его конфигурация придёт из Glaber main.

## Рекомендуемая схема с agent

Лучше сделать так:

```text
Alertmanager → bridge → локальные JSON-файлы → zabbix_agent2 UserParameter → Glaber proxy
```

То есть bridge принимает webhook и пишет состояние алертов в файл:

```text
/var/lib/am-glaber-bridge/lld.json
/var/lib/am-glaber-bridge/alerts/<fingerprint>/status
/var/lib/am-glaber-bridge/alerts/<fingerprint>/summary
/var/lib/am-glaber-bridge/alerts/<fingerprint>/labels.json
```

А `zabbix_agent2` отдаёт эти значения через `UserParameter`.

Преимущество: Glaber proxy не должен ходить HTTP-запросами в bridge. Он опрашивает только agent:

```text
Glaber proxy → zabbix_agent2:10050
```

# Вариант A: passive zabbix_agent2

## 1. Конфиг agent

Файл:

```text
/etc/zabbix/zabbix_agent2.conf
```

Пример:

```ini
Server=IP_ИЛИ_DNS_GLABER_PROXY
ListenPort=10050
Hostname=prometheus-alerts-work-agent

Include=/etc/zabbix/zabbix_agent2.d/*.conf
```

`Server` должен содержать адрес Glaber proxy, с которого будут приходить passive-запросы. Для passive checks именно proxy/server инициирует запрос к agent. ([Zabbix][1])

## 2. UserParameter для bridge

Файл:

```text
/etc/zabbix/zabbix_agent2.d/am-glaber-bridge.conf
```

```ini
UserParameter=am.bridge.health,cat /var/lib/am-glaber-bridge/health.txt 2>/dev/null || echo 0
UserParameter=am.alerts.discovery,cat /var/lib/am-glaber-bridge/lld.json 2>/dev/null || echo '{"data":[]}'
UserParameter=am.alert.status[*],cat /var/lib/am-glaber-bridge/alerts/$1/status 2>/dev/null || echo 0
UserParameter=am.alert.severity_num[*],cat /var/lib/am-glaber-bridge/alerts/$1/severity_num 2>/dev/null || echo 0
UserParameter=am.alert.summary[*],cat /var/lib/am-glaber-bridge/alerts/$1/summary 2>/dev/null || echo ""
UserParameter=am.alert.labels[*],cat /var/lib/am-glaber-bridge/alerts/$1/labels.json 2>/dev/null || echo "{}"
UserParameter=am.alert.annotations[*],cat /var/lib/am-glaber-bridge/alerts/$1/annotations.json 2>/dev/null || echo "{}"
```

Перезапуск agent:

```bash
systemctl restart zabbix-agent2
```

Проверка локально:

```bash
zabbix_agent2 -t am.alerts.discovery
zabbix_agent2 -t 'am.alert.status[testnode001]'
```

```text

resolved → status = 0
```

Например, после webhook от Alertmanager bridge создаёт:

```text

/var/lib/am-glaber-bridge/alerts/testnode001/labels.json
/var/lib/am-glaber-bridge/alerts/testnode001/annotations.json
```

LLD-файл:

```json
{
  "data": [
```

# Вариант B: active agent

Если Glaber proxy **не может подключиться к agent на TCP/10050**, но сервер с agent может сам подключаться к proxy на TCP/10051, тогда можно использовать active checks.

Схема:

```text
zabbix_agent2 → Glaber proxy
```

Для active checks agent сам инициирует соединение, получает список checks и отправляет значения. В документации Zabbix указано, что для active checks proxy должен быть указан в параметре `ServerActive`. ([Zabbix][2])

Конфиг:

```ini
Server=IP_ИЛИ_DNS_GLABER_PROXY
ServerActive=IP_ИЛИ_DNS_GLABER_PROXY:10051
Hostname=prometheus-alerts-work-agent

Include=/etc/zabbix/zabbix_agent2.d/*.conf
```

В Glaber items тогда должны быть типа:

```text
Zabbix agent active
```

Но для вашей ситуации я бы сначала выбрал **passive agent через proxy**, если TCP/10050 от proxy до agent доступен.

# Что лучше: HTTP-agent bridge или zabbix_agent2?

При отсутствии доступа к конфигурации proxy я бы выбрал так:

```text
Если Glaber proxy может ходить на bridge-сервер по TCP/10050:
    zabbix_agent2 passive + UserParameter

Если только bridge-сервер может ходить к Glaber proxy по TCP/10051:
    zabbix_agent2 active + UserParameter

Если можно создавать только items, но нельзя ставить agent:
    HTTP agent item через proxy, как в предыдущем варианте
```

## Практически лучший вариант для вас

```text
Alertmanager → bridge на отдельном сервере work
bridge пишет state-файлы
zabbix_agent2 отдаёт эти файлы через UserParameter
Glaber proxy опрашивает agent
Glaber создаёт items/triggers через LLD
```

Это минимально инвазивно: **proxy не трогаем**, Prometheus/Alertmanager наружу не открываем, а дежурная смена видит проблемы в Glaber.

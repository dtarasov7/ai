Да. При условии **один непересекающийся CIDR на каждый тип** максимальный удобный блок для ingress — **`10.241.14.80/28`**, то есть до **16 ingress-узлов**.

## Оптимальная схема

| Тип узлов | Логический CIDR на FW | IP узлов   |       Максимум |
| --------- | --------------------- | ---------- | -------------: |
| Master    | `10.241.14.68/30`     | `.68–.71`  | 4, требуется 3 |
| System    | `10.241.14.72/31`     | `.72–.73`  |              2 |
| Egress    | `10.241.14.74/31`     | `.74–.75`  |              2 |
| Bootstrap | `10.241.14.76/32`     | `.76`      |              1 |
| Ingress   | `10.241.14.80/28`     | `.80–.95`  |         **16** |
| Worker    | `10.241.14.96/27`     | `.96–.126` |         **31** |

На всех серверах:

```text
Маска:   255.255.255.192 (/26)
Gateway: 10.241.14.65
```

Зарезервированы:

```text
10.241.14.65 — default gateway
10.241.14.66
10.241.14.67
```

Свободными остаются:

```text
10.241.14.77–10.241.14.79
```

### Важный момент

Поскольку это логические блоки, адрес `10.241.14.80` не является адресом сети для серверов. При реальной маске `/26` его можно назначать серверу ingress. Аналогично можно использовать адреса `.96`, `.112` и другие границы логических CIDR.

В блоке worker `10.241.14.96/27` адрес `.127` назначать нельзя, поскольку это broadcast фактической сети `10.241.14.64/26`. Поэтому доступен **31 worker**, а не 32.

Итоговая максимальная конфигурация:

```text
Master:     3 из 4
System:     2
Egress:     2
Bootstrap:  1
Ingress:   16
Worker:    31
```

Увеличить ingress свыше **16** при одном CIDR на тип и сохранении минимум 20 worker уже нельзя: следующий размер CIDR — `/27`, а свободного выровненного блока на 32 адреса для ingress в этой `/26` нет.

Развернут кластер k8s на котором будут работать gitlab runners. Будут собираться контейнеры с приложепниями на java , .net.
Сборка контейнеров - multistage.
Сборка осуществляется с помощью kaniko. 
Gitlab находится НЕ в k8s. Также имеется S3 сервис (ceph 17.2.7), развернутый НЕ в k8s.
Kaniko может использовать для кеширования как s3 так и registry cache.
Опиши пожалуйста архитектуру решения (gitlab runner - kaniko). 
Если есть несколько вариантов архитектуры, то опиши их все, указав сильные и слабые стороны каждой.
Необходимо развернуть несколько изолированных gitlab runners, которые будут запускаться на фиксированных worker узлах (требования ИБ)
и обслуживать разные пайплайны. То есть должна быть реализованная мультитенантность.
Соответственно у разных gitlab runners должны быть разные кеши. 
Для разных gitlab runners кеши должны быть в разных бакетах и под разными учетками.
Каждый gitlab runners должен параллельно выполнять до 20 заданий.
gitlab runners доолжны мониториться через запущеный в k8s prometheus.
Приведи конфигурации gitlab runners. Приведи примеры пайплайн.
Приведи манифесты и скрипты, необходимые для деплоя решения.
Примеров dockerfile приводить не надо.

Оставил только нужные направления и расширил их до формата, который можно использовать как основу для презентации, концепции или раздела документа по внедрению локальной LLM в IT.

# Варианты использования локальной LLM для IT-подразделения

## 1. Эксплуатация инфраструктуры и DevOps

### 1.1. Анализ логов, ошибок и технических сообщений

**Суть сценария:**
Локальная LLM помогает инженерам быстрее разбирать логи приложений, операционных систем, Kubernetes, баз данных, middleware, CI/CD и сетевых компонентов. Модель не заменяет системы мониторинга, а выступает как интеллектуальный слой интерпретации: объясняет ошибку, группирует похожие сообщения, выделяет вероятную причину и предлагает шаги диагностики.

**Что может делать LLM:**

* объяснять ошибки из логов простым техническим языком;
* группировать однотипные ошибки;
* выделять первопричину среди большого количества событий;
* сопоставлять логи с недавними изменениями, релизами или инцидентами;
* предлагать команды для проверки состояния системы;
* формировать краткую сводку по инциденту;
* готовить черновик postmortem-отчета.

**Источники данных:**

* ELK / OpenSearch;
* Grafana Loki;
* Kubernetes logs;
* system logs;
* application logs;
* CI/CD logs;
* database logs;
* сообщения из мониторинга и алертинга.

**Пример запроса:**

> Проанализируй эти логи Kubernetes и объясни, почему pod уходит в CrashLoopBackOff.

**Ожидаемый результат:**

* краткое описание проблемы;
* вероятная причина;
* затронутые компоненты;
* рекомендации по проверке;
* список команд для диагностики;
* возможный workaround;
* рекомендации по предотвращению повторения.

**Польза для IT:**

* сокращение времени диагностики;
* снижение зависимости от отдельных экспертов;
* ускорение обработки инцидентов;
* накопление знаний по типовым ошибкам;
* улучшение качества runbook’ов.

---

### 1.2. Отслеживание логов, метрик и трейсов в online-режиме

**Суть сценария:**
LLM используется как интеллектуальный помощник поверх observability-платформы. Она анализирует поток событий, алертов, логов, метрик и distributed traces, помогает обнаруживать предупреждения, ошибки и аномалии, а также формирует понятные объяснения для инженеров.

Важно: LLM не должна напрямую обрабатывать весь сырой поток логов в реальном времени. Оптимальный подход — передавать в модель уже отфильтрованные, агрегированные или подозрительные события, выявленные системами мониторинга, корреляции и anomaly detection.

**Что может делать LLM:**

* анализировать всплески ошибок;
* объяснять деградацию метрик;
* находить связь между логами, метриками и трассировками;
* выделять аномалии в поведении сервисов;
* объяснять, какой сервис является вероятной точкой отказа;
* определять, является ли проблема инфраструктурной, сетевой, прикладной или связанной с БД;
* формировать краткий incident brief для дежурной команды;
* предлагать дальнейшие шаги расследования.

**Типовые сигналы для анализа:**

* рост HTTP 5xx;
* увеличение latency;
* рост количества timeout’ов;
* ошибки подключения к БД;
* падение pod’ов;
* рост CPU, RAM, disk I/O;
* saturation connection pool;
* деградация очередей;
* рост числа failed jobs;
* unusual traffic pattern;
* сбои внешних интеграций.

**Источники данных:**

* Prometheus;
* Grafana;
* OpenTelemetry;
* Jaeger / Tempo;
* ELK / OpenSearch;
* Loki;
* Zabbix;
* Kubernetes events;
* APM-системы;
* CI/CD events;
* CMDB;
* календарь изменений и релизов.

**Пример работы сценария:**

1. Monitoring фиксирует рост latency в сервисе оплаты.
2. Система корреляции находит связанные логи и traces.
3. LLM получает агрегированную картину: метрики, ошибки, затронутые сервисы, недавние изменения.
4. LLM формирует объяснение:
   «Вероятная причина — деградация соединений с БД после релиза версии X. Ошибки начались через 8 минут после деплоя. В traces видно увеличение времени на операции INSERT».
5. LLM предлагает следующие действия: проверить connection pool, rollback, статус БД, нагрузку на таблицы, slow queries.

**Ожидаемый результат:**

* краткая сводка проблемы;
* вероятная первопричина;
* список затронутых сервисов;
* временная шкала событий;
* связанные алерты;
* рекомендации по диагностике;
* предлагаемые действия;
* уровень уверенности;
* ссылки на источники: лог-записи, графики, traces, релизы.

**Польза для IT:**

* снижение MTTR;
* ускорение triage;
* уменьшение шума от алертов;
* лучшее понимание распределенных инцидентов;
* более быстрый переход от симптома к причине;
* повышение качества дежурств.

**Ограничения и меры контроля:**

* LLM не должна самостоятельно выполнять destructive actions;
* автоматические действия допустимы только после подтверждения инженера;
* все рекомендации должны сопровождаться ссылками на источники;
* критичные выводы должны иметь confidence score;
* нужен аудит всех запросов и ответов;
* необходимо ограничение доступа по ролям.

### 1.3. Помощник по инцидентам

**Суть сценария:**
Во время инцидента LLM помогает дежурному инженеру быстро собрать контекст, структурировать информацию и предложить план действий.

**Что может делать LLM:**

* собирать информацию из алертов, логов, метрик, CMDB и истории изменений;
* формировать краткую картину инцидента;
* предлагать гипотезы;
* составлять план диагностики;
* подсказывать ответственные команды;
* готовить update для бизнеса;
* формировать postmortem после завершения инцидента.

**Пример результата:**

```text
Инцидент: Рост ошибок 5xx в сервисе авторизации.

Начало: 10:42  
Затронутые системы: auth-service, user-profile-service  
Симптомы: рост 5xx до 18%, latency p95 выросла с 220 мс до 1,8 сек  
Вероятная причина: деградация подключения к Redis-кластеру  
Недавние изменения: деплой auth-service v2.17.4 в 10:35  
Рекомендованные действия:
1. Проверить Redis latency и connection pool.
2. Сравнить конфигурацию v2.17.4 с предыдущей версией.
3. При подтверждении проблемы выполнить rollback.
4. Проверить наличие аналогичных ошибок в предыдущих инцидентах.
```

**Польза:**

* ускорение первичного анализа;
* единый формат работы с инцидентами;
* лучшее документирование;
* меньше хаоса во время аварий;
* быстрая подготовка коммуникаций.

---

### 1.4. Генерация и актуализация runbook’ов

**Суть сценария:**
LLM помогает создавать, обновлять и улучшать runbook’и для типовых операций и инцидентов.

**Что может делать LLM:**

* создавать runbook по описанию системы;
* превращать историю инцидента в инструкцию;
* обновлять устаревший runbook;
* проверять полноту инструкции;
* добавлять разделы rollback, verification, escalation;
* приводить инструкции к единому корпоративному шаблону.

**Структура runbook’а, которую может генерировать LLM:**

* название сценария;
* симптомы;
* возможные причины;
* предварительные проверки;
* команды диагностики;
* действия по восстановлению;
* критерии успешного восстановления;
* rollback;
* escalation path;
* ссылки на dashboards, логи, CMDB и владельцев системы.

**Польза:**

* стандартизация эксплуатации;
* снижение зависимости от экспертов;
* ускорение onboarding;
* повышение готовности дежурных команд;
* формирование базы знаний на основе реальных инцидентов.

---

### 1.5. Проверка конфигураций

**Суть сценария:**
LLM анализирует конфигурационные файлы и помогает находить ошибки, небезопасные настройки, дублирование, конфликтующие параметры и потенциальные проблемы эксплуатации.

**Что можно проверять:**

* Kubernetes YAML;
* Helm charts;
* Terraform;
* Ansible playbooks;
* Dockerfile;
* Nginx / Apache;
* systemd units;
* firewall rules;
* CI/CD pipelines;
* настройки БД;
* cloud security groups;
* IAM policies.

**Примеры проверок:**

* отсутствуют resource limits в Kubernetes;
* небезопасный privileged mode;
* отсутствует readiness probe;
* слишком широкие IAM-права;
* secrets указаны в открытом виде;
* нет rollback-стратегии;
* не задан timeout;
* потенциально опасные firewall rules.

**Польза:**

* снижение количества ошибок в production;
* улучшение качества инфраструктурного кода;
* повышение безопасности;
* ускорение ревью конфигураций.

## 2. Документация, регламенты и база знаний

### 2.1. Диалоговый поиск по внутренней документации

**Суть сценария:**
Локальная LLM подключается к внутренним источникам знаний и позволяет сотрудникам искать информацию в формате диалога, а не через ручной поиск по Confluence, Wiki, файловым хранилищам и регламентам.

**Что может делать LLM:**

* отвечать на вопросы по внутренним документам;
* находить нужный регламент или инструкцию;
* объяснять сложные документы простым языком;
* показывать ссылки на источники;
* сравнивать несколько документов;
* выявлять противоречия между инструкциями;
* отвечать с учетом роли пользователя и его прав доступа.

**Источники данных:**

* Confluence;
* SharePoint;
* Wiki;
* Service Desk knowledge base;
* Git repositories;
* PDF-инструкции;
* архитектурные документы;
* эксплуатационная документация;
* регламенты ИБ;
* CMDB;
* внутренние стандарты.

**Пример запроса:**

> Как правильно запросить доступ к production-базе данных?

**Ожидаемый ответ:**

* краткое описание процесса;
* ссылка на актуальный регламент;
* необходимые согласования;
* SLA;
* ограничения;
* контакт ответственной команды.

**Польза:**

* быстрый доступ к знаниям;
* меньше обращений в поддержку;
* снижение потерь времени на поиск;
* повышение использования существующей документации;
* уменьшение зависимости от неформальных каналов знаний.

---

### 2.2. Суммаризация технических документов

**Суть сценария:**
LLM помогает быстро получать краткие и понятные выжимки из длинных документов: ТЗ, RFC, архитектурных описаний, отчетов, протоколов, инструкций и vendor-документации.

**Что может делать LLM:**

* кратко пересказывать длинные документы;
* выделять ключевые требования;
* находить риски и ограничения;
* формировать executive summary;
* выделять action items;
* сравнивать версии документов;
* превращать документ в чек-лист.

**Примеры документов:**

* техническое задание;
* архитектурное решение;
* описание интеграции;
* инструкция по эксплуатации;
* отчет по инциденту;
* протокол совещания;
* политика информационной безопасности;
* vendor manual.

**Пример результата:**

```text
Краткая выжимка:
Документ описывает переход сервиса X на новую схему авторизации через OAuth2. 
Основные изменения затрагивают API Gateway, IAM и mobile backend.

Ключевые риски:
1. Возможна несовместимость старых клиентов.
2. Требуется миграция токенов.
3. Не описан rollback-план.
4. Не определен владелец мониторинга после запуска.

Action items:
1. Уточнить стратегию миграции.
2. Добавить rollback-сценарий.
3. Обновить runbook.
4. Согласовать изменения с ИБ.
```

**Польза:**

* экономия времени инженеров и руководителей;
* ускорение согласований;
* повышение качества анализа документов;
* выявление пробелов до начала реализации.

---

### 2.3. Автоматическое создание и обновление базы знаний

**Суть сценария:**
LLM анализирует повторяющиеся тикеты, инциденты, чаты поддержки и обращения пользователей, после чего предлагает новые статьи базы знаний или обновления существующих инструкций.

**Что может делать LLM:**

* находить часто повторяющиеся вопросы;
* предлагать темы для новых статей;
* генерировать черновики инструкций;
* находить устаревшие статьи;
* объединять дублирующиеся инструкции;
* добавлять FAQ на основе реальных обращений;
* приводить статьи к единому шаблону.

**Источники данных:**

* ITSM-система;
* база знаний;
* чаты поддержки;
* инциденты;
* postmortem-отчеты;
* обращения пользователей;
* внутренние инструкции.

**Пример результата:**

```text
Предложение для новой статьи базы знаний:

Тема: Ошибка подключения к VPN после смены пароля

Причина появления темы:
За последние 30 дней обнаружено 48 обращений с похожими симптомами.

Рекомендуемый раздел:
Удаленный доступ / VPN

Черновик инструкции:
1. Перезагрузить VPN-клиент.
2. Обновить сохраненные учетные данные.
3. Проверить MFA.
4. При повторении ошибки создать заявку в Service Desk.
```

**Польза:**

* база знаний становится актуальнее;
* снижается количество повторных обращений;
* Service Desk быстрее закрывает типовые вопросы;
* знания фиксируются системно, а не остаются в чатах.

### 2.4. Приведение документации к единому стандарту

**Суть сценария:**
LLM помогает стандартизировать технические документы: инструкции, runbook’и, архитектурные описания, регламенты, эксплуатационные процедуры.

**Что может делать LLM:**

* переписывать документы по корпоративному шаблону;
* добавлять недостающие разделы;
* упрощать сложные формулировки;
* переводить технический язык в понятный для бизнеса;
* проверять полноту документа;
* выявлять противоречия;
* создавать чек-листы на основе регламентов.

**Единый шаблон может включать:**

* цель документа;
* область применения;
* ответственные роли;
* предварительные условия;
* пошаговую инструкцию;
* риски;
* rollback;
* контрольный список;
* ссылки на связанные документы.

**Польза:**

* единый стиль документации;
* проще аудит;
* легче onboarding;
* меньше ошибок при выполнении процедур;
* выше доверие к базе знаний.

## 3. Архитектура и проектирование

### 3.1. Архитектурный ассистент

**Суть сценария:**
Локальная LLM помогает архитекторам, техлидам и инженерам готовить варианты решений, анализировать требования, фиксировать ограничения и формировать архитектурные документы.

**Что может делать LLM:**

* помогать описывать целевую архитектуру;
* предлагать варианты реализации;
* сравнивать плюсы и минусы подходов;
* выявлять архитектурные риски;
* проверять соответствие внутренним стандартам;
* помогать готовить design document;
* формировать вопросы к бизнесу и смежным командам.

**Пример запроса:**

> Подготовь варианты архитектуры для интеграции новой CRM с существующей шиной данных.

**Ожидаемый результат:**

* несколько вариантов архитектуры;
* плюсы и минусы каждого варианта;
* ограничения;
* риски;
* зависимости;
* требования к безопасности;
* требования к мониторингу;
* рекомендации по выбору.

**Польза:**

* ускорение подготовки архитектурных решений;
* более качественная проработка альтернатив;
* снижение риска забыть важные ограничения;
* единый формат design-документов.

---

### 3.2. Анализ требований

**Суть сценария:**
LLM помогает выявлять неясности, противоречия и неполноту в бизнес- и технических требованиях до начала разработки.

**Что может делать LLM:**

* находить неоднозначные формулировки;
* выделять функциональные и нефункциональные требования;
* определять отсутствующие требования;
* формировать вопросы к заказчику;
* проверять требования на тестируемость;
* выявлять зависимости от других систем;
* находить требования, влияющие на безопасность, производительность и отказоустойчивость.

**Пример результата:**

```text
Выявленные проблемы в требованиях:

1. Не определены SLA и RTO/RPO.
2. Не описано поведение системы при недоступности внешнего API.
3. Не указаны требования к аудиту действий пользователей.
4. Не определена максимальная нагрузка.
5. Не описаны требования к хранению персональных данных.

Вопросы для уточнения:
1. Какой допустимый простой системы?
2. Нужно ли логировать все изменения пользовательских данных?
3. Какие роли пользователей должны быть предусмотрены?
4. Какой ожидаемый объем запросов в час?
```

**Польза:**

* меньше доработок на поздних этапах;
* лучшее качество постановки задач;
* снижение риска архитектурных ошибок;
* ускорение согласования требований.

---

### 3.3. Генерация Architecture Decision Records

**Суть сценария:**
LLM помогает фиксировать архитектурные решения в формате ADR. Это особенно полезно для крупных IT-ландшафтов, где важно понимать, почему было принято то или иное решение.

**Что может делать LLM:**

* создавать ADR по итогам обсуждения;
* формировать краткий контекст решения;
* описывать рассмотренные альтернативы;
* фиксировать плюсы и минусы;
* описывать последствия решения;
* связывать ADR с задачами, проектами и системами.

**Типовая структура ADR:**

* название решения;
* статус;
* контекст;
* проблема;
* рассмотренные варианты;
* принятое решение;
* последствия;
* риски;
* связанные документы.

**Пример ADR:**

```text
ADR: Использование Kafka для асинхронного обмена событиями между CRM и DWH

Статус: Proposed

Контекст:
Новая CRM должна передавать события о клиентах и сделках в DWH почти в реальном времени.

Рассмотренные варианты:
1. Прямая интеграция через REST API.
2. Передача файлов по расписанию.
3. Асинхронная передача событий через Kafka.

Решение:
Использовать Kafka как основной механизм передачи событий.

Причины:
- высокая устойчивость к временной недоступности потребителя;
- возможность масштабирования;
- поддержка нескольких потребителей;
- соответствие текущей event-driven архитектуре.

Последствия:
- требуется мониторинг lag;
- требуется схема управления контрактами событий;
- необходимо определить владельцев топиков.
```

**Польза:**

* прозрачность архитектурных решений;
* проще аудит и сопровождение;
* быстрее onboarding новых архитекторов и разработчиков;
* меньше повторных обсуждений уже принятых решений.

---

### 3.4. Архитектурное ревью

**Суть сценария:**
LLM помогает проводить предварительное ревью архитектурных решений до рассмотрения на архитектурном комитете.

**Что может проверять LLM:**

* соответствие внутренним стандартам;
* наличие мониторинга и логирования;
* требования к безопасности;
* отказоустойчивость;
* масштабируемость;
* интеграционные риски;
* использование персональных данных;
* наличие rollback-плана;
* соответствие целевой архитектуре.

**Пример результата:**

```text
Предварительное архитектурное ревью:

Сильные стороны:
1. Решение использует стандартный API Gateway.
2. Предусмотрена асинхронная обработка событий.
3. Есть базовое описание мониторинга.

Замечания:
1. Не описан сценарий недоступности внешней системы.
2. Не указаны лимиты API.
3. Не определен владелец данных.
4. Не описан механизм повторной обработки сообщений.
5. Не хватает требований к аудиту.

Рекомендации:
1. Добавить retry policy и dead-letter queue.
2. Описать RTO/RPO.
3. Добавить security review.
4. Уточнить схему мониторинга.
```

**Польза:**

* повышение качества материалов до комитета;
* сокращение времени архитектурного ревью;
* меньше типовых замечаний;
* единый стандарт проверки решений.

---

## 4. Подготовка рекомендаций по hardening

### 4.1. Hardening операционных систем, платформ и сервисов

**Суть сценария:**
Локальная LLM помогает анализировать текущие настройки систем и готовить рекомендации по усилению безопасности. Сценарий особенно полезен для Linux/Windows-серверов, Kubernetes, БД, middleware, веб-серверов и сетевых сервисов.

**Что может делать LLM:**

* анализировать конфигурации;
* сравнивать настройки с внутренними стандартами;
* выявлять небезопасные параметры;
* объяснять риск каждой настройки;
* предлагать безопасные значения;
* формировать hardening checklist;
* готовить план внедрения изменений;
* разделять рекомендации по критичности.

**Объекты анализа:**

* Linux servers;
* Windows servers;
* Kubernetes clusters;
* Docker hosts;
* PostgreSQL, Oracle, MS SQL;
* Nginx, Apache;
* Kafka, RabbitMQ;
* Redis;
* Active Directory;
* firewall rules;
* IAM/RBAC policies;
* cloud/private cloud security settings.

**Пример результата:**

```text
Hardening-рекомендации для Kubernetes namespace:

Критичные:
1. Запретить privileged containers.
2. Убрать hostPath volumes без обоснования.
3. Ограничить service account permissions.
4. Добавить resource limits.

Средние:
1. Включить readOnlyRootFilesystem.
2. Добавить network policies.
3. Ограничить egress-трафик.
4. Проверить imagePullPolicy.

Низкие:
1. Добавить labels владельца сервиса.
2. Стандартизировать annotations.
3. Добавить описание runbook.
```

**Польза для IT и ИБ:**

* быстрее подготовка hardening-рекомендаций;
* снижение числа небезопасных конфигураций;
* повышение зрелости инфраструктуры;
* упрощение аудита;
* единый подход к security baseline.

**Контрольные меры:**

* рекомендации должны проходить ревью ИБ и владельца системы;
* LLM не должна автоматически менять настройки;
* необходимо хранить ссылки на исходные конфигурации;
* критичные изменения должны проходить Change Management;
* для каждого изменения нужен rollback-план.

---

### 4.2. Подготовка hardening checklist

**Суть сценария:**
LLM формирует чек-листы для проверки систем перед вводом в эксплуатацию или после изменений.

**Примеры чек-листов:**

* чек-лист безопасной настройки Linux-сервера;
* чек-лист Kubernetes deployment;
* чек-лист безопасной настройки PostgreSQL;
* чек-лист публикации веб-сервиса;
* чек-лист сетевой сегментации;
* чек-лист IAM-ролей;
* чек-лист production readiness.

**Польза:**

* стандартизация проверок;
* снижение риска пропустить важные настройки;
* удобство для эксплуатации, ИБ и аудита;
* повторяемый процесс подготовки систем.

---

## 5. Документирование кода

### 5.1. Генерация технической документации по коду

**Суть сценария:**
Локальная LLM анализирует исходный код внутри периметра компании и помогает автоматически создавать или обновлять техническую документацию.

**Что может делать LLM:**

* описывать назначение модулей;
* объяснять функции и классы;
* генерировать README;
* описывать API;
* формировать sequence diagram в текстовом формате;
* готовить описание зависимостей;
* выделять бизнес-логику;
* объяснять сложные участки legacy-кода;
* обновлять документацию после изменений.

**Источники данных:**

* GitLab / GitHub Enterprise;
* Bitbucket;
* исходный код;
* OpenAPI / Swagger;
* комментарии в коде;
* CI/CD pipelines;
* issue tracker;
* архитектурные документы.

**Пример результата:**

```text
Модуль payment-retry отвечает за повторную обработку неуспешных платежей.

Основная логика:
1. Получает события failed_payment из очереди.
2. Проверяет тип ошибки.
3. Для временных ошибок создает задачу повторной обработки.
4. Для постоянных ошибок отправляет событие payment_failed_final.
5. Записывает результат в audit log.

Зависимости:
- payment-service;
- billing-db;
- Kafka topic failed_payment;
- audit-service.

Риски:
- нет ограничения количества повторов;
- отсутствует circuit breaker при недоступности billing-db;
- не описан rollback.
```

**Польза:**

* ускорение onboarding разработчиков;
* снижение зависимости от авторов кода;
* актуализация технической документации;
* упрощение сопровождения legacy-систем;
* повышение прозрачности систем.

## 6. Рекомендуемый порядок внедрения

### Этап 1. Быстрый пилот

**Приоритетные сценарии:**

1. Диалоговый поиск по внутренней документации.

3. Генерация runbook’ов.
4. Суммаризация технических документов.
5. Документирование кода для одного-двух пилотных репозиториев.

**Почему эти сценарии подходят для старта:**

* быстро демонстрируют пользу;
* не требуют полной автоматизации;
* человек остается в контуре принятия решений;
* можно легко измерить эффект;
* риски ниже, чем при автоматическом выполнении действий.

**Добавить:**

3. Проверку конфигураций.
4. Помощника по инцидентам.
5. Автоматическое создание статей базы знаний.

**Что потребуется:**

* интеграция с monitoring и observability;
* подключение ITSM;
* подключение CMDB;
* настройка RBAC;
* аудит запросов;
* контроль доступа к источникам данных;
* шаблоны ответов и runbook’ов.

5. Проверку соответствия внутренним стандартам.

**Ожидаемый эффект:**

* повышение качества архитектурных документов;
* ускорение согласований;

## 8. Общие требования к локальной LLM

Для всех перечисленных сценариев важно предусмотреть следующие требования:

1. **Локальное размещение**
   Данные, код, логи и внутренние документы не должны передаваться во внешние облачные LLM-сервисы.

2. **RAG по внутренним источникам**
   Ответы должны строиться на основе актуальных внутренних документов, логов, тикетов, runbook’ов и архитектурных материалов.

3. **Ссылки на источники**
   Модель должна показывать, на основании каких документов, логов, метрик или конфигураций сделан вывод.

4. **RBAC и разграничение доступа**
   Пользователь должен видеть только ту информацию, к которой у него есть права.

5. **Аудит действий**
   Все запросы, ответы, использованные источники и рекомендации должны логироваться.

6. **Human-in-the-loop**
   Для критичных сценариев LLM должна быть советником, а не автономным исполнителем.

7. **Защита от галлюцинаций**
   Если данных недостаточно, модель должна явно сообщать об этом, а не придумывать ответ.

8. **Интеграция с IT-процессами**
   Решение должно быть связано с ITSM, CMDB, мониторингом, Git, Wiki, SIEM и Change Management.

9. **Шаблоны ответов**
   Для инцидентов, hardening, архитектурного ревью, runbook’ов и документации нужны утвержденные форматы.

10. **Оценка качества**
    Необходимо регулярно измерять точность ответов, полезность рекомендаций и удовлетворенность пользователей.

Для следующего шага логично превратить это в более управленческий формат: таблица «use case — ценность — сложность — риски — данные — интеграции — метрики» для выбора пилота.

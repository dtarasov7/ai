## 2. Что остается в плане

В обновленном плане остаются сценарии, которые могут работать без доступа к истории тикетов и внутренней документации.

Основные допустимые источники данных:

* текущие логи;
* текущие метрики;
* distributed traces;
* алерты мониторинга;
* конфигурационные файлы;
* infrastructure as code;
* исходный код;
* OpenAPI / Swagger;
* результаты сканеров безопасности;
* вручную предоставленные требования или архитектурные материалы;
* текущий контекст инцидента, без обращения к историческим тикетам.

# 4. Приоритеты внедрения

## Группа A. Рекомендуется для первого пилота

| Use case                          | Почему подходит для пилота                                                 |
| --------------------------------- | -------------------------------------------------------------------------- |
| Анализ логов и ошибок             | Быстрый практический эффект для эксплуатации и DevOps                      |
| Проверка конфигураций             | Хорошо работает на ограниченном наборе файлов и не требует истории тикетов |
| Документирование кода             | Полезно для разработки, не требует базы знаний или внутренней документации |
| Документирование API              | Дает понятный результат по коду и OpenAPI-спецификациям                    |
| Подготовка hardening-рекомендаций | Ценно для ИБ и эксплуатации, можно начать с read-only анализа конфигураций |

## Группа B. Второй этап

| Use case                                        | Почему лучше после пилота                                           |
| ----------------------------------------------- | ------------------------------------------------------------------- |
| Online-анализ логов, метрик и трейсов           | Требует более сложной интеграции с observability-платформой         |
| Помощник по текущему инциденту                  | Нужна интеграция с алертами, метриками и коммуникационными каналами |
| Документирование legacy-кода                    | Высокая ценность, но сложность зависит от качества старого кода     |
| Анализ требований по предоставленному материалу | Полезно, но важно заранее определить шаблон оценки требований       |

## Группа C. Зрелый этап

| Use case                | Почему позже                                                              |
| ----------------------- | ------------------------------------------------------------------------- |
| Архитектурный ассистент | Требует дисциплины в описании входного контекста                          |
| Архитектурное ревью     | Нужны формализованные критерии проверки                                   |
| Генерация ADR           | Лучше запускать после утверждения шаблона ADR и процесса хранения решений |

Для первого пилота предлагается оставить 5 сценариев:

| № | Use case                          | Основной результат пилота                                                    |
| - | --------------------------------- | ---------------------------------------------------------------------------- |
| 1 | Анализ логов и ошибок             | LLM объясняет ошибки, группирует события и предлагает шаги диагностики       |
| 2 | Проверка конфигураций             | LLM анализирует IaC и конфигурации на ошибки, риски и небезопасные параметры |
| 3 | Подготовка hardening-рекомендаций | LLM формирует список рекомендаций по усилению безопасности                   |
| 4 | Документирование кода             | LLM создает описание модулей, функций и зависимостей                         |
| 5 | Документирование API              | LLM формирует или обновляет описание API по коду и OpenAPI                   |

# 6. Границы первого пилота

## Включить в пилот

* 1–2 сервиса;
* 1–2 репозитория кода;
* ограниченный набор логов;
* ограниченный набор конфигураций;
* один тип платформы для hardening, например Kubernetes или Linux;
* read-only режим;
* ручную проверку всех рекомендаций инженером;
* сохранение результатов в виде отчетов или merge request comments.

## Не включать в пилот

* доступ к истории тикетов;
* доступ к базе знаний;
* доступ к Confluence / Wiki / SharePoint;
* анализ старых инцидентов;
* автоматическое создание статей базы знаний;
* автоматическое изменение production-конфигураций;
* автоматическое выполнение команд;
* автоматический rollback;
* принятие решений без инженера.

---

# 7. Цели первого пилота

## Основная цель

Проверить, насколько локальная LLM может помогать IT-командам в анализе технических данных без подключения к истории тикетов и внутренней документации.

## Практические цели

1. Сократить время первичного анализа логов и ошибок.
2. Найти ошибки и риски в конфигурациях.
3. Подготовить hardening-рекомендации по выбранной платформе.
4. Создать техническое описание для выбранных репозиториев.
5. Подготовить или обновить API-документацию по выбранному сервису.

# 8. Критерии успешности пилота

| Направление           | Целевая метрика                                                        |
| --------------------- | ---------------------------------------------------------------------- |
| Анализ логов          | Сокращение времени первичного triage на 20–30%                         |
| Проверка конфигураций | Выявление полезных замечаний минимум в 60–70% проверенных конфигураций |
| Hardening             | Подготовка применимого hardening checklist по выбранной платформе      |
| Документирование кода | Подготовка документации минимум по 2 репозиториям                      |
| Документирование API  | Документирование ключевых endpoint’ов выбранного сервиса               |
| Качество ответов      | Не менее 80% результатов признаны полезными пилотной группой           |
| Безопасность          | Отсутствие доступа к данным вне пилотного контура                      |
| Контроль              | Все рекомендации проходят ручную проверку инженером                    |

# 9. Риски и меры снижения

| Риск                                                    | Меры снижения                                                           |
| ------------------------------------------------------- | ----------------------------------------------------------------------- |
| Модель неверно объясняет ошибку                         | Показывать исходные логи, уровень уверенности и альтернативные гипотезы |
| Модель делает опасные рекомендации                      | Использовать read-only режим и обязательное ревью инженером             |
| Модель не понимает контекст системы                     | Ограничить выводы рамками предоставленных данных                        |
| Рекомендации по hardening могут нарушить работу сервиса | Требовать impact analysis и rollback-план перед применением             |
| Документация по коду может быть неточной                | Проверка владельцем репозитория перед публикацией                       |
| API-документация может расходиться с реализацией        | Сравнивать код, OpenAPI и тесты                                         |
| Слишком широкий пилот                                   | Ограничить пилот 1–2 сервисами и 1–2 платформами                        |

# 10. Рекомендуемая последовательность внедрения

## Шаг 1. Определить пилотный контур

* выбрать 1–2 сервиса;
* выбрать 1–2 репозитория;
* выбрать тип логов для анализа;
* выбрать тип конфигураций для проверки;
* выбрать платформу для hardening.

## Шаг 2. Развернуть локальную LLM

* развернуть модель внутри периметра;
* настроить LLM Gateway;
* включить аудит запросов;
* ограничить доступ только пилотной группе;
* запретить передачу данных во внешние LLM-сервисы.

## Шаг 3. Подключить технические источники

* Git-репозитории;
* CI/CD;
* лог-хранилище;
* observability-систему;
* хранилище конфигураций;
* результаты security scanners.

## Шаг 4. Настроить шаблоны ответов

Для каждого сценария нужен отдельный формат ответа:

* анализ ошибки;
* анализ аномалии;
* проверка конфигурации;
* hardening checklist;
* описание модуля кода;
* описание API endpoint’а;
* архитектурное замечание.

## Шаг 5. Провести пилот

* прогнать реальные кейсы;
* собрать обратную связь инженеров;
* измерить время выполнения задач до и после;
* зафиксировать успешные и неуспешные сценарии;
* определить ограничения модели.

## Шаг 6. Принять решение о масштабировании

По итогам пилота определить:

* какие use case дают максимальную пользу;
* какие требуют доработки;
* какие источники данных можно безопасно добавить позже;
* стоит ли переходить к online-анализу логов, метрик и трейсов;
* нужно ли расширять сценарии на архитектуру и legacy-код.

# 11. Обновленная итоговая рекомендация

Для первого этапа внедрения локальной LLM в IT-подразделении рекомендуется сфокусироваться на технических сценариях, которые не требуют доступа к истории тикетов и внутренней документации.

Оптимальный стартовый контур:

1. Анализ логов и ошибок.
2. Проверка конфигураций.
3. Подготовка hardening-рекомендаций.
4. Документирование кода.
5. Документирование API.

Следующим этапом можно добавить:

1. Online-анализ логов, метрик и трейсов.
2. Помощника по текущему инциденту.
3. Документирование legacy-кода.
4. Анализ требований по предоставленному материалу.
5. Архитектурное ревью по конкретному предоставленному design document.

Такой подход позволяет начать с практических IT-задач, сохранить безопасность данных и не зависеть от зрелости базы знаний, истории тикетов или качества внутренней документации.
